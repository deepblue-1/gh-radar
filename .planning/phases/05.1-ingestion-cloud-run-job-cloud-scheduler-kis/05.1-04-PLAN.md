**사용자가 로컬 셸에서 다음을 실행:**

1. 이메일 주소 env 주입:
   ```bash
   export GCP_PROJECT_ID=gh-radar
   export OPS_EMAIL="<사용자 운영 이메일>"
   ```

2. 이메일 notification channel 생성(이미 존재하면 재사용):
   ```bash
   # 기존 동일 이메일 채널 검색
   EXISTING=$(gcloud beta monitoring channels list \
     --filter="type=email AND labels.email_address=${OPS_EMAIL}" \
     --format='value(name)' | head -n1)

   if [[ -n "$EXISTING" ]]; then
     CHANNEL_NAME="$EXISTING"
     echo "✓ reusing channel: $CHANNEL_NAME"
   else
     cat > /tmp/email-channel.json <<JSON
   {
     "type": "email",
     "displayName": "gh-radar ops email",
     "labels": {"email_address": "${OPS_EMAIL}"},
     "enabled": true
   }
   JSON
     CHANNEL_NAME=$(gcloud beta monitoring channels create \
       --channel-content-from-file=/tmp/email-channel.json \
       --format='value(name)')
     rm -f /tmp/email-channel.json
     echo "✓ created channel: $CHANNEL_NAME"
     echo ""
     echo "⚠️  GCP가 ${OPS_EMAIL}로 verification 메일 발송. inbox 확인 후 링크 클릭 필요."
   fi
   echo "CHANNEL_NAME=$CHANNEL_NAME"   # 예: projects/gh-radar/notificationChannels/1234567890
   ```

3. 신규 채널이면 **verification 이메일의 링크를 클릭**해 검증 완료. (기존 재사용이면 skip.)

4. YAML placeholder 치환 후 policy 생성:
   ```bash
   # sed는 channel resource name에 '/'가 있어 delimiter를 '|'로 지정
   sed "s|\\${NOTIFICATION_CHANNEL_ID}|${CHANNEL_NAME}|g" \
     ops/alert-ingestion-failure.yaml > /tmp/policy.yaml

   # 동일 displayName policy 존재 시 먼저 삭제(idempotent하게 재생성)
   EXISTING_POLICY=$(gcloud alpha monitoring policies list \
     --filter='displayName="gh-radar-ingestion-failure"' \
     --format='value(name)' | head -n1)
   if [[ -n "$EXISTING_POLICY" ]]; then
     gcloud alpha monitoring policies delete "$EXISTING_POLICY" --quiet
     echo "✓ deleted existing policy"
   fi

   gcloud alpha monitoring policies create --policy-from-file=/tmp/policy.yaml
   rm -f /tmp/policy.yaml
   ```

5. 생성 결과 확인:
   ```bash
   gcloud alpha monitoring policies list \
     --filter='displayName="gh-radar-ingestion-failure"' \
     --format='table(displayName,enabled,notificationChannels)'
   ```

POLICY_NAME=$(gcloud alpha monitoring policies list \
  --filter='displayName="gh-radar-ingestion-failure"' \
  --format='value(name)')

  | jq -e '.conditions[0].conditionThreshold.filter | contains("gh-radar-ingestion") and contains("result") and contains("failed")' \

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cloud Run Job metrics → Cloud Monitoring | 메트릭은 GCP 내부 파이프라인으로 전달 — 신뢰 경계 내. |
| Cloud Monitoring → 이메일 서버 | 외부 SMTP(gmail 등)로 알림 전송. 이메일 verification을 통해 수신자 소유 확인. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.1-10 | Spoofing | notification channel 등록 | mitigate | GCP Monitoring의 이메일 verification 필수(V4). 사용자가 실제로 소유한 inbox만 채널로 등록됨. |
| T-05.1-11 | Information Disclosure | alert 이메일 본문 | mitigate | YAML `documentation.content`는 일반적 가이드만 포함 — secret/token 값 없음. Cloud Run Job metric은 count 집계만 — payload 미노출. |
| T-05.1-12 | Tampering | `/tmp/policy.yaml` 치환 후 | mitigate | 치환 결과는 `/tmp/`에서 수명 짧음 + `rm -f` 즉시 삭제. 원본 YAML(리포)은 placeholder 유지 — channel ID 평문 커밋 없음. |
| T-05.1-13 | Denial of Service | alert noise (false positive) | accept | 첫 주 실측 후 threshold 조정. 초기 failures > 0 (매우 민감)은 operator burnout 리스크 — 1주 후 re-evaluate. |
</threat_model>
