## 배포 스크립트 + Job 설정

### deploy-ingestion.sh 구조

| Option | Description | Selected |
|--------|-------------|----------|
| server 패턴 100% 동일 구조 (추천) | scripts/deploy-ingestion.sh: gcloud guard → build(amd64) → push → jobs deploy(create-or-update) → Scheduler 등록 → smoke. 일관성 최우선. | ✓ |
| Job과 Scheduler 분리 (deploy + setup-scheduler 2개) | scripts/deploy-ingestion.sh는 이미지와 Job까지, scripts/setup-ingestion-scheduler.sh로 Scheduler는 1회성 설정. | |
| 단일 스크립트 + idempotent 플래그 | --skip-scheduler 같은 플래그 제공. 기본은 전체 실행, Scheduler가 이미 존재하면 update. | |

**User's choice:** server 패턴 100% 동일 구조

### Job 실행 리소스/타임아웃

| Option | Description | Selected |
|--------|-------------|----------|
| 초안 그대로 (cpu=1, mem=512Mi, timeout=120s, retries=0) | DEPLOY.md의 값을 유지. KIS 토큰 발급(~5s) + 28개 ranking 페이지(~30s) + upsert(~10s) 기준 120s 충분. | ✓ |
| 타임아웃 증가 (180s) + retries=1 | 첫 호출 시 콜드 스타트/네트워크 지연 대응. retries=1로 일시적 KIS 장애 자동 회복. | |
| Claude가 결정 | 실제 실행 시간 측정 후 planner/executor가 판단. | |

**User's choice:** 초안 그대로 (cpu=1, mem=512Mi, timeout=120s, retries=0)

## Cloud Scheduler 전략

### cron 일정 구성

| Option | Description | Selected |
|--------|-------------|----------|
| Scheduler 2개 분리 (정규장 + 종가) | gh-radar-ingestion-trading-hours: '* 9-15 * * 1-5' / gh-radar-ingestion-eod: '35 15 * * 1-5'. | |
