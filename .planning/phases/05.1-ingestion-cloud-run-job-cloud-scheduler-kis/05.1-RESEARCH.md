# Phase 05.1: Ingestion 운영 배포 — Cloud Run Job + Cloud Scheduler - Research

**Researched:** 2026-04-14
**Domain:** GCP 운영 배포 (Cloud Run Job + Cloud Scheduler + Secret Manager + Cloud Monitoring)
**Confidence:** HIGH (대부분 공식 문서 + Phase 2 실배포 패턴 미러)

## Summary

Phase 05.1는 **코드를 새로 작성하지 않는 100% 운영 배포 페이즈**다. Phase 1에서 완성된 `workers/ingestion/` 코드를 GCP 리소스(Cloud Run Job, Cloud Scheduler, Secret Manager, Monitoring)에 실제 연결해 평일 장 시간 자동 수집을 활성화한다.

연구의 초점은 (1) 정확한 `gcloud` CLI 플래그 이름과 idempotent 호출 패턴, (2) Cloud Scheduler → Cloud Run Job OIDC 호출 계약, (3) 운영 안정성 pitfall(특히 execution overlap과 Scheduler 1-분 granularity)에 있다. Phase 2 `scripts/deploy-server.sh`를 레퍼런스로 100% 패턴 미러하고, 차이점은 `gcloud run deploy` → `gcloud run jobs deploy`, HTTP smoke → Job execute + Supabase row check + Cloud Logging grep으로 교체하는 것뿐이다.

**Primary recommendation:** `scripts/deploy-ingestion.sh`는 `scripts/deploy-server.sh`의 7-section 구조를 글자 단위로 미러하되 (a) Scheduler 등록을 idempotent `describe → create|update`로 구현, (b) IAM/Secret/SA 선행 리소스는 별도 `scripts/setup-ingestion-iam.sh`로 분리, (c) alert policy는 `gcloud alpha monitoring policies create --policy-from-file=ops/alert-ingestion-failure.yaml` + YAML 커밋 방식으로 재현성 확보. Job의 `parallelism=1`은 single execution 내의 task 직렬화일 뿐 **multiple executions 동시 실행을 막지 못한다** — Scheduler 매분 트리거가 장 시간 120s task-timeout을 초과할 경우 overlap 가능성이 있으므로 실측 관찰 포함.

### Script layout — mirror 원칙

Phase 2 `scripts/deploy-server.sh`를 **글자 단위로 미러**해 두 스크립트를 diff하면 차이점만 드러나게 유지.

```
scripts/
├── deploy-server.sh                (Phase 2 기존)
├── smoke-server.sh                 (Phase 2 기존)
├── setup-ingestion-iam.sh          (신규 — 1회성 선행 리소스)
├── deploy-ingestion.sh             (신규 — 반복 배포)
└── smoke-ingestion.sh              (신규 — deploy-ingestion.sh 말미에서 호출)
```

**권장 근거 (Claude's Discretion 결정):** IAM/SA/Secret 생성은 `gcloud run jobs deploy`와 분리하는 것이 안전하다. 매 배포마다 `iam service-accounts add-iam-policy-binding`을 실행하면 idempotent하긴 해도 로그가 어지러워지고 IAM API quota를 낭비한다. 1회성 `setup-ingestion-iam.sh`로 분리하고 README에 "최초 1회 실행" 기재 후, `deploy-ingestion.sh` Section 1 가드에서 해당 SA 존재를 조회(`gcloud iam service-accounts describe`)해 없으면 즉시 exit하며 `setup-ingestion-iam.sh` 실행을 안내.

### Pattern 1: `gcloud run jobs deploy` (create-or-update 통합)

[VERIFIED: cloud.google.com/sdk/gcloud/reference/run/jobs/deploy + Phase 2 deploy-server.sh 패턴]

```bash
gcloud run jobs deploy gh-radar-ingestion \
  --image="${IMAGE}" \
  --region=asia-northeast3 \
  --service-account="gh-radar-ingestion-sa@${PROJECT}.iam.gserviceaccount.com" \
  --cpu=1 \
  --memory=512Mi \
  --task-timeout=120s \
  --max-retries=0 \
  --parallelism=1 \
  --tasks=1 \
  --set-env-vars="^@^SUPABASE_URL=${SUPABASE_URL}@KIS_BASE_URL=https://openapi.koreainvestment.com:9443@LOG_LEVEL=info@APP_VERSION=${SHA}" \
  --set-secrets="KIS_APP_KEY=gh-radar-kis-app-key:latest,KIS_APP_SECRET=gh-radar-kis-app-secret:latest,KIS_ACCOUNT_NUMBER=gh-radar-kis-account-number:latest,SUPABASE_SERVICE_ROLE_KEY=gh-radar-supabase-service-role:latest"
```

**핵심 포인트:**
- `jobs deploy`는 create-or-update **통합 명령** — Job이 없으면 생성, 있으면 revision 업데이트 [CITED: cloud.google.com/sdk/gcloud/reference/run/jobs/deploy].
- `--set-secrets`는 콤마 구분 (Phase 2 `--update-secrets` 대신 `--set-secrets` 사용 — Job은 매 배포 선언적 재설정이 더 명확).
- `--set-env-vars`는 delimiter `^@^` 사용해 URL 안의 `:` 충돌 회피 (Phase 2에서 이미 검증된 패턴).
- `--tasks=1`은 명시적으로 task 1개만 실행 (parallelism과 함께 설정해 의도 분명히).
- `--service-account`는 **런타임 SA** (Scheduler SA 아님).

### Pattern 2: Cloud Scheduler → Cloud Run Job (OIDC)

[VERIFIED: cloud.google.com/run/docs/triggering/using-scheduler + docs/scheduler/docs/http-target-auth]

```bash
JOB_URI="https://${REGION}-run.googleapis.com/apis/run.googleapis.com/v1/namespaces/${PROJECT}/jobs/gh-radar-ingestion:run"
SCHED_SA="gh-radar-scheduler-sa@${PROJECT}.iam.gserviceaccount.com"

# Idempotent: describe → create|update
if gcloud scheduler jobs describe gh-radar-ingestion-scheduler \
     --location=asia-northeast3 >/dev/null 2>&1; then
  gcloud scheduler jobs update http gh-radar-ingestion-scheduler \
    --location=asia-northeast3 \
    --schedule="* 9-15 * * 1-5" \
    --time-zone="Asia/Seoul" \
    --uri="${JOB_URI}" \
    --http-method=POST \
    --oauth-service-account-email="${SCHED_SA}"
else
  gcloud scheduler jobs create http gh-radar-ingestion-scheduler \
    --location=asia-northeast3 \
    --schedule="* 9-15 * * 1-5" \
    --time-zone="Asia/Seoul" \
    --uri="${JOB_URI}" \
    --http-method=POST \
    --oauth-service-account-email="${SCHED_SA}"
fi
```

**인증 방식 결정 — OAuth vs OIDC:**
- Cloud Run Job `:run` 엔드포인트는 Google API(`*.run.googleapis.com`) — **OAuth 액세스 토큰**(`--oauth-service-account-email`)을 사용한다 [VERIFIED: Phase 1 DEPLOY.md 초안 + cloud.google.com/scheduler/docs/http-target-auth "Use OAuth for Google APIs, OIDC for *.run.app"].
- `--oidc-service-account-email`은 **Cloud Run Service의 `*.run.app` 엔드포인트** 호출용 [CITED: cloud.google.com/run/docs/triggering/using-scheduler sample].
- D-07 CONTEXT.md가 "OIDC"라고 기재했지만 Job `:run` API는 OAuth가 정답. Planner는 이 discrepancy를 명확히 잡아 `--oauth-service-account-email`을 사용하고 DEPLOY-LOG에 기록.
- OAuth scope는 기본 `https://www.googleapis.com/auth/cloud-platform` — 대부분의 경우 생략 가능.

### Pattern 3: Service Account 생성 + IAM 바인딩 (idempotent)

[VERIFIED: cloud.google.com/iam/docs/service-accounts-create + Phase 2 실배포]

```bash
# A. SA 생성 (이미 있으면 AlreadyExists 에러 → idempotent하게 무시)
for SA_NAME in gh-radar-scheduler-sa gh-radar-ingestion-sa; do
  gcloud iam service-accounts describe \
    "${SA_NAME}@${PROJECT}.iam.gserviceaccount.com" >/dev/null 2>&1 \
    || gcloud iam service-accounts create "${SA_NAME}" \
         --display-name="gh-radar ${SA_NAME}"
done

# B. Scheduler SA → Job invoker
gcloud run jobs add-iam-policy-binding gh-radar-ingestion \
  --region=asia-northeast3 \
  --member="serviceAccount:gh-radar-scheduler-sa@${PROJECT}.iam.gserviceaccount.com" \
  --role=roles/run.invoker

# C. Runtime SA → Secret accessor (각 secret 단위)
for SECRET in gh-radar-kis-app-key gh-radar-kis-app-secret \
              gh-radar-kis-account-number gh-radar-supabase-service-role; do
  gcloud secrets add-iam-policy-binding "$SECRET" \
    --member="serviceAccount:gh-radar-ingestion-sa@${PROJECT}.iam.gserviceaccount.com" \
    --role=roles/secretmanager.secretAccessor
done
```

**주의:**
- `run jobs add-iam-policy-binding`는 Job이 존재해야 작동 → **deploy 먼저, IAM 바인딩 나중**이 순서. Setup 스크립트는 SA 생성 + Secret 생성만 처리하고, `deploy-ingestion.sh` Section 5 (jobs deploy) 직후 Section 5.5에서 IAM 바인딩 실행하는 방법이 무난.
- `--member=serviceAccount:...`에서 `serviceAccount:` 접두사 필수 (오타 시 `INVALID_ARGUMENT`).
- 기존 바인딩이 있어도 `add-iam-policy-binding`은 idempotent (같은 binding은 중복 저장 안 됨) [CITED: cloud.google.com/iam/docs/granting-changing-revoking-access].

### Pattern 4: Secret Manager 등록 (idempotent)

[CITED: cloud.google.com/secret-manager/docs/creating-and-accessing-secrets]

```bash
create_or_update_secret() {
  local name="$1"
  local value="$2"

  if ! gcloud secrets describe "$name" >/dev/null 2>&1; then
    gcloud secrets create "$name" --replication-policy=automatic
  fi
  printf '%s' "$value" | gcloud secrets versions add "$name" --data-file=-
}

# Setup 스크립트는 환경변수에서 값을 받음 (절대 인라인 금지)
: "${KIS_APP_KEY:?}"
: "${KIS_APP_SECRET:?}"
: "${KIS_ACCOUNT_NUMBER:?}"
: "${SUPABASE_SERVICE_ROLE_KEY:?}"

create_or_update_secret gh-radar-kis-app-key          "$KIS_APP_KEY"
create_or_update_secret gh-radar-kis-app-secret       "$KIS_APP_SECRET"
create_or_update_secret gh-radar-kis-account-number   "$KIS_ACCOUNT_NUMBER"
# Phase 2에서 이미 생성된 secret — 버전 추가 스킵(or 새 값이면 add)
gcloud secrets describe gh-radar-supabase-service-role >/dev/null 2>&1 \
  || create_or_update_secret gh-radar-supabase-service-role "$SUPABASE_SERVICE_ROLE_KEY"
```

**핵심 포인트:**
- `--replication-policy=automatic` 생략 시 대화형 프롬프트 → 스크립트 freeze. 반드시 명시.
- 값은 **환경변수로 주입** — 절대 스크립트 안에 하드코딩 금지, 커맨드라인 인자로 전달 금지(프로세스 리스트에 노출).
- Phase 2에서 이미 `gh-radar-supabase-service-role`을 만들었으면 `:latest` 별칭이 공유됨 — 동일 secret 재사용 OK.

### Pattern 5: Alert Policy — YAML import (재현성 우선)

[VERIFIED: cloud.google.com/sdk/gcloud/reference/alpha/monitoring/policies/create + Medium 2025 가이드]

**Claude's Discretion 결정 근거:** CLI flag 방식(`--condition-filter`, `--condition-threshold-*`)은 MQL/PromQL 표현이 길어질수록 쉘 quoting 지옥에 빠진다. YAML 파일을 리포에 커밋하면 (a) 리뷰 가능, (b) 재생성 시 재현성 보장, (c) 다른 alert 추가 시 패턴 복사 용이.

**`ops/alert-ingestion-failure.yaml`**

```yaml
displayName: "gh-radar-ingestion-failure"
documentation:
  content: "Cloud Run Job gh-radar-ingestion 실행이 최근 5분 내 실패했습니다. Cloud Logging의 'cycle failed' 항목 및 KIS/Supabase 연결 상태를 확인하세요."
  mimeType: text/markdown
combiner: OR
conditions:
  - displayName: "gh-radar-ingestion 실행 실패 발생"
    conditionThreshold:
      filter: |
        resource.type = "cloud_run_job"
        AND resource.labels.job_name = "gh-radar-ingestion"
        AND metric.type = "run.googleapis.com/job/completed_execution_count"
        AND metric.labels.result = "failed"
      comparison: COMPARISON_GT
      thresholdValue: 0
      duration: 0s
      aggregations:
        - alignmentPeriod: 300s
          perSeriesAligner: ALIGN_SUM
alertStrategy:
  autoClose: 3600s
notificationChannels:
  - "${NOTIFICATION_CHANNEL_ID}"   # setup에서 치환
```

  "enabled": true
}
JSON
CHANNEL_NAME=$(gcloud beta monitoring channels create \
  --channel-content-from-file=/tmp/email-channel.json \
  --format='value(name)')

# YAML 내 placeholder 치환 후 policy 생성
sed "s|\${NOTIFICATION_CHANNEL_ID}|${CHANNEL_NAME}|g" ops/alert-ingestion-failure.yaml \

**메트릭 확인:**
- `run.googleapis.com/job/completed_execution_count` with label `result="failed"` — Cloud Run Job 공식 메트릭 [CITED: cloud.google.com/run/docs/monitoring].
- `resource.type="cloud_run_job"`, `resource.labels.job_name="gh-radar-ingestion"` 필터로 다른 Job의 실패에 반응하지 않도록 격리.

### Anti-Patterns to Avoid

- **Scheduler가 Job URI를 `*.run.app`로 착각하는 경우:** Service의 `<service>-<hash>-<region>.a.run.app`과 혼동 금지. Job은 **Google API endpoint**(`*.run.googleapis.com`)로만 호출 가능.
- **`--oidc-service-account-email`을 Job에 사용:** 403 PERMISSION_DENIED가 될 수 있고, 성공해도 의도와 다른 인증 경로. OAuth 사용.
- **`gcloud run jobs replace`**를 플래그 조합 없이 사용: YAML 선언적 관리로 전환해야 하는 비용 발생. D-02가 이미 `deploy`를 채택한 이유.
- **Secret 값을 `--set-env-vars`로 주입:** 메타데이터에 평문 저장됨. 반드시 `--set-secrets`.
- **IAM binding을 프로젝트 단위(`gcloud projects add-iam-policy-binding PROJECT --role=roles/run.invoker`)로**: 최소 권한 위반 + 다른 Cloud Run 리소스에도 invoke 권한 부여. **반드시 리소스 단위**(`gcloud run jobs add-iam-policy-binding gh-radar-ingestion`).

## Don't Hand-Roll

| 문제 | 빌드하지 말 것 | 대신 사용 | 이유 |
|------|----------------|-----------|------|
| Cron 스케줄링 | Node.js `node-cron` 컨테이너 + `min-instances=1` | Cloud Scheduler + Cloud Run Job | 스테이트풀 노드 유지 비용, 다중 인스턴스 경쟁, HA 구성 필요 — 관리형 서비스로 해결 |
| Secret 저장 | `.env` 파일 이미지 내 COPY | Google Secret Manager + `--set-secrets` | 이미지 layer에 secret 노출, rotation 불가, IAM 감사 없음 |
| 실패 알림 | `catch { sendEmail(err) }` SMTP | Cloud Monitoring alert policy | SMTP 설정 유지비, 재시도 로직, dedup 없음 — 메트릭 기반이 표준 |
| 휴장일 캘린더 | 하드코딩 `HOLIDAYS=['2026-...']` 배열 | Phase 1 `isHoliday()` (API 응답 기반) | 이미 구현됨. 캘린더 수동 유지보수 불필요 |
| KIS 토큰 재발급 관리 | 매 실행마다 새로 발급 | Phase 1 `tokenStore` (Supabase `kis_tokens` 테이블 캐싱) | 이미 구현됨. KIS는 1일당 토큰 발급 횟수 제한 — 캐싱 필수 |
| IAM 권한 체크 로직 | Job 코드에서 service account 검증 | IAM 자체 | 선언적, 감사 가능, 권한 부족 시 fail-fast |

**Key insight:** 본 페이즈는 **"custom infra 금지"가 핵심**이다. Phase 1 코드는 건드리지 않고 Google의 관리형 프리미티브(Jobs, Scheduler, Secret Manager, Monitoring)만 엮어야 한다.

## Runtime State Inventory

> 본 페이즈는 **greenfield infra 배포**(기존 운영 리소스 없음)지만, 리네임/참조 영향을 받는 state가 일부 있어 기록.

| 카테고리 | 발견된 항목 | 필요한 조치 |
|---------|-----------|------------|
| Stored data | Supabase `kis_tokens` 테이블(Phase 1) — `service_role`로 읽기/쓰기. 첫 배포 시 Cloud Run Job의 런타임 SA가 service_role_key를 secret으로 주입받아 토큰 row insert. 기존 row 없음(첫 실행). | 조치 없음 — 런타임에 자동 생성 |
| Live service config | **n8n/외부 워크플로 없음.** 기존 Cloud Run service `gh-radar-server`(Phase 2)는 동일 Supabase/동일 CORS — 영향 없음. | 조치 없음 |
| OS-registered state | **없음** (로컬 cron 등록 없음, systemd 없음). | 조치 없음 |
| Secrets/env vars | 신규 3개 secret(`gh-radar-kis-*`) 생성. `gh-radar-supabase-service-role`은 Phase 2 기생성 — **재사용**(동일 key, 버전 공유). 로컬 `.env.deploy` 파일은 `.gitignore` 상태 유지. | Secret 3개 신규 생성, 1개 재사용 |
| Build artifacts / installed packages | Artifact Registry repo `gh-radar`는 Phase 2 생성 — 재사용. 신규 이미지 이름 `ingestion` (path: `gh-radar/ingestion`). 기존 `gh-radar/server:latest` 태그와 충돌 없음. | 조치 없음 — 같은 repo에 다른 image name으로 공존 |

**특이사항:** Phase 2 배포 시점에 Phase 1 Dockerfile 스텁이 이미 존재했고 D-31이 Phase 1 Dockerfile을 node:22-alpine으로 승급시켰다고 기재 — 본 페이즈에서 이미지 재빌드 시 `packages/shared` + `workers/ingestion` 소스가 최신 커밋(`git rev-parse --short HEAD`) 기준으로 빌드되므로 SHA가 변경될 때마다 stocks 테이블에 기록되는 `APP_VERSION` 필드가 자동 추적됨(Phase 2 D-35 승계).

## Common Pitfalls

### Pitfall 1: `parallelism=1`은 execution overlap을 막지 못한다

**무엇이 문제:** Scheduler가 매분 트리거하는데 직전 execution이 120s task-timeout 안에 안 끝나면 **두 개의 execution이 동시에 실행**될 수 있다. `parallelism=1`은 하나의 execution 내부에서 task 직렬화만 제어한다.

**왜 발생:** Cloud Run Jobs의 parallelism은 "execution 내 task 개수" 설정이고, "동시 execution 수" 제어가 아니다 [VERIFIED: cloud.google.com/run/docs/configuring/parallelism — "maximum number of tasks in a job execution that can run in parallel"].

**어떻게 피할지:**
- 실측 관찰: cycle 실행 시간은 Phase 1 DEPLOY.md 추정 "토큰 5s + 28페이지 30s + upsert 10s ≈ 45s"로 120s 버퍼 충분. 실제 p99를 로그로 추적.
- 방어 1단계: upsert는 **멱등**(Phase 1 D-15) — overlap 발생해도 데이터 손상 없음.
- 방어 2단계: KIS rateLimiter(15 req/sec)는 **프로세스 로컬** → 두 execution이 동시 실행되면 30 req/sec 초과 가능. EGW00201 에러 발생 시 Phase 1 retry가 1→2→4s 백오프 후 실패 종료. alert policy가 잡음.
- 방어 3단계 (필요 시, 본 페이즈 범위 밖): future work로 Cloud Scheduler의 `--attempt-deadline`을 짧게 두고 execution 내에서 "이미 실행 중" 감지 시 조기 exit. 본 페이즈는 실측 후 판단.

**경고 신호:** Cloud Logging에서 동일 분(`timestamp` 초 단위 포함)에 `job_run_id`가 서로 다른 `cycle complete`가 2건 이상 → overlap.

### Pitfall 2: OAuth vs OIDC 혼동 — Scheduler가 Job을 호출할 때

**문제:** CONTEXT.md D-07은 "OIDC"로 기재돼 있지만 Cloud Run **Job** API는 **OAuth** 토큰을 요구.

**왜 발생:** Cloud Run *Service*(`*.run.app`)는 OIDC, Cloud Run *Job* `:run` API(`*.run.googleapis.com`)는 OAuth. 표면 문법이 닮아서 헷갈림 [CITED: cloud.google.com/scheduler/docs/http-target-auth].

**어떻게 피할지:** Scheduler 생성 시 반드시 `--oauth-service-account-email`을 사용. Planner는 DEPLOY-LOG와 PLAN.md에 이 구분을 명시적으로 기록.

**검증 신호:** Scheduler 실행 시 Cloud Logging에 `401 UNAUTHENTICATED` 또는 `invalid_audience` → OIDC 사용 중.

### Pitfall 3: KIS "하루 첫 토큰 발급" 지연

**문제:** KIS OAuth2 토큰은 하루 첫 발급 시 10초 이상 지연될 수 있다 (Phase 1 실측 컨텍스트). cron이 09:00:00에 트리거되면 토큰 발급 + 28페이지 ranking + upsert가 120s task-timeout을 근접할 수 있음.

**어떻게 피할지:**
- 09:00 첫 실행은 휴장일 감지 후 일찍 종료되거나(휴장일) 정상 실행(거래일) 하나. 거래일에도 Phase 1 tokenStore가 이후 사이클에서 캐시된 토큰 사용 → 첫 분만 무거움.
- 실측 관찰: Phase 1 tokenStore가 Supabase `kis_tokens` 테이블에 저장 → 콜드스타트 방지. 주말 후 월요일 첫 토큰이 expired면 자동 재발급.
- 필요 시 Scheduler cron을 `58 8-14 * * 1-5` + `* 9-15 * * 1-5`로 분리해 warm-up 실행(08:58, 08:59) 추가 — 본 페이즈 범위 밖 (deferred).

**경고 신호:** Cloud Logging `cycle failed` + `task_timeout_exceeded` 09:00 KST 레코드.

### Pitfall 4: `gcloud alpha monitoring policies create` 실패 모드

**문제:** `alpha` 명령은 quote 이스케이프에 민감하고 inline flag 방식이 filter MQL을 파싱 못 하는 경우가 있다.

**어떻게 피할지:** YAML 파일 방식 사용 (`--policy-from-file=`). JSON도 가능하나 YAML이 리뷰 친화적. 변수 치환은 `envsubst` 또는 `sed`로 사전 처리.

**경고 신호:** `INVALID_ARGUMENT: The request was not valid.` — quote 실패. YAML로 전환.

### Pitfall 5: `gcloud config list` 가드 실패 (non-TTY)

**문제:** CI/bash 환경에서 `gcloud config get-value project`가 경고를 stderr로 출력하며 non-zero exit 가능.

**어떻게 피할지:** `scripts/deploy-server.sh`가 이미 `2>/dev/null || true` 패턴으로 처리 — `deploy-ingestion.sh`도 동일 패턴 복사.

### Pitfall 6: Artifact Registry credentials 만료

**문제:** `gcloud auth configure-docker asia-northeast3-docker.pkg.dev` 설정이 새 터미널/새 계정 전환 시 누락.

**어떻게 피할지:** Section 1 가드에서 `docker credential-gcloud` helper 존재 여부 확인하거나, 최소한 README에 1회성 setup으로 명시. Phase 2 D-39가 이미 이 설정을 선행 조건으로 기재.

### Pitfall 7: `--task-timeout` 단위 혼동

**문제:** gcloud는 `120s`, `2m` 등 시간 접미사 허용하나 숫자만 주면 초 단위로 해석 [CITED: cloud.google.com/sdk/gcloud/reference/run/jobs/deploy]. `120` = 120초 OK. `120m`으로 오타 시 2시간 → 이상한 타임아웃 동작.

**어떻게 피할지:** 명시적 `120s` 사용. 스크립트 상단에 상수 선언 후 치환.

### Pitfall 8: Secret Manager replication-policy 미지정 시 프롬프트

**문제:** `gcloud secrets create SECRET` 단독 실행 시 replication policy 선택 프롬프트가 떠 스크립트가 hang.

**어떻게 피할지:** `--replication-policy=automatic` 반드시 명시 (다중 리전 복제 원치 않으면 `--replication-policy=user-managed --locations=asia-northeast3`).

# Section 5: Deploy Job (D-02, D-03, D-10, D-11)
# ═══════════════════════════════════════════════════════════════
echo "▶ gcloud run jobs deploy..."

# ═══════════════════════════════════════════════════════════════

    --schedule="* 9-15 * * 1-5" \

    --uri="$JOB_URI" \
    --http-method=POST \

  gcloud scheduler jobs create http "$SCHED" \
    --location="$REGION" \

### Example 2: smoke-ingestion.sh 핵심 검증 (D-12, D-13)

```bash
#!/usr/bin/env bash
set -uo pipefail
JOB="${1:-gh-radar-ingestion}"
REGION="${2:-asia-northeast3}"
PASS=0; FAIL=0; declare -a FAILED

check() {
  local name="$1"; shift
  echo -n "  $name ... "
  if "$@" >/dev/null 2>&1; then echo PASS; PASS=$((PASS+1));
  else echo FAIL; FAIL=$((FAIL+1)); FAILED+=("$name"); fi
}

# INV-1: Job을 즉시 실행하고 완료까지 대기
check "INV-1 jobs execute --wait exit 0" \
  gcloud run jobs execute "$JOB" --region="$REGION" --wait

# INV-2: Cloud Logging에서 최근 5분 내 성공 패턴 검증 (cycle complete OR non-trading day)
check "INV-2 logs: cycle complete|non-trading day" bash -c "
  gcloud logging read '
    resource.type=\"cloud_run_job\"
    AND resource.labels.job_name=\"$JOB\"
    AND timestamp>=\"\$(date -u -v-5M +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -d \"-5 minutes\" +%Y-%m-%dT%H:%M:%SZ)\"
    AND (jsonPayload.msg=\"cycle complete\" OR jsonPayload.msg=\"non-trading day detected, exiting\")
  ' --limit=5 --format='value(jsonPayload.msg)' | grep -Eq '(cycle complete|non-trading day)'
"

# INV-3: Cloud Logging에서 최근 5분 실패 0건
check "INV-3 logs: no cycle failed / EGW00201" bash -c "
  COUNT=\$(gcloud logging read '
    resource.type=\"cloud_run_job\"
    AND resource.labels.job_name=\"$JOB\"
    AND timestamp>=\"\$(date -u -v-5M +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -d \"-5 minutes\" +%Y-%m-%dT%H:%M:%SZ)\"
    AND (jsonPayload.msg=\"cycle failed\" OR textPayload:\"EGW00201\")
  ' --limit=1 --format='value(timestamp)' | wc -l)
  [ \"\$COUNT\" -eq 0 ]
"

# INV-4: 장 시간이면 stocks 테이블 max(updated_at) < 60s, 휴장일이면 skip
# Claude's Discretion 결정: service_role_key는 .env.deploy에서 로드, anon SELECT 대신 명시적 service_role 사용(권한 일관성)
check "INV-4 Supabase stocks freshness or holiday" bash -c "
  : \${SUPABASE_URL:?}; : \${SUPABASE_SERVICE_ROLE_KEY:?}
  REST_URL=\"\${SUPABASE_URL}/rest/v1/stocks?select=updated_at&order=updated_at.desc&limit=1\"
  MAX_TS=\$(curl -fsS \"\$REST_URL\" \
    -H \"apikey: \$SUPABASE_SERVICE_ROLE_KEY\" \
    -H \"Authorization: Bearer \$SUPABASE_SERVICE_ROLE_KEY\" \
    | jq -r '.[0].updated_at')
  if [ -z \"\$MAX_TS\" ] || [ \"\$MAX_TS\" = null ]; then exit 1; fi
  # 휴장일이면 INV-2가 non-trading day 감지 → 여기서는 WARN만, 최종 exit은 INV-2에 위임
  AGE=\$(( \$(date +%s) - \$(date -d \"\$MAX_TS\" +%s 2>/dev/null || date -jf '%Y-%m-%dT%H:%M:%S' \"\${MAX_TS%%.*}\" +%s) ))
  [ \"\$AGE\" -lt 120 ] || gcloud logging read '
    resource.type=\"cloud_run_job\" AND resource.labels.job_name=\"'\"$JOB\"'\"
    AND jsonPayload.msg=\"non-trading day detected, exiting\"
  ' --limit=1 --format='value(timestamp)' --freshness=10m | grep -q .
"

# INV-5: Scheduler 존재 + ENABLED 상태
check "INV-5 scheduler enabled" bash -c "
  STATE=\$(gcloud scheduler jobs describe gh-radar-ingestion-scheduler --location=\"$REGION\" --format='value(state)')
  [ \"\$STATE\" = ENABLED ]
"

echo ""
echo "PASS: $PASS  FAIL: $FAIL"
[ "$FAIL" -eq 0 ] || { echo "Failed: ${FAILED[*]}"; exit 1; }
echo "✅ All smoke invariants passed"
```

## Open Questions

1. **Scheduler redeploy 시 일시 중단 정책**
   - 아는 것: `gcloud scheduler jobs pause/resume`으로 제어 가능. Job revision 업데이트는 수 초 내 완료.
   - 불확실한 것: 업데이트 중 trigger 발생 시 이전 revision 실행 vs 새 revision 실행 — Cloud Run Job의 latest revision 정책.
   - 권장: Phase 05.1 범위에서는 `pause` 생략하고 실측. Upsert가 멱등이라 overlap이나 old-revision 실행도 데이터 오염 없음. Deferred로 명시.

2. **장 외 시간 Scheduler 실행 비용**
   - 아는 것: cron `* 9-15 * * 1-5` = 420분/주 × Job 실행(30~45s) ≈ 315분 CPU-시간/주. Cloud Run Job 무료 tier 내.
   - 불확실한 것: 휴장일(공휴일) 감지 후 early-exit 실행도 Cloud Run Job에서 billable 초로 계산되나? (probable: 최소 실행 시간 기본 1~2s에는 과금 그 이상).
   - 권장: 배포 첫 주 말 GCP 빌링 확인 후 필요 시 조치. 본 페이즈 범위 밖.

3. **KIS 주말 월요일 첫 토큰 지연 실측**
   - 아는 것: Phase 1 tokenStore 캐싱. 주말은 Scheduler 비활성(`1-5`만).
   - 불확실한 것: 월요일 09:00 첫 실행 시 토큰 만료 여부. KIS 토큰 TTL 공식 문서 기준 ~24h → 금요일 15:59 발급 토큰은 월요일까지 expired.
   - 권장: 월요일 09:00 첫 실행 후 Cloud Logging에서 "token refresh" 로그 관찰. 본 페이즈 smoke는 배포 당일 기준이라 직접 실증 불가 — 배포 후 첫 월요일 09:05에 확인.

### Known Threat Patterns for GCP Operational Deployment

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Secret leak via build log | Info Disclosure | Docker `--build-arg` 금지 for secrets. `--set-secrets` runtime injection만 사용 |
| IAM overprivilege — project-wide Run Invoker | Elevation of Privilege | 리소스 단위 바인딩(`gcloud run jobs add-iam-policy-binding`) |
| Scheduler trigger replay attack | Spoofing | OAuth SA 필수 + Cloud Run Job invoker role로 제한 |
| Secret rotation 없음 | Tampering/Info Disclosure | Secret Manager versions + `:latest` 별칭 정책 — 본 페이즈는 rotation 자동화 미포함(v2) |
| 토큰 payload 로그 출력 | Info Disclosure | pino redact (Phase 1 완성) |
| Supabase service_role key 네트워크 노출 | Info Disclosure | Cloud Run Job 컨테이너 내부에서만 사용, outbound만 Supabase REST. Ingress 없음 |
| `:latest` tag 미스매치 | Tampering | 배포마다 `:SHA` 태그 우선, `:latest`는 추적용. Job은 `:SHA`로 바인딩 |

**SBOM/이미지 취약성:** Phase 2 가치안정화 단계에서 `docker scout`/`trivy` 스캔 도입 가능 — 본 페이즈 범위 밖 (deferred).
