## Phase Boundary

Phase 1에서 완성한 `workers/ingestion/` 코드를 GCP에 실제 배포하여 평일 장 시간 동안 KIS API → Supabase stocks 테이블 자동 폴링이 동작하게 만든다.

**포함:**
- Artifact Registry에 ingestion 컨테이너 이미지 빌드/푸시 (`asia-northeast3`, `gh-radar` repo 재사용)
- KIS/Supabase secret 4종을 Google Secret Manager에 등록
- Cloud Run Job(`gh-radar-ingestion`) 생성 + 환경변수/시크릿 바인딩
- 전용 Cloud Scheduler service account(`gh-radar-scheduler-sa`) 생성 + `roles/run.invoker` 부여
- Cloud Scheduler 단일 cron(`* 9-15 * * 1-5`, KST) 등록 → Job run API OIDC 호출
- 배포 자동화: `scripts/deploy-ingestion.sh`(server 패턴 미러) + `scripts/smoke-ingestion.sh`
- Cloud Logging 기반 실패 알림(이메일 채널) 설정
- DEPLOY-LOG에 실배포 결과/명령 기록

**제외:**
- 코드 변경(Phase 1 ingestion 로직은 그대로). 환경변수 추가나 종료 코드 정책 등 운영 보강만 허용.
- GitHub Actions/CI 자동 배포 (수동 스크립트 유지 — Phase 2 D-36 승계, v2)

<decisions>
## Implementation Decisions

### 배포 스크립트 + Cloud Run Job 설정
- **D-01:** `scripts/deploy-ingestion.sh`는 `scripts/deploy-server.sh`와 동일한 섹션 구조로 작성 — Section 1 gcloud guard → 2 변수 → 3 build(`--platform=linux/amd64` + `GIT_SHA` 주입) → 4 push(`:SHA` + `:latest`) → 5 `gcloud run jobs deploy` → 6 Scheduler 등록(idempotent) → 7 smoke. 두 스크립트를 나란히 두면 한눈에 같은 패턴임을 확인할 수 있도록 한다.
- **D-02:** `gcloud run jobs deploy gh-radar-ingestion` 사용(생성+업데이트 통합 명령). `jobs replace` 대신 평문 플래그로 환경변수/시크릿/리소스를 명시. Scheduler는 별도 단계에서 `gcloud scheduler jobs describe` → 없으면 `create`, 있으면 `update`로 idempotent.
- **D-03:** Job 리소스/실행 옵션은 DEPLOY.md 초안 그대로: `--cpu=1 --memory=512Mi --task-timeout=120s --max-retries=0 --parallelism=1 --region=asia-northeast3`. KIS 토큰 발급(~5s) + 28페이지 ranking(~30s) + upsert(~10s) 기준 120s 충분, 실패는 다음 cron 사이클이 자연스럽게 재시도(Phase 1 D-14 승계).
- **D-04:** 이미지 태그 정책은 server와 동일 — `git rev-parse --short HEAD` 태그 + `:latest` 동시 푸시. Artifact Registry repo는 server와 동일한 `asia-northeast3-docker.pkg.dev/${PROJECT}/gh-radar/ingestion:${SHA}`(이미지 이름만 `ingestion`).
- **D-05:** 컨테이너는 non-root(Phase 1 Dockerfile 패턴 유지). gcloud 가드는 `EXPECTED_CONFIG=gh-radar` + `GCP_PROJECT_ID` 필수 — server 스크립트와 동일.

### Cloud Scheduler 전략
- **D-06:** Scheduler는 **단일 작업** `gh-radar-ingestion-scheduler`만 생성. cron `* 9-15 * * 1-5`(KST, 평일 09:00~15:59 매 분, 60회/일). 09:00~15:30 정규장 + 종가 15:30 + EOD 15:35 모두 매분 호출에 자연 포함되므로 Phase 1 D-10의 별도 종가 cron을 분리 등록할 필요가 없다. 휴장일/장 외 분에는 `isHoliday()` 가드가 즉시 exit하므로 비용/로그 노이즈 허용.
- **D-07:** Scheduler 호출 인증은 **전용 service account + 최소 권한** — `gh-radar-scheduler-sa@${PROJECT}.iam.gserviceaccount.com` 생성 후 `roles/run.invoker`를 Job 단위(또는 프로젝트 단위)로 부여. Scheduler는 OAuth/OIDC 헤더로 위 SA 신원을 사용해 `https://${REGION}-run.googleapis.com/apis/run.googleapis.com/v1/namespaces/${PROJECT}/jobs/gh-radar-ingestion:run`을 POST.
- **D-08:** time zone은 `Asia/Seoul`. cron 표현은 `* 9-15 * * 1-5`로 명시.

### Secret / 환경변수 명세
- **D-09:** 4종 전부 Google Secret Manager로 관리:
  - `gh-radar-kis-app-key` ← `KIS_APP_KEY`
  - `gh-radar-kis-app-secret` ← `KIS_APP_SECRET`
  - `gh-radar-kis-account-number` ← `KIS_ACCOUNT_NUMBER`
  - `gh-radar-supabase-service-role` ← `SUPABASE_SERVICE_ROLE_KEY` (Phase 2와 동일 시크릿 재사용 — `:latest` 별칭 공유)

  Cloud Run Job 런타임 service account에 각 시크릿의 `roles/secretmanager.secretAccessor` 부여.
- **D-10:** 평문 환경변수(`--set-env-vars`):
  - `SUPABASE_URL` (server와 동일 값)
  - `KIS_BASE_URL=https://openapi.koreainvestment.com:9443`
  - `LOG_LEVEL=info`
  - `APP_VERSION=${SHA}` (Phase 2 D-35 패턴 승계, 로그 추적용)
- **D-11:** Cloud Run Job의 런타임 service account는 별도 생성 — `gh-radar-ingestion-sa@${PROJECT}.iam.gserviceaccount.com`. 위 4개 시크릿 접근 권한만 보유. **Scheduler SA(`gh-radar-scheduler-sa`)와 분리** — invoker(scheduler) vs runtime(job) 책임 분리.

### 검증 / 모니터링
- **D-12:** `scripts/smoke-ingestion.sh` 자동화 — deploy 스크립트 마지막 단계로 호출. 동작:
  1. `gcloud run jobs execute gh-radar-ingestion --region=asia-northeast3 --wait` → exit code 0 검증
  2. Supabase REST API로 `stocks` count 및 `MAX(updated_at)` 조회 — 60초 이내면 PASS, 아니면 WARN
  3. 최근 5분 Cloud Logging에서 `cycle complete` 또는 `non-trading day detected` 패턴 grep — 둘 중 하나 매치 필수
  4. `EGW00201`(rate limit) 또는 `cycle failed` 0건 검증
- **D-13:** 휴장일/장 외 시간에 smoke 실행 시 step 2의 updated_at 검증은 SKIP(holiday early-exit) — `isHoliday=true` 또는 `non-trading day detected` 로그가 보이면 PASS로 처리.
- **D-14:** 운영 중 실패 알림은 **Cloud Monitoring alert policy** 1개로 시작:
  - 조건: Cloud Run Job execution 실패(metric `run.googleapis.com/job/completed_execution_count` with `result="failed"`) 5분 윈도우에 1건 이상
  - 알림 채널: 이메일(notification channel 1개 등록 — 사용자 본인 이메일)
  - PagerDuty/Slack은 v2
- **D-15:** **DEPLOY-LOG.md** 작성 — 실배포 시 사용한 명령/이미지 SHA/Scheduler 이름/SA 이름/Secret ID/alert policy ID를 기록. Phase 2 패턴 그대로(`.planning/phases/02-backend-api/02-DEPLOY-LOG.md` 참조).

### Claude's Discretion
- Service account 생성/IAM 부여 명령을 deploy 스크립트에 넣을지, 별도 1회성 setup 스크립트(`scripts/setup-ingestion-iam.sh`)로 분리할지는 planner 판단(idempotency만 보장).
- Smoke 단계에서 Supabase 조회를 service_role REST 호출로 할지, anon SELECT(stocks가 anon SELECT 허용)로 할지 planner 결정.
- alert policy를 `gcloud alpha monitoring policies create` CLI로 만들지 콘솔 1회 설정 후 export YAML을 리포에 보관할지 planner 결정.
- Scheduler가 있는 phase에서 Job을 redeploy할 때 trigger를 일시 중단할지(`gcloud scheduler jobs pause`) planner 판단.

## Existing Code Insights

### Reusable Assets
- **`scripts/deploy-server.sh`**: 거의 그대로 미러링 가능. 변경점은 ① `gcloud run deploy` → `gcloud run jobs deploy`, ② Service URL/smoke URL 호출 → Job execute + Supabase/Logging 확인, ③ Scheduler 등록 단계 추가.
- **`scripts/smoke-server.sh`**: HTTP 헬스 확인 패턴 → Job 실행 + DB row 확인으로 변형.
- **`workers/ingestion/Dockerfile`**: 그대로 사용. 변경 불필요.
- **`workers/ingestion/src/index.ts` exit code 정책**: 이미 `catch → process.exit(1)` 구현 → Cloud Run Job retry/alert 시스템과 호환.

### Established Patterns
- **gcloud guard 패턴(server)**: `EXPECTED_CONFIG=gh-radar` + `GCP_PROJECT_ID` 필수 검증을 deploy-ingestion에도 동일 적용 — Phase 2 D-39 잠금.
- **시크릿 명명**: server는 `gh-radar-supabase-service-role`. ingestion도 `gh-radar-{kind}-{name}` 컨벤션 따라 일관성 유지.
- **APP_VERSION=git SHA 주입**: 운영 로그에서 어느 커밋인지 추적 가능 — Phase 2 D-35 승계.

### Integration Points
- Cloud Run Job 실행 시 **server와 동일한 Supabase 인스턴스**에 쓰기 → SCAN-01 등 Scanner UI가 자동으로 새 데이터를 읽게 됨.
- Phase 5 Scanner UI는 이미 `/api/scanner` 폴링 60s + Vercel 배포 완료 → 이번 페이즈 완료 시 E2E 데이터 흐름 첫 활성화.

## Specific Ideas

- **단일 Scheduler 단순화**: 사용자가 명시적으로 "15:35 자동 포함되니 1개로 단순화"를 선택. 운영 단순함을 우선.
- **Service account 분리**: Scheduler invoker(`gh-radar-scheduler-sa`)와 Job runtime(`gh-radar-ingestion-sa`)을 분리해 책임 명확화 + 침해 시 영향 범위 축소.
- **DEPLOY.md는 보존하되 deprecated 표기**: 새 deploy-ingestion.sh가 사실상 대체하지만 명령어 참조용으로 남김.
