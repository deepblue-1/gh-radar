## Phase Boundary

평일 장중 (09:00~15:30 KST) 동안 활성 거래 종목 (~1,898) 의 현재가/등락/거래량/OHLC/상한가·하한가/시가총액을 **키움 REST API 만으로** 매분 수집하여 `stock_quotes`, `top_movers`, `stock_daily_ohlcv` 테이블에 갱신한다. 본 phase 완료 후 **KIS 호출은 시스템 전체에서 0** 이 된다.

### 매분 cycle 구조 (intraday-sync 워커, cron `* 9-15 * * 1-5` KST)

**STEP 1 — ka10027 페이지네이션 1회 (~700ms)**
- 응답: 활성 1,898 종목의 `cur_prc / pred_pre / flu_rt / now_trde_qty / stk_cd / stk_nm`
- 갱신:
  - `top_movers` 매분 재구성 (등락률 상위 60, stale cleanup)
  - `stock_quotes` UPSERT: `price / change_amount / change_rate / volume / name / market` (활성 1,898 행)
  - `stock_daily_ohlcv` 오늘자 row UPSERT: `close / change_amount / change_rate / volume` (활성 1,898 행)
  - `tradeAmount = volume × close` 근사값 함께 갱신 (D-Q4)

**STEP 2 — ka10001 hot set N회 (~10.5초, 200 ∪ watchlist ≈ 250 calls × 1/24s)**
- hot set = (STEP1 산출 등락률 상위 200) ∪ (전체 watchlists.stock_code unique)
- 응답: 종목별 `open_pric / high_pric / low_pric / cur_prc / upl_pric / lst_pric / mac`
- 갱신:
  - `stock_quotes` UPSERT: `open / high / low / upper_limit / lower_limit / market_cap` (hot set 만, 250 행)
  - `stock_daily_ohlcv` 오늘자 row UPSERT: `open / high / low` (hot set 만)

### 스코프 IN
- 신규 워커 `workers/intraday-sync` (Cloud Run Job + Cloud Scheduler)
- 키움 OAuth2 client + ka10027 페이지네이션 + ka10001 단일 종목
- 인프라: VPC `gh-radar-vpc` + Cloud NAT + Static External IP 1개 (Cloud Run Job + Cloud Run service 공유)
- Cloud Run **service** (`server/`) 의 outbound 도 동일 VPC/NAT/IP 경유하도록 재배포 (Q6 결정)
- `server/src/kis/*` → `server/src/kiwoom/*` 로 교체 (Q2 결정 — server on-demand 도 키움 동기 호출)
- KIS ingestion 워커 완전 폐기 (Cloud Scheduler / Job / SA / Secret / `kis_tokens` 테이블 정리)
- 마이그레이션: `kis_tokens` drop + `kiwoom_tokens` create (Q5 결정)
- `Stock.tradeAmount` 정책 변경: 정확값 → 근사값 (D-Q4), 코드 주석 갱신

### 스코프 OUT (의도적 제외)
- 분봉(minute candle) 시계열 신규 테이블 — Phase 09.x 후보

## Implementation Decisions

### Strategic — KIS 완전 대체 (사전 컨텍스트 D-NEW-01 ~ D-NEW-08)

- **D-01:** **KIS 사용처 0** — workers/ingestion 디렉터리 폐기, server/src/kis/ 폐기, packages/shared 의 KIS 타입(`KisRankingRow`, `KisInquirePriceRow`) 제거, supabase `kis_tokens` 테이블 drop migration.
- **D-02:** **타 워커 무변경 확인** — master-sync (cron `10 8`), candle-sync (`30 17` + `10 8`), news-*, discussion-* 모두 영향 없음.
- **D-03:** **신규 워커 `workers/intraday-sync`** — candle-sync 1:1 mirror 구조 (Dockerfile + config/logger/retry + services/supabase + kiwoom client + pipeline + index.ts).
- **D-04:** **인프라** — Direct VPC Egress + Cloud NAT + Static External IP **1개** (Q6 결정). Cloud Run Job (intraday-sync) + Cloud Run service (server) 공유.
- **D-05:** **EOD overlay 보존** — candle-sync 17:30 가 `stock_daily_ohlcv` 오늘자 row 의 **모든 OHLCV 컬럼을 공식값으로 덮어쓰기** (intraday 의 hot-set 외 종목 open/high/low 가 STEP1 close 만으로 부정확해도 EOD 에서 정정).

### STEP 1 — ka10027 페이지네이션

- **D-06:** `mrkt_tp="000"` (KOSPI+KOSDAQ 통합), `sort_tp="1"` (상승률 정렬), `cont-yn` + `next-key` 헤더 페이지네이션. 10 페이지 × 200 행 = 1,800~1,900 종목.
- **D-07:** `updown_incls="1"` (등락 미발생 종목 포함). 거래정지/관리는 응답에서 자연 제외.
- **D-08:** `now_trde_qty` 누적성 가설 (KIS `acml_vol` 컨벤션) — Plan 단계 fixture 캡처로 확정.
- **D-09:** `cur_prc` 부호 prefix (`+`, `-`, 빈값) 는 등락 방향 마커, 가격은 절댓값 사용. `pred_pre`, `flu_rt` 도 동일 패턴. `parseSignedPrice(s): { price, direction }` mapper.
- **D-10:** `stk_cd` 의 `_AL` 접미사 strip → 6자 단축코드. mapper 에서 `/^\d{6}$/` 검증, 실패 시 throw.

### STEP 2 — ka10001 Hot Set (Q1 결정)

- **D-11:** **Hot set 정의** = (STEP1 응답에서 추출한 등락률 상위 200) ∪ (`watchlists.stock_code` unique 전체). 중복 제거 후 ka10001 순차 호출.
- **D-12:** **호출 시간 예산** — ka10001 24 req/s, 250 calls = ~10.5초. STEP1 (~700ms) + STEP2 (~10.5s) + RPC/network = **총 ~13초**. 60초 cycle 매우 여유.
- **D-13:** **watchlist 조회** — `supabase.from('watchlists').select('stock_code').limit(...)` 로 unique 종목 추출. 매 cycle 1회 (~수십ms).
- **D-14:** **Hot set 외 ~1,700 종목** — open/high/low/upper_limit/lower_limit/market_cap 갱신 안 됨 → 어제값 또는 NULL 유지, EOD 17:30 에 일괄 정정. 의도된 동작.
- **D-15:** ka10001 응답 `mac` (시가총액) 단위 — Plan 단계 fixture 캡처. 가설: 억원(KIS `hts_avls` 와 동일). 정수형/× 1억 변환 정확 검증 필요.
- **D-16:** ka10001 의 `cur_prc / pred_pre / flu_rt` 도 ka10027 와 동일하게 부호 prefix 처리.

### server on-demand 종목 상세 (Q2 결정)

- **D-17:** **`server/src/kis/inquirePrice.ts` → `server/src/kiwoom/inquirePrice.ts` 로 교체** — server 도 키움 client 동기 호출. `inquirePriceToQuoteRow` (server/src/mappers/stock.ts) 의 KIS 필드 매핑을 ka10001 필드로 재매핑.
- **D-18:** **server static IP 공유** — Cloud Run service 도 `--vpc-egress=all-traffic --network=gh-radar-vpc --subnet=...` 옵션으로 outbound 가 intraday-sync 와 동일 NAT/Static IP 경유. 키움 화이트리스트에 IP 1개만 등록.
- **D-19:** **server 토큰 전략** — Q5 결정에 따라 Supabase `kiwoom_tokens` 테이블 공유 (worker 와 동일 캐시 row). 동시성: 만료 시 worker/server 중 먼저 refresh 하는 쪽이 새 토큰을 row 에 INSERT/UPDATE, 다른 쪽은 다음 호출 시 fresh row 를 SELECT.

### stock_quotes 누적 정책 (Q3 결정)

- **D-20:** **활성 1,898 종목 매분 누적** — 매 cycle ka10027 응답 종목만 UPSERT (`onConflict: code`). 비활성/관리/거래정지 종목은 마지막 cycle 값 유지 (stale cleanup 없음).
- **D-21:** **stale cleanup 없음** — top_movers 와 달리 stock_quotes 는 영구 누적. 비활성 전환 종목의 마지막 가격이 그대로 노출되어도 UI 가 구분 안 함 (DB 의 `updated_at` 으로 신선도 판단 가능).
- **D-22:** **trade_amount 갱신 빈도** — STEP1 에서 1,898 종목 × `volume × close` 근사값 매분 UPSERT (활성 종목 전체).

### trade_amount 정책 변경 (Q4 결정)

- **D-23:** **`Stock.tradeAmount = volume × close` 근사값 채택** — 트레이딩 시그널(대량거래 식별) 용도 명시. 정확도 ~3-7% 오차 (분봉 가격 변동에 따른 평균체결단가 차이). 회계/리포팅 용도 아님.
- **D-24:** **코드 주석 갱신** — `workers/intraday-sync/src/pipeline/map.ts` 와 (폐기 전) `workers/ingestion/src/pipeline/map.ts:5` 의 "근사값은 허용하지 않음" 주석을 새 정책으로 교체. 사유와 사용처 명시 ("트레이딩 시그널 용도, 정확값 endpoint 미도입").
- **D-25:** **UI 영향** — `stock_quotes.trade_amount` schema 유지 (DROP 없음). UI 가 표시하는 값이 근사값임을 명시할 필요 없음 (사용자 결정).

### 키움 OAuth Token 캐시 (Q5 결정)

- **D-26:** **Supabase `kiwoom_tokens` 테이블** — kis_tokens 패턴 1:1 mirror. `(token_type, access_token, expires_at, fetched_at)` schema. RLS REVOKE anon/authenticated, GRANT service_role only.
- **D-27:** **24h TTL refresh** — 만료 ≤ 5분 남았으면 재발급. worker/server 동시 만료 시 race condition 은 마지막 INSERT 가 승 (idempotent — 키움이 동일 토큰 반환하거나 새 토큰 발급, 둘 다 동작 가능).
- **D-28:** **kiwoom_tokens 마이그레이션** — Wave 0 에 kis_tokens drop + kiwoom_tokens create 통합 마이그레이션.

### 인프라 — Static IP 공유 (Q6 결정)

- **D-29:** **VPC/NAT/Static IP 단일 인스턴스** — `gh-radar-vpc` + subnet `gh-radar-subnet-an3` + Cloud Router `gh-radar-router` + Cloud NAT `gh-radar-nat` + Static IP `gh-radar-static-ip` (asia-northeast3). Cloud Run Job (intraday-sync) + Cloud Run service (server) 둘 다 동일 VPC connector.
- **D-30:** **server 재배포 필요** — 현재 server 는 default network 사용. 본 phase 에서 `gh-radar-vpc` 로 마이그레이션 + 재배포.
- **D-31:** **키움 IP 화이트리스트 등록** — Static IP 1개만 등록 (사용자 액션). Phase 09.1-06-PLAN [BLOCKING] 단계에 명시.
- **D-32:** **비용** — Cloud NAT $1~3/월 (저 traffic), Static IP $1.46/월 (사용 중 무료, idle 만 과금). 실측 추정 < $5/월.

### RPC 정책 (이전 RESEARCH §3 계승 + 확장)

- **D-33:** **`intraday_upsert_close(jsonb)` RPC** — STEP1 의 close/change/volume UPSERT. 09:00 첫 cycle 시 오늘자 row 부재 처리 → INSERT 시 `open=high=low=close=cur_prc` 임시 초기화 + `volume=now_trde_qty` + `trade_amount=cur_prc × now_trde_qty`. 이후 cycle 은 `high=GREATEST(high, cur_prc)`, `low=LEAST(low, cur_prc)`, `close=cur_prc`, `volume=now_trde_qty` UPDATE.
- **D-34:** **`intraday_upsert_ohlc(jsonb)` RPC (신규)** — STEP2 의 hot set OHLC + 상한가/하한가/시가총액 UPSERT. ka10001 응답이 종목별 정확한 open/high/low 제공하므로 STEP1 의 임시값 덮어쓰기.
- **D-35:** **두 RPC 모두 `REVOKE EXECUTE FROM PUBLIC, anon, authenticated; GRANT EXECUTE TO service_role`** — MEMORY 의 feedback_supabase_rpc_revoke 룰 준수.
- **D-36:** **stock_quotes UPSERT 는 client-side** — Supabase JS client 의 `.upsert(rows, { onConflict: 'code' })` 직접 호출. RPC 미사용 (단순 컬럼 UPSERT).

### Conflict Avoidance (이전 D-17~D-19 계승)

- **D-37:** **테이블 분리** — intraday-sync 는 `stock_quotes` / `top_movers` / `stock_daily_ohlcv` 만 건드림. `stocks` (마스터) 무변경.
- **D-38:** **시간대 분리** — candle-sync EOD `30 17` 와 intraday `* 9-15` 시간대 겹치지 않음. recover `10 8` 도 별도.
- **D-39:** **API bucket 분리** — KIS 폐기로 단일 키움 bucket 만 남음. 키움 IP-단위 통합 rate-limit 가설 (공식 미공개, Plan 단계 실측 검증).
