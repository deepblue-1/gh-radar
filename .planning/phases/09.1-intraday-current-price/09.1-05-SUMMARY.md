## Decisions Made

- **parseMac 의 곱셈 상수 (×10^8) 가설 잠금**: D-15 의 '억원' 가설을 `Math.round(n * 100_000_000)` 한 줄로 구현. 단위 확정 책임은 Plan 06 의 production smoke. 잘못된 단위 발견 시 본 함수 1줄 + 본 plan 의 mapOhlc.test.ts expectation (`4209000 * 100_000_000`) 동시 갱신. 본 plan 의 fixture 가 placeholder 라는 점 SUMMARY 명시.
- **stk_cd 6자 검증 (ka10001 vs ka10027 차이)**: ka10027 응답은 `_AL` suffix 있어 stripAlSuffix 통과해야 하지만, ka10001 응답은 처음부터 6자 단축코드. /^\\d{6}$/ 정규식 직접 검증 후 throw. Plan 04 의 stripAlSuffix 미사용.
- **open/high/low 가 parseSignedPrice 의 .value 만 사용**: direction 정보 폐기. ka10001 응답의 `+70000` 부호 prefix 는 ka10027 와 동일하게 등락 방향 마커지만 OHLC 자체는 양수 절댓값. parseSignedPrice 가 절댓값 보장하므로 .value 만 추출.
- **upperLimit/lowerLimit 에 Math.abs() fail-safe**: parseOptionalSignedNumber 는 부호 유지 (pred_pre/flu_rt 패턴 재사용) 이지만, 상한가/하한가 컬럼은 음수 가능성 없음 — 절댓값 강제. 응답이 양수든 음수든 결과 동일.
- **computeHotSet 빈 watchlist 정상 동작 (T-09.1-18 mitigate)**: D-11 정책 그대로 — watchlists 0 rows 시 top N 만 반환 + Plan 06 가 hot set size 0 시 STEP2 skip (ka10001 0 calls).
- **Promise.allSettled 채택 (Promise.all 아님)**: T-09.1-16 mitigate. forEach + status='fulfilled' 분류로 successful/failures 분리. Promise.allSettled 가 입력 순서 보존하므로 `failures[idx].code = codes[idx]` 매핑 안전.
- **fetchHotSet 의 rate limiter 가 각 호출 직전 acquireKiwoomRateToken**: T-09.1-17 mitigate. Promise.allSettled 가 ~250 호출 동시 시작하지만 token bucket 이 자연 직렬화 (capacity 24 + 24 refill/s). 250 calls = ~10.5초 예산 보장.
- **fetchHotSet test 에서 rateLimiter 재설정 패턴**: beforeEach 마다 `configureKiwoomRateLimiter({ capacity: 1000, refillRatePerSec: 1000 }) + resetKiwoomRateLimiter()` 호출. Plan 04 의 24 req/s default 가 단위 테스트 직렬화 대기 ↑ 방지. Plan 04 가 export 한 두 함수 정확히 활용.
