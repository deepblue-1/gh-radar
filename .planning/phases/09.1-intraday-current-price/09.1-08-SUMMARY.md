## Decisions Made

- **Static IP 1개 공유 (CONTEXT D-29/D-32)**: gh-radar-static-ip 단일 reserved external IP 가 Cloud Run Job (intraday-sync) + Cloud Run service (server) 양쪽 outbound 의 NAT egress. 키움 IP whitelist 1개만 등록하면 됨. 비용 < $5/월. Cloud NAT 의 `--nat-external-ip-pool=gh-radar-static-ip` 가 IP 바인딩.
- **compute.networkUser 3 바인딩 일괄 처리 (RESEARCH §4.7)**: setup-intraday-sync-iam.sh Section 9 가 Cloud Run Service Agent + intraday-sync SA + server default compute SA 에 모두 `roles/compute.networkUser` 바인딩. Direct VPC Egress 의 hard requirement. deploy 스크립트는 별도 IAM 액션 불필요.
- **Scheduler cron `* 9-15 * * 1-5` + task-timeout=60s (RESEARCH §9.4)**: CONTEXT D-12 의 매분 cycle (STEP1 ~700ms + STEP2 ~10.5s = 총 ~13s) 가 60초에 매우 여유. candle-sync 의 300s 대비 5배 짧음. 매분 트리거 가능.
- **OAuth 인증만 (OIDC 금지, T-09.1-34 mitigate)**: `--oauth-service-account-email=gh-radar-scheduler-sa` 사용. Phase 05.1 D-07 lesson 승계. Scheduler → Cloud Run Job invoke 시 OIDC 사용 시 401 발생 (Phase 05.1 실측).
- **KIWOOM Secrets 빈 secret 신설**: setup 스크립트가 gcloud secrets create 만 수행 + accessor 바인딩. value 추가는 Plan 09 [BLOCKING] 사용자 액션 (키움 portal 발급 + `echo -n '<APPKEY>' | gcloud secrets versions add gh-radar-kiwoom-appkey --data-file=-`). 사용자가 키움 등록과 IP whitelist 동시에 처리.
- **KIS env/secret 의도적 유지 (Wave 4 cleanup 까지)**: deploy-server.sh 의 KIS_APP_KEY/SECRET + KIS_BASE_URL 유지. server config.ts 가 KIS optional() 처리 (Plan 07). server.ts 가 createKisRuntime 호출 안 함 → dead code, outbound 0. Wave 4 Plan 11 cleanup 단계에서 일괄 제거.
