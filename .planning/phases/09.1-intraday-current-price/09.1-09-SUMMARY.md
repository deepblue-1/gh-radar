**3. [Rule 1 - Bug] STEP1 payload 의 upper_limit/lower_limit NOT NULL 위반**
- **Found during:** Task 6 (cycle 3)
- **Issue:** `null value in column "upper_limit" of relation "stock_quotes" violates not-null constraint`. stock_quotes.upper_limit/lower_limit 는 Phase 1 시절 NOT NULL — ka10027 STEP1 응답에 미포함.
- **Fix:** payload 에 한국 시장 일일 변동폭 ±30% 도메인 규칙 임시값 (`u.price * 1.3` / `u.price * 0.7`) 채움. STEP2 가 정확값으로 덮어씀.
- **Files modified:** workers/intraday-sync/src/pipeline/upsertQuotes.ts + tests/upsertQuotes.test.ts
- **Commit:** cc6fa22

**4. [Rule 1 - Bug] STEP2 hot set 에 STEP1 미처리 종목 포함 시 INSERT 시도 → price NOT NULL 위반**
- **Found during:** Task 6 (cycle 4)
- **Issue:** computeHotSet 이 watchlist 종목을 합치므로 STEP1 ka10027 응답에 없는 종목이 STEP2 hot set 에 포함될 수 있음.
- **Fix:** `step2Updates = step2UpdatesRaw.filter(u => step1Codes.has(u.code))` + `droppedFromStep2` 카운트 log.
- **Files modified:** workers/intraday-sync/src/index.ts
- **Commit:** d8d05d1

**5. [Rule 1 - Bug] upsertQuotesStep2 UPSERT 가 신규 row INSERT 단계에서 모든 NOT NULL 평가**
- **Found during:** Task 6 (cycle 5)
- **Issue:** Supabase `upsert(onConflict: code)` 는 `INSERT...ON CONFLICT DO UPDATE` 로 변환되어 conflict 미발생 시 INSERT 분기에서 NOT NULL 평가. STEP2 페이로드에 price 없으면 violation. PLAN 06 의 "race-free UPSERT 컬럼 분리" 패턴이 STEP2 신규 종목 case 에 깨짐.
- **Fix:** upsert → 종목별 `.update({...}).eq("code", code)` 직렬 호출. UPDATE 는 INSERT 분기 평가 안 함 → STEP1 매분 컬럼 자연 보존. ~250 종목 직렬 = 수십 ms.
- **Files modified:** workers/intraday-sync/src/pipeline/upsertQuotes.ts + tests/upsertQuotes.test.ts
- **Commit:** ef120aa (**결정적 fix**)
