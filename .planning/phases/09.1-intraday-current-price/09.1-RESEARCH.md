## Summary

키움 REST API `ka10027` (등락률순위) 는 페이지네이션으로 활성 거래 종목 ~1,898 의 현재가/등락/누적거래량을 ~1초 만에 수집할 수 있는 single source. 본 phase 의 핵심 결정 5건은 다음과 같다:

1. **응답 매핑은 사용자 실측 (2026-05-13) 으로 잠금** — `cur_prc`/`pred_pre`/`flu_rt` 의 부호 prefix 와 `stk_cd` 의 `_AL` 접미사는 실측 확인. 단 `now_trde_qty` 가 누적/분단위 거래량 인지는 공식 docs 부족 — Plan 단계 첫 호출 fixture 로 잠금 (확률 99%: KIS `acml_vol` 과 동일한 누적 거래량).
2. **UPSERT 는 partial column update** — `ON CONFLICT (code, date) DO UPDATE SET close, change_amount, change_rate, volume, inserted_at` 4 컬럼만. `open/high/low/trade_amount` 는 candle-sync EOD 가 채우는 영역이라 intraday 가 덮어쓰지 않음. **그러나 09:00 첫 cycle 시점에 candle-sync EOD row 가 아직 없음** — INSERT 시 `open/high/low` 가 NOT NULL 제약이라 채워야 함. §3.2 의 정책 결정 필요.
3. **Direct VPC Egress + Static IP "Cloud NAT 대체" 가설은 공식 docs 와 충돌** — Google Cloud 공식 문서 [docs.cloud.google.com/run/docs/configuring/static-outbound-ip] 에 따르면 Static outbound IP 를 위해 Cloud NAT 가 **필수** (Direct VPC Egress 는 internal IP 만 부여). 사용자가 추정한 "$33/월" 도 과대 추정 — 실측 ~$1-3/월. **R1 BLOCKER: 사용자 D-12 재확인 필요 — Cloud NAT 비용은 미미하므로 정공법 채택 권장**.
4. **토큰 캐시는 in-memory (단일 cycle 내 1회 발급)** 권장 — 매분 cycle, ~1초 완료 → 매 cycle 1 token 발급 호출 + 1 페이지네이션 sequence. 24h cache 의 이득 미미 + Supabase RTT 추가 단점 더 큼. Plan 단계 Pitfall: 키움 토큰 expires_in 이 실제 30분일 가능성 (검색 결과 일부 출처는 `expires_in=1800`). 첫 호출 시 응답 캡처해 확정.
5. **Wave 0 prerequisite 1개 — 키움 token + ka10027 첫 호출 실측** (Phase 9 의 KRX endpoint 별 승인 검증 lesson 승계). 사용자가 KIWOOM_APPKEY/SECRETKEY 보유는 명시 — 단, 키움 ka10027 endpoint 별 사용 등록 상태 + IP 화이트리스트 등록 여부 확인 필요.

### 1.4 부호 Prefix 의미

**[VERIFIED: 사용자 실측 2026-05-13]**

- `cur_prc`, `pred_pre`, `flu_rt` 모두 첫 글자가 `+` (상승) / `-` (하락) / `0` (보합) 또는 부호 생략.
- 가격 본체는 **절댓값** — 부호는 등락 방향 표시.
- 예: `cur_prc="+6760"` = 현재가 6,760원, 상승 / `cur_prc="-274250"` = 274,250원, 하락.
- mapper 파싱 함수:

```typescript
function parseSignedPrice(s: string | undefined): {
  value: number;
  direction: "up" | "down" | "flat";
} {
  if (!s) throw new Error("missing signed price");
  const trimmed = s.trim();
  const sign = trimmed[0];
  const rest = (sign === "+" || sign === "-") ? trimmed.slice(1) : trimmed;
  const abs = Number(rest.replace(/,/g, ""));
  if (!Number.isFinite(abs)) throw new Error(`invalid signed price: "${s}"`);
  return {
    value: abs,
    direction: sign === "+" ? "up" : sign === "-" ? "down" : "flat",
  };
}
```

**`change_amount` (전일대비) 의 부호 처리:** `pred_pre="+100"` → +100, `pred_pre="-100"` → -100. **부호 그대로 유지** (가격이 아니므로 부호가 의미). mapper 는 `cur_prc` 와 다르게 `pred_pre`/`flu_rt` 는 `Number("+100")=100`, `Number("-100")=-100` 으로 정상 변환. JS 의 `Number()` 가 leading `+/-` 처리 가능.

**중요: `flu_rt` 도 부호 유지** — `flu_rt="-1.5"` 는 `-1.5%` 로 저장. DB `change_rate numeric(10,4)` 가 음수 허용 (Phase 9 hotfix 으로 확장됨).

### 1.5 `stk_cd` 의 `_AL` 접미사

**[VERIFIED: 사용자 실측 2026-05-13]**

- 응답의 `stk_cd` 는 6자 단축코드 뒤에 `_AL` 접미사가 붙음. 예: `"007460_AL"`.
- `_AL` = "All" 일반 거래 추정 (키움 내부 마커). stocks 마스터의 `code` 는 6자 단축코드만 사용.
- mapper 에서 strip:
```typescript
function stripAlSuffix(stkCd: string): string {
  return stkCd.replace(/_AL$/, "");
}
```

**테스트 fixture 필요 케이스:**
- 일반 종목: `"007460_AL"` → `"007460"`

### 1.6 `now_trde_qty` 의 의미 (누적 vs 분단위)

**판정: MEDIUM — 강한 가설 (누적 거래량), Plan 02 fixture 로 확정 필요 (R2)**

**증거:**
- KIS API 의 `acml_vol` 은 누적 거래량 (Phase 1 검증) — 한국 거래 데이터 컨벤션은 누적이 일반적.

- 별도 분단위 거래량 필드 (예: `min_trde_qty`) 는 ka10027 응답에 등장하지 않음.

**가설:** `now_trde_qty` = 당일 누적 거래량 = ACC_TRDVOL. CONTEXT D-06 의 가정과 일치.

**R2 BLOCKER (낮은 확률):** 만약 분단위 거래량이라면 누적값으로 변환 위해 별도 endpoint 호출 필요 — 본 phase 의 1초 cycle 가정 무효화. **Plan 02 의 첫 호출 응답에서 1분 후 두 번째 호출의 동일 종목 `now_trde_qty` 비교** 로 확정:
- 두 호출 사이 값이 단조 증가 → 누적
- 변동 패턴 (감소 가능) → 분단위

이 검증은 Plan 02 의 fixture 캡처 task 에 포함.

### 1.7 Rate Limit

**판정: MEDIUM — 공식 명시 없음, KIS 비교 추정**

## 3. UPSERT 정책 — Partial Column Update

### 3.1 SQL 명세

```sql
-- intraday-sync 의 UPSERT (Supabase JS v2)
-- ON CONFLICT (code, date) DO UPDATE SET 명시한 컬럼만
INSERT INTO stock_daily_ohlcv (code, date, open, high, low, close, volume, trade_amount, change_amount, change_rate, inserted_at)
VALUES (...)
ON CONFLICT (code, date) DO UPDATE
  SET close         = EXCLUDED.close,
      change_amount = EXCLUDED.change_amount,
      change_rate   = EXCLUDED.change_rate,
      volume        = EXCLUDED.volume,
      inserted_at   = now();
```

Supabase JS v2 의 `.upsert()` 는 기본적으로 모든 컬럼을 UPDATE. **partial column update 를 위해서는 두 가지 옵션 중 택1:**

**옵션 A: `.upsert()` + 전 컬럼 페이로드 (open/high/low 는 기존 값 보존)**
- INSERT 시 open/high/low/trade_amount 도 채워야 함 (NOT NULL 제약).
- UPDATE 시 EXCLUDED.open 등은 새 페이로드의 값으로 덮어쓸 위험 → mapper 가 open/high/low 를 기존 row 에서 read 후 같은 값 echo 필요. → race condition.

**옵션 B: 명시 SQL (Supabase RPC 또는 raw SQL)**
- `supabase.rpc("intraday_upsert", { rows })` — Postgres 함수로 partial UPDATE 명시.
- 또는 `supabase.from().upsert()` 대신 `supabase.postgrest.rpc()` 직접 호출.

**옵션 C (권장): `.upsert()` 페이로드 컬럼 제한 + 분기 처리**
- 오늘자 row 가 이미 존재하면 (candle-sync EOD 또는 이전 intraday cycle 작성): UPDATE만 (`supabase.from().update().eq("code",...).eq("date", today)` 분기) — 매분 ~1,898 row UPDATE.
- 오늘자 row 가 없으면 (09:00 첫 cycle): INSERT — open=high=low=close=cur_prc, trade_amount=0 으로 임시 (§3.2).

→ **권장 옵션: B (Postgres RPC) 또는 D (별도 staging UPSERT + ON CONFLICT)**. 단순성 위해 **D 선호** — §3.3 참조.

### 3.2 09:00 첫 Cycle: Today Row 부재 처리 (R4 결정)

**문제:** Phase 9 의 candle-sync EOD 는 17:30 발화 → 09:00 첫 intraday cycle 시점에 오늘자 row 는 **존재하지 않음**. `stock_daily_ohlcv` 의 `open/high/low/trade_amount` 는 NOT NULL.

**옵션 비교:**

| 옵션 | INSERT 시 open/high/low | trade_amount | EOD 가 덮어쓰는지 | 정확성 |
|---|---|---|---|---|
| **A. close 임시 채움** | open=high=low=close=cur_prc | 0 | YES — EOD 가 올바른 OHLC 로 overlay | OK (intraday 동안 잘못된 OHL 표시) |
| **B. NULL 허용으로 스키마 변경** | NULL | NULL | YES — EOD 가 정확 값 | 스키마 변경 = 마이그레이션 추가 |
| **C. INSERT 스킵 — UPDATE only** | (row 없으면 skip) | — | candle-sync 가 17:30 까지 첫 INSERT | intraday 동안 오늘자 데이터 = 비어있음 (목적 모순) |
| **D. open=cur_prc, high=low=cur_prc 동적 갱신** | 첫 cycle open=cur_prc, 이후 cycle 마다 high=max(high,close), low=min(low,close) | 0 | YES — EOD overlay | 정확 (시가/고가/저가 의미 정상) |

**권장: 옵션 D (intraday OHLC tracking)** — 9시 첫 cycle 의 cur_prc 를 open 으로 기록 + 매 cycle 마다 high=max(현재 high, close), low=min(현재 low, close). EOD 가 17:30 에 KRX 공식 데이터로 overlay → 최종 정확.

**Plan 단계 구현:**
1. UPSERT 페이로드 의 open/high/low 결정 로직:
   - 기존 row 가 없거나 candle-sync EOD 가 아직 안 채움 (e.g., `inserted_at < today 09:00 KST`): **UPDATE** 시 `high=GREATEST(high, EXCLUDED.close)`, `low=LEAST(low, EXCLUDED.close)`, `open` 은 첫 INSERT 만 set.
   - **Postgres GREATEST/LEAST** 는 NULL 안전 (NULL 무시), Supabase JS v2 에서 RPC 또는 raw expression 으로 전달.
2. RPC 함수 신설 (`intraday_upsert_close(rows jsonb[])`) — Plan 02 의 Supabase 마이그레이션에 포함.

**대안: 옵션 A (단순)** — Plan 단계에서 사용자가 정확성보다 단순성 선호 시. EOD 17:30 까지 잘못된 open/high/low 가 일시 표시되지만 17:30 이후 정확. 사용자 결정 필요.

**RESEARCH 권장: 옵션 D — Postgres RPC `intraday_upsert_close`** (정확성 + 코드 단순성 균형). 단 Plan 단계에서 사용자 confirm 권장.

### 3.3 Supabase Postgres RPC 명세 (옵션 D)

```sql
-- supabase/migrations/20260514120000_intraday_upsert_close_rpc.sql
BEGIN;

CREATE OR REPLACE FUNCTION public.intraday_upsert_close(
  p_rows jsonb
)
RETURNS integer
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  r jsonb;
  affected integer := 0;
BEGIN
  FOR r IN SELECT * FROM jsonb_array_elements(p_rows)
  LOOP
    INSERT INTO public.stock_daily_ohlcv
      (code, date, open, high, low, close, volume, trade_amount, change_amount, change_rate, inserted_at)
    VALUES (
      r->>'code',
      (r->>'date')::date,
      (r->>'close')::numeric(20,2),  -- INSERT 시: open=cur_prc
      (r->>'close')::numeric(20,2),  -- INSERT 시: high=cur_prc
      (r->>'close')::numeric(20,2),  -- INSERT 시: low=cur_prc
      (r->>'close')::numeric(20,2),
      (r->>'volume')::bigint,
      0,
      NULLIF(r->>'change_amount', '')::numeric(20,2),
      NULLIF(r->>'change_rate', '')::numeric(10,4),
      now()
    )
    ON CONFLICT (code, date) DO UPDATE
      SET close         = EXCLUDED.close,
          high          = GREATEST(stock_daily_ohlcv.high, EXCLUDED.close),
          low           = LEAST(stock_daily_ohlcv.low, EXCLUDED.close),
          volume        = EXCLUDED.volume,
          change_amount = EXCLUDED.change_amount,
          change_rate   = EXCLUDED.change_rate,
          inserted_at   = now();
          -- open, trade_amount: 의도적 omit
    affected := affected + 1;
  END LOOP;
  RETURN affected;
END;
$$;

-- 보안: anon/authenticated 의 자동 grant 차단 (사용자 메모리 룰 준수)
REVOKE EXECUTE ON FUNCTION public.intraday_upsert_close(jsonb) FROM PUBLIC;
REVOKE EXECUTE ON FUNCTION public.intraday_upsert_close(jsonb) FROM anon, authenticated;

-- service_role 만 허용
GRANT EXECUTE ON FUNCTION public.intraday_upsert_close(jsonb) TO service_role;

COMMIT;
```

**호출 측 (worker):**
```typescript
const payload = batchRows.map(toJsonbRow);
const { data, error } = await supabase.rpc("intraday_upsert_close", {
  p_rows: payload,
});
```

**Chunked execution:** 1,898 row 를 1000 row chunk 로 분할 (candle-sync `upsert.ts` 패턴). RPC 1 call = 1 chunk → 2 calls/cycle.

## 4. Infrastructure — Direct VPC Egress + Static Outbound IP

### 4.1 핵심 사실 (R1 가설 충돌)

**판정: HIGH — Cloud NAT 가 Static IP 필수 (Direct VPC Egress 만으로 불가)**

**[CITED: docs.cloud.google.com/run/docs/configuring/static-outbound-ip]**
> "To enable a Cloud Run service to route requests using a static IP address, configure the Cloud Run service's VPC egress to route all outbound traffic through a VPC network that has a **Cloud NAT gateway** configured with the static IP address."

**[CITED: docs.cloud.google.com/run/docs/configuring/vpc-direct-vpc]**
> "Direct VPC egress enables Cloud Run services, functions, jobs, and worker pools to connect to VPC networks without requiring a Serverless VPC Access connector."

**해석:** Direct VPC Egress 는 internal IP 부여 + VPC 라우팅만 처리. 외부 endpoint (키움) 로 Static IP 보장은 **Cloud NAT 가 처리**. 사용자 CONTEXT D-12 의 "Direct VPC Egress + Static IP (Cloud NAT 대체)" 는 사실과 다른 가정 → planner 가 사용자 confirm 후 정공법 채택.

### 4.2 비용 정정

**[CITED: cloud.google.com/nat/pricing + groups.google.com/g/gce-discussion]**
- Cloud NAT: $0.0014/hr × VM 수 (1 VM = ~$1/월), 데이터 처리 $0.045/GB. 매분 응답 ~200KB × 1440 cycle/day = ~290MB/day → 데이터 비용 ~$0.40/월. **합산 ~$1.5-3/월** — 사용자 추정 $33/월 대비 1/10 수준.
- Static reserved IP (사용 중): $0/월. 미사용 시 ~$1.5/월.
- **결론: Cloud NAT 정공법 채택 = 무리한 비용 절감 아님**.

### 4.6 Limitations / Pitfalls

**[CITED: docs.cloud.google.com/run/docs/configuring/vpc-direct-vpc]**
- 인스턴스 시작 시 connection establishment 1분+ 지연 가능 — 본 phase 는 매분 cycle 이므로 첫 cycle 지연 → 실 데이터는 두 번째 cycle (09:01) 부터 안정 도착. 첫 cycle 미수신은 R4 의 09:00 row INSERT 옵션 D 에서 자연 보완.
- 최대 throughput: 1 Gbps/instance — 매분 ~200KB 응답이라 무관.
- Jobs 1h+ 실행 시 connection break 가능 — 본 phase task-timeout=60s 라 무관.
- Subnet 최소 `/26`. Cloud Run 은 `/28` block 단위 reserve. Jobs 는 task 당 1 IP + 7분 post-completion. 단일 task / parallelism=1 → 충분.

### 4.7 사용자 D-12 confirm 항목

본 RESEARCH 는 D-12 의 "Cloud NAT 대체" 가설을 **R1 BLOCKER** 로 분류하고 정공법 (Direct VPC Egress + Cloud NAT + Static IP) 을 권장. Plan 단계 또는 user-confirm step 에서:

- ❌ "Cloud NAT 대체" 는 GCP 에서 지원하지 않음 — Direct VPC Egress 만으로 Static outbound IP 부여 불가.
- 추가 옵션: Static IP 를 Compute Engine VM 으로 우회 — 비용/복잡도 모두 NAT 보다 큼. 비권장.

## 5. 키움 OAuth2 Token Caching Strategy

### 5.1 옵션 비교

| 옵션 | 장점 | 단점 | 권장 |
|---|---|---|---|
| **A. In-memory (cycle 별 1회 발급)** | 단순, RTT 0, 매 cycle 격리 | 매분 1회 token 발급 호출 (1440/일) | ✅ **권장** |
| **B. In-memory + 모듈 전역 cache** | 단일 process 내 토큰 재사용 | Cloud Run Job 은 task 단위 새 컨테이너 → 재사용 X | ❌ 의미 없음 |
| **C. Supabase `kiwoom_tokens` 테이블 (kis_tokens mirror)** | Cross-cycle 재사용, token 발급 호출 절약 | 마이그레이션 추가, Supabase RTT, 매 cycle 2 read/write 추가 | ❌ 이득 < 비용 |

**판정: A (in-memory, 매 cycle 1회 발급)** — 키움 token 발급 endpoint 가 별도 rate limit 없을 것으로 가정 (KIS 도 무제한 발급). cycle 격리 → 토큰 만료 race condition 자연 회피.

### 5.2 구현 패턴

```typescript
// workers/intraday-sync/src/kiwoom/auth.ts
import axios from "axios";
import type { Config } from "../config";

export type KiwoomToken = {
  token: string;
  tokenType: string; // "bearer"
  expiresAt: Date;   // expires_dt YYYYMMDDhhmmss 파싱 결과
};

export async function issueKiwoomToken(config: Config): Promise<KiwoomToken> {
  const res = await axios.post(
    `${config.kiwoomBaseUrl}/oauth2/token`,
    {
      grant_type: "client_credentials",
      appkey: config.kiwoomAppkey,
      secretkey: config.kiwoomSecretkey,
    },
    {
      timeout: 10_000,
      headers: { "content-type": "application/json" },
    },
  );

  // expires_dt format: "YYYYMMDDhhmmss" (KST)
  const dt = res.data.expires_dt as string;
  const expiresAt = parseKiwoomExpiresDt(dt);

  return {
    token: res.data.token,
    tokenType: res.data.token_type,
    expiresAt,
  };
}

function parseKiwoomExpiresDt(s: string): Date {
  // "20260513093013" → Date (KST → UTC 조정)
  if (!/^\d{14}$/.test(s)) throw new Error(`Invalid expires_dt: ${s}`);
  const y = Number(s.slice(0, 4));
  const mo = Number(s.slice(4, 6)) - 1;
  const d = Number(s.slice(6, 8));
  const h = Number(s.slice(8, 10));
  const mi = Number(s.slice(10, 12));
  const se = Number(s.slice(12, 14));
  // KST = UTC+9 → UTC 변환
  return new Date(Date.UTC(y, mo, d, h - 9, mi, se));
}
```

### 5.3 토큰 만료 시간 가설 (R3)

- 사용자 CONTEXT D-16: "24h 유효" 가정.
- 검색 결과 (pabburi.co.kr): 응답에 `expires_in: 1800` (30분) 명시.
- 공식 docs (openapi.kiwoom.com/guide/apiguide): `expires_dt` 만 명시, 만료 기간 명시 없음.

**Plan 02 fixture 캡처 작업으로 확정** — 첫 호출 응답의 `expires_dt` 와 호출 시각 차이 측정.

**가정 시나리오:**
- 30분 만료: 매 cycle 1회 발급 = 1440 calls/일 — 부담 없음. 옵션 A 가 답.
- 24h 만료: 매 cycle 1회 발급도 OK. 옵션 A 동일.
- 5분 미만 만료 (가능성 낮음): 매 cycle 1회 발급 시 cycle 종료 전 만료 위험 — Plan 02 검증 후 분기.

**결론:** 옵션 A (in-memory, cycle 격리) 는 모든 만료 시나리오에 안전.

### 8.2 Threat Model for {keep|key 노출, race, IP}

| Pattern | STRIDE | Standard Mitigation |
|---|---|---|
| **T-1: KIWOOM_APPKEY/SECRETKEY 노출** | Information Disclosure | Secret Manager 저장 + Cloud Run `--set-secrets` 사용 (env var 평문 X) + Cloud Logging redact (pino redact paths 확장) + .env.deploy 는 gitignore. **Plan 02 의 logger.ts 에 `*.kiwoomAppkey`, `*.kiwoomSecretkey` redact paths 추가** |
| **T-2: ka10027 응답 동시 cycle race condition** | Tampering (timing-based double-update) | `--parallelism=1 --max-retries=0` + cycle 60s + cron 매분 → 최악 시나리오에서도 single in-flight. Cloud Run Job task 동시 실행 0 보장 |
| **T-3: stock_daily_ohlcv open/high/low 무단 overwrite** | Tampering | RPC 의 `ON CONFLICT DO UPDATE SET` 에 open/trade_amount 의도적 omit. high=GREATEST, low=LEAST 단조 갱신 → EOD candle-sync 가 17:30 에 overlay |
| **T-4: 키움 token race (cycle 1 의 만료가 cycle 2 와 충돌)** | DoS | In-memory cache + cycle 격리 (각 Cloud Run task 새 container 새 토큰). cross-cycle 공유 없음 → race 없음 |
| **T-5: Static IP 누설 → 키움 IP whitelist 우회** | Spoofing | Static IP 는 reserved external IP — GCP 내부 routing. Cloud NAT egress 만. ingress 차단 (Cloud Run Job 은 ingress 없음 — `:run` only) |
| **T-6: RPC `intraday_upsert_close` 의 anon/authenticated 호출** | Elevation of Privilege | REVOKE FROM anon, authenticated 명시 (사용자 메모리 룰: Supabase RPC REVOKE 명시 필수). GRANT service_role only |
| **T-7: 다른 워커와의 stock_daily_ohlcv 충돌 (candle-sync, master-sync)** | Concurrency | 시간대 분리 (D-18: candle-sync `30 17`, intraday `* 9-15`, recover `10 8`, master `10 8` for stocks 마스터) + 컬럼 분리 (open/high/low/trade_amount = candle-sync 만, close/change/volume = intraday + candle-sync) |
| **T-8: Cloud Run Job task-timeout 60s 초과** | DoS / data loss | task-timeout 60s + 매분 cron — timeout 1 cycle 손실은 다음 cycle 에서 회복 (idempotent UPSERT). Plan 단계 watchdog log → alert |
| **T-9: 키움 API 응답 schema 변경 (필드명 변경 / 추가)** | Tampering (upstream) | zod 응답 validation + MIN_EXPECTED_ROWS 가드 + Cloud Monitoring alert |
| **T-10: 09:00 첫 cycle 의 Direct VPC Egress 1분+ 시작 지연** | Availability | 첫 cycle 미수신 → 두 번째 cycle (09:01) 부터 정상. RPC INSERT 옵션 D 의 open=close 임시 채움이 자연 보완 (high/low 도 첫 INSERT 시 = close). |

// workers/intraday-sync/src/kiwoom/fetchRanking.ts
import type { AxiosInstance } from "axios";
import type { KiwoomKa10027Row } from "@gh-radar/shared";

const HARD_CAP = 5000;

type Ka10027Response = {
  return_code: number;
  return_msg?: string;
  pred_pre_flu_rt_upper?: KiwoomKa10027Row[];
};

export async function fetchKa10027(
  client: AxiosInstance,
  token: string,
): Promise<KiwoomKa10027Row[]> {
  const body = {
    mrkt_tp: "000",
    sort_tp: "1",
    trde_qty_cnd: "0000",
    stk_cnd: "0",
    crd_cnd: "0",
    updown_incls: "1",
    pric_cnd: "0",
    trde_prica_cnd: "0",
  };

  const all: KiwoomKa10027Row[] = [];
  let contYn: "Y" | "N" = "N";
  let nextKey: string | undefined;

  do {
    const headers: Record<string, string> = {
      authorization: `Bearer ${token}`,
      "api-id": "ka10027",
      "content-type": "application/json;charset=utf-8",
      "cont-yn": contYn,
    };
    if (nextKey) headers["next-key"] = nextKey;

    let res;
    try {
      res = await client.post<Ka10027Response>(
        "/api/dostk/rkinfo",
        body,
        { headers },
      );
    } catch (err: any) {
      if (err?.response?.status === 401) {
        throw new Error(
          "키움 401 — KIWOOM_APPKEY/SECRETKEY 또는 토큰 발급 실패",
        );
      }
      if (err?.response?.status === 429) {
        throw new Error("키움 429 — rate limit 초과 (예상치 못함)");
      }
      throw err;
    }

    const rows = res.data.pred_pre_flu_rt_upper ?? [];
    all.push(...rows);

    contYn = (res.headers["cont-yn"] === "Y") ? "Y" : "N";
    nextKey = res.headers["next-key"];

    if (all.length >= HARD_CAP) {
      throw new Error(`pagination hard cap ${HARD_CAP} exceeded`);
    }
  } while (contYn === "Y" && nextKey);

  return all;
}

### Pattern 2: candle-sync 의 `upsertOhlcv` → intraday-sync 의 RPC 호출

```typescript
// workers/intraday-sync/src/pipeline/upsert.ts
import type { SupabaseClient } from "@supabase/supabase-js";
import type { IntradayCloseUpdate } from "@gh-radar/shared";
import { logger } from "../logger";

const CHUNK_SIZE = 1000;

export async function intradayUpsertChunked(
  supabase: SupabaseClient,
  rows: IntradayCloseUpdate[],
): Promise<{ count: number }> {
  if (rows.length === 0) return { count: 0 };

  let total = 0;
  for (let i = 0; i < rows.length; i += CHUNK_SIZE) {
    const chunk = rows.slice(i, i + CHUNK_SIZE);
    const payload = chunk.map((r) => ({
      code: r.code,
      date: r.date,
      close: r.close,
      volume: r.volume,
      change_amount: r.changeAmount,
      change_rate: r.changeRate,
    }));

    const { data, error } = await supabase.rpc("intraday_upsert_close", {
      p_rows: payload,
    });

    if (error) {
      logger.error({ error, chunkStart: i, chunkSize: chunk.length }, "rpc failed");
      throw error;
    }
    total += (typeof data === "number" ? data : chunk.length);
  }

  return { count: total };
}
```

### Pattern 3: 매분 cycle index.ts (단일 entry, MODE 없음)

```typescript
// workers/intraday-sync/src/index.ts
import "dotenv/config";
import { loadConfig } from "./config";
import { logger } from "./logger";
import { issueKiwoomToken } from "./kiwoom/auth";
import { createKiwoomClient } from "./kiwoom/client";
import { fetchKa10027 } from "./kiwoom/fetchRanking";
import { ka10027RowToUpdate } from "./pipeline/map";
import { bootstrapMissingStocks } from "./pipeline/bootstrapStocks";
import { intradayUpsertChunked } from "./pipeline/upsert";
import { createSupabaseClient } from "./services/supabase";
import { withRetry } from "./retry";

function todayIsoKst(): string {
  const now = new Date();
  const kst = new Date(now.getTime() + 9 * 60 * 60 * 1000);
  const y = kst.getUTCFullYear();
  const m = String(kst.getUTCMonth() + 1).padStart(2, "0");
  const d = String(kst.getUTCDate()).padStart(2, "0");
  return `${y}-${m}-${d}`;
}

export async function runIntradayCycle(deps: { log: typeof logger }): Promise<{ count: number; dateIso: string }> {
  const { log } = deps;
  const config = loadConfig();
  const dateIso = todayIsoKst();
  const log2 = log.child({ dateIso });

  log2.info("intraday cycle start");

  // 1. token 발급 (cycle 격리)
  const token = await withRetry(() => issueKiwoomToken(config), "issueKiwoomToken");
  log2.info({ expiresAt: token.expiresAt.toISOString() }, "token issued");

  // 2. 키움 client + ka10027 페이지네이션
  const kiwoom = createKiwoomClient(config);
  const rows = await withRetry(() => fetchKa10027(kiwoom, token.token), "fetchKa10027");
  log2.info({ rows: rows.length }, "ka10027 fetched");

  // 3. 가드
  if (rows.length === 0) {
    log2.warn("ka10027 0 rows — 휴장일 또는 키움 미응답");
    return { count: 0, dateIso };
  }
  if (rows.length < config.minExpectedRows) {
    throw new Error(`ka10027 ${rows.length} < ${config.minExpectedRows} — partial response`);
  }

  // 4. mapper + bootstrap
  const supabase = createSupabaseClient(config);
  const boot = await withRetry(() => bootstrapMissingStocks(supabase, rows), "bootstrapMissingStocks");
  if (boot.inserted > 0) log2.info({ bootstrapped: boot.inserted }, "stocks bootstrap");

  const updates = rows
    .map((r) => {
      try { return ka10027RowToUpdate(r, dateIso); }
      catch (err) { log2.warn({ err: (err as Error).message, stk_cd: r.stk_cd }, "mapper skip"); return null; }
    })
    .filter((u): u is NonNullable<typeof u> => u !== null);

  // 5. RPC chunked upsert
  const { count } = await withRetry(() => intradayUpsertChunked(supabase, updates), "intradayUpsertChunked");

  log2.info({ count, dropped: rows.length - updates.length }, "intraday cycle complete");
  return { count, dateIso };
}

async function main(): Promise<void> {
  try {
    const out = await runIntradayCycle({ log: logger });
    logger.info({ ...out }, "intraday-sync complete");
    process.exit(0);
  } catch (err) {
    logger.error({ err }, "intraday-sync failed");
    process.exit(1);
  }
}

if (process.argv[1] && process.argv[1].endsWith("index.js")) {
  main();
}
```
