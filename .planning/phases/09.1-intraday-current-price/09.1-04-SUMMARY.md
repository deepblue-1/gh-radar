## Decisions Made

- **parseSignedPrice 의 부호 처리 (D-09 구현)**: 첫 글자 가 `+` 또는 `-` 면 slice(1) 후 절댓값. 그 외 (숫자) 는 flat direction + 그대로 절댓값. trim() + comma strip 까지 1 함수에서 처리. 빈/undefined/잘못된 숫자는 throw (fail-fast).
- **parseOptionalSignedNumber + parseOptionalNumber 분리 (D-09 의 부호 유지 vs 양수 only)**: 구현 자체는 동일 (Number(s.replace(/,/g, ""))). 이름이 caller intent 를 명확히 — pred_pre/flu_rt 는 부호 유지, now_trde_qty 는 양수 only. 향후 양수 only 검증 추가 시 parseOptionalNumber 만 변경.
- **stk_cd 6자 정규식 가드 (/^\\d{6}$/) (D-10 구현)**: strip 후에도 6자 숫자 아니면 throw — 본 phase scope 는 KOSPI/KOSDAQ 종목코드만. 7자/8자 특수 종목코드 (예: 채권/펀드) 는 미지원, 응답에서 자연 제외 가정.
- **tradeAmount = Math.round(volume * price) (D-23 구현)**: 정수형 강제 (회계 호환, DB numeric 컬럼 fit). 음수 cur_prc 가능성 없음 — parseSignedPrice 가 value 를 절댓값 보장. volume × price 가 Number.MAX_SAFE_INTEGER 초과는 한국 시장 실측 미발생 (삼성전자 1억주 × 10만원 = 10^13, MAX_SAFE_INTEGER 9.0×10^15 이내).
- **rateLimiter 의 default 24 req/s 잠금 + configureKiwoomRateLimiter override**: Plan 06 의 runIntradayCycle 이 config.ka10001RateLimitPerSec (Plan 03 의 KA10001_RATE_LIMIT env, default 24) 로 동적 설정. resetKiwoomRateLimiter 도 export — vitest 의 test 간 bucket 격리 시 활용 가능 (본 plan 테스트에서는 미사용, Plan 05/06 에서 필요 시 호출).
- **tokenStore 가 axios 직접 호출 (createKiwoomClient 미사용)**: token endpoint (POST /oauth2/token) 는 Bearer 미필요 — body 에 appkey/secretkey 만. client factory 의 default headers 와 무관하므로 createKiwoomClient 거치지 않고 axios.post 직접. RESEARCH §5.3 패턴 그대로.
- **fetchKa10027 의 next-key 헤더 echo 패턴**: 첫 호출 cont-yn=N (next-key 미설정), 응답에 cont-yn=Y + next-key 있으면 2번째 호출에 cont-yn=Y + next-key=<응답값>. 응답 헤더 키는 소문자 (`cont-yn` / `next-key`) — axios 가 자동 소문자 변환. 본 test 의 makeClient mock 도 소문자 키로 echo.
- **hardCap=5000 의 의미 명확화**: '한 페이지 row 수' 가 아니라 '누적 row 가 5000 초과 시 throw'. CONTEXT D-06 의 활성 ~1,898 종목 기준 2.6x 마진. 본 test 의 hardCap=2 case 는 page1 2 row push 직후 `all.length >= 2` 도달로 throw — 의도된 경계 동작.
