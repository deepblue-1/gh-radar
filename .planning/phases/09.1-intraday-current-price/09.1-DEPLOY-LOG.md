## Deviations (자동 fix 5건)

본 plan 실행 중 발견된 production 환경 차이 5건 — 모두 Rule 1 (Bug) 자동 수정. 사용자 confirmation 없이 진행 (auto mode):

1. **키움 ka10027 spec 변경 — stex_tp 필수 파라미터 추가** (commit `ba2df5c`)
   - 키움이 spec 갱신: `[1511:필수 입력 값에 값이 존재하지 않습니다. 필수입력 파라미터=stex_tp]`
   - mrkt_tp=000 (KOSPI+KOSDAQ 통합) 의도와 일치하는 `stex_tp="3"` (통합) 채택.
   - 단위 테스트 fetchRanking.test.ts 에 stex_tp="3" 기대값 추가.

2. **stock_quotes payload 의 name/market 제거** (commit `ad9714f`)
   - PLAN 06 SUMMARY 가 "STEP1 stock_quotes UPSERT 컬럼: ..name/market.." 으로 작성됐으나 실제 schema 에는 name/market 컬럼 부재 (stocks 마스터 + top_movers 에 보유).
   - `Could not find the 'market' column of 'stock_quotes' in the schema cache` 에러 발견.
   - upsertQuotesStep1 의 payload 에서 두 키 제거 + 시그니처에서 market Map 인자 제거. caller (index.ts) 동기화.

3. **STEP1 payload 에 upper_limit/lower_limit 임시값 채움 (±30%)** (commit `cc6fa22`)
   - stock_quotes.upper_limit/lower_limit 는 Phase 1 unified-stocks 시절 NOT NULL.
   - 키움 ka10027 STEP1 응답에 미포함 → 신규 종목 INSERT 시 violation.
   - 한국 시장 일일 변동폭 ±30% 도메인 규칙으로 근사값 채워 NULL 위반 방지. STEP2 (ka10001 hot set) 가 정확값으로 덮어씀.

4. **STEP2 hot set 을 STEP1 처리 종목으로 intersect** (commit `d8d05d1`)
   - computeHotSet 이 watchlist 종목을 합치므로 STEP1 ka10027 응답에 없는 종목이 STEP2 hot set 에 포함될 수 있음.
   - 그런 종목에 STEP2 upsertQuotesStep2 호출 시 신규 row INSERT 시도 → price NOT NULL violation.
   - 해결: `step2Updates = step2UpdatesRaw.filter(u => step1Codes.has(u.code))`. `droppedFromStep2` 카운트 로그 추가.

5. **upsertQuotesStep2 UPSERT → UPDATE 변경** (commit `ef120aa`, **결정적 fix**)
   - Supabase `upsert(onConflict: code)` 는 `INSERT...ON CONFLICT DO UPDATE` 로 변환되어 신규 row INSERT 시 모든 NOT NULL constraint 평가. STEP2 페이로드에 price 없으면 violation.
   - PLAN 06 SUMMARY 의 "STEP1/STEP2 컬럼 분리로 race-free UPSERT" 가정이 신규 종목 INSERT case 에 깨짐.
   - 해결: ~250 종목 직렬 `.update().eq("code", code)` 호출. UPDATE 는 INSERT 분기 평가 안 함 → STEP1 매분 컬럼 자연 보존.
