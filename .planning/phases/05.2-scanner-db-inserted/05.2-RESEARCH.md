## Summary

이 페이즈는 7개 파일 변경 + 마이그레이션 1개 신규로 완결되는 소규모 개선이다. CONTEXT.md D-01~D-22에 설계가 완전히 잠겨 있고, 코드베이스 직접 검증으로 각 변경 지점이 정확히 식별됐다.

주요 발견 3가지: (1) `apiFetch`가 현재 JSON body만 반환하므로 `fetchScannerStocks`의 반환 타입을 `{ stocks, lastUpdatedAt }` 복합 타입으로 교체해야 하며, 이때 `usePolling`의 제네릭 타입 `T`가 자동으로 새 타입에 적응한다 — 다른 call site가 없어 충돌 없음. (2) `cors-config.ts`에 `exposedHeaders` 키가 현재 없으므로 한 줄 추가만 필요하며 환경변수 불필요. (3) `KisRankingRow`에 인덱스 시그니처 `[key: string]: string`이 이미 선언되어 있어 `acml_tr_pbmn` 필드를 명시적으로 추가하지 않아도 런타임 에러는 없지만, D-04 결정대로 명시적 타입 추가가 정확함.

**Primary recommendation:** 변경 순서를 DB migration → ingestion 타입/map/upsert → shared 타입 → server(COLS + mapper + CORS + 헤더) → webapp(api + polling + UI) 순서로 진행. 역방향 배포 없이 이 순서를 지키면 빌드 타입 오류도 없고 production 무중단.

## 영역별 Findings

### 1. Migration 전략 — ALTER TABLE 충분성

**Finding:** `ALTER TABLE stocks ADD COLUMN trade_amount bigint NOT NULL DEFAULT 0` 단독 실행으로 충분하다. 기존 마이그레이션 `20260413120000_init_tables.sql`은 `CREATE TABLE IF NOT EXISTS stocks`를 사용하고 있어 재실행해도 idempotent하나, ALTER는 별도 파일이 맞다. [VERIFIED: 코드베이스 직접 확인]

- RLS 정책이 `stocks` 테이블에 걸려 있더라도 Postgres의 `ALTER TABLE ADD COLUMN`은 row-level policy가 아닌 column-level ACL에만 영향받는다. Supabase 공식 문서에 따르면 RLS는 row 접근을 제어하며 컬럼 추가 시 기존 policy가 자동 적용된다. [CITED: CONTEXT.md D-03]
- DEFAULT 0 덕분에 기존 row backfill 불필요. 다음 ingestion cycle(1분)에서 `acml_tr_pbmn` 값으로 자동 덮어쓰인다. [VERIFIED: upsert.ts — `onConflict: "code"` 전략]

**Implementation hint:** 마이그레이션 파일에 주석으로 `-- Phase 05.2: SCAN-04 trade_amount` 명시. `volume` 컬럼 바로 아래에 추가하면 스키마 가독성 유지.

### 2. KIS `acml_tr_pbmn` 필드 스키마

**Finding:** KIS 등락률 순위 응답의 모든 숫자형 필드는 JSON 문자열로 반환된다. `acml_vol: "655925"`가 실증(`map.test.ts` fixture)으로 확인됐고, 같은 응답 내 `acml_tr_pbmn`도 동일 직렬화 패턴을 따른다. [VERIFIED: 코드베이스 실측 fixture `acml_vol: "655925"`]

- `Number("")`는 `0`을 반환하므로 빈 문자열 응답(장 외 시간, 데이터 없음)도 안전하게 처리된다. `Number("0")`도 당연히 `0`.
- 일일 최대 거래대금 추정치 30조원 < `Number.MAX_SAFE_INT` (9007조) — JS 부동소수점 범위 초과 없음. [VERIFIED: 로컬 계산]
- `KisRankingRow`에 이미 `[key: string]: string` 인덱스 시그니처가 있어 런타임은 동작하나 D-04 지시대로 명시적 `acml_tr_pbmn: string` 필드 추가가 타입 안전성 확보에 필요하다.

**Implementation hint:** `kis.ts`에서 `acml_vol: string` 바로 아래에 `acml_tr_pbmn: string` 추가. `map.ts`의 `toStock` 함수 반환 객체에 `tradeAmount: Number(row.acml_tr_pbmn)` 한 줄 추가.

### 3. CORS `exposedHeaders` 적용 지점

**Finding:** CORS 설정은 `server/src/services/cors-config.ts`의 `corsOptions()` 함수 단독 진리원이다. 현재 반환 객체에 `exposedHeaders` 키가 없다. [VERIFIED: cors-config.ts 직접 확인]

```typescript
// 현재
return {
  origin: ...,
  credentials: false,
  maxAge: 600,
};

// 추가 후
return {
  origin: ...,
  credentials: false,
  maxAge: 600,
  exposedHeaders: ['X-Last-Updated-At', 'X-Request-Id'],  // X-Request-Id는 기존 헤더
};
```

- `X-Request-Id`도 이미 서버가 세팅하고 있으므로 함께 노출하는 것이 일관성 있음(현재는 브라우저에서 읽을 수 없는 상태).
- 환경변수 분리 불필요 — 이 헤더 목록은 코드 계약이지 환경별 설정이 아니다.
- `helmet()`은 CORS 헤더를 건드리지 않으므로 충돌 없음. [ASSUMED: helmet은 security 헤더만 담당한다는 일반 원칙 적용]

**Implementation hint:** `cors-config.ts` 한 파일, 한 줄 추가. `app.ts` 변경 불필요.

### 4. `apiFetch` 헤더 접근 — 시그니처 변경 전략

**Finding:** `apiFetch<T>`는 현재 `response.json()`의 결과 `T`만 반환한다. `response` 객체 자체를 caller에게 노출하지 않는다. [VERIFIED: webapp/src/lib/api.ts 직접 확인]

세 옵션 평가:

| 옵션 | 설명 | 평가 |
|------|------|------|
| (a) `fetchScannerStocks` 가 `fetch` 직접 호출로 대체 | `apiFetch` 우회 | 에러 처리/타임아웃 로직 중복 — 비추천 |
| (b) `apiFetch`에 `raw: true` 옵션 추가 → `Response` 반환 | 제네릭 오염 | 다른 call site에 영향, 과도 |
| **(c)** `fetchScannerStocks`가 `apiFetch` 대신 `fetch` + 수동 에러 처리 OR `apiFetch` 호출 후 별도 `fetch` 헤더 조회 | 비효율 | 이중 fetch |
| **(권장) (d)** `scanner-api.ts` 내부에서 `fetch`를 직접 사용하되 `apiFetch`의 에러처리 패턴을 인라인 복제 | 국소적 변경 | 가장 단순. `apiFetch` 계약 무변경, 다른 call site 안전 |

**실제 가장 단순한 구현:** `fetchScannerStocks`를 `apiFetch` wrapper에서 raw fetch로 교체하고 `{ stocks: StockWithProximity[], lastUpdatedAt: string | null }` 반환. `api.ts`의 에러 처리 로직은 `apiFetch` 그대로 두고 scanner 전용 래퍼에서 `apiFetch` 호출 + 응답 헤더 추출을 동시에 하는 방식이 안된다는 점에 주의(json() 호출 후 `response`에 접근 불가).

**최종 권장 패턴:**
```typescript
// scanner-api.ts — raw fetch 사용
export async function fetchScannerStocks(
  state: ScannerState,
  signal: AbortSignal,
): Promise<{ stocks: StockWithProximity[]; lastUpdatedAt: string | null }> {
  const url = `${resolveBaseUrl()}/api/scanner?${params}`;
  const response = await fetch(url, { signal, cache: 'no-store', ... });
  if (!response.ok) throw new ApiClientError(...);
  const stocks = (await response.json()) as StockWithProximity[];
  const lastUpdatedAt = response.headers.get('X-Last-Updated-At');
  return { stocks, lastUpdatedAt };
}
```
`resolveBaseUrl()`은 `api.ts`에서 export 추가하거나 인라인 복제.

**Implementation hint:** `api.ts`에서 `resolveBaseUrl`을 export로 올리면 `scanner-api.ts`에서 import해 재사용 가능 — 1줄 변경으로 DRY 유지.

### 5. `usePolling<T>` 제네릭 확장 영향 분석

**Finding:** `usePolling`은 scanner 외 단 한 곳, `scanner-client.tsx`에서만 사용된다. [VERIFIED: grep 결과 — use-polling.ts 사용처 2개: use-polling.test.ts, scanner-client.tsx]

`fetchScannerStocks` 반환 타입이 `StockWithProximity[]` → `{ stocks, lastUpdatedAt }` 로 변경되면:

- `usePolling<{ stocks: StockWithProximity[]; lastUpdatedAt: string | null }>`으로 자동 추론됨
- `result.data.stocks`, `result.data.lastUpdatedAt` 접근으로 `scanner-client.tsx` 내 구조분해 변경 필요
- `usePolling` 훅 자체는 **변경 불필요** — `T`가 복합 타입으로 바뀔 뿐
- `lastUpdatedAt` 상태는 현재 `usePolling` 내부에서 `Date.now()`로 독립 관리 중 → 이 역할을 `data.lastUpdatedAt`으로 이전하면 `UsePollingResult.lastUpdatedAt: number | undefined`는 삭제하거나 유지(D-17 의도는 삭제).

**주의:** `usePolling` 결과의 `lastUpdatedAt`을 제거하면 `use-polling.test.ts`도 수정 필요. 또는 `scanner-client.tsx`에서 `data?.lastUpdatedAt`을 ISO 문자열 → epoch ms(`new Date(isoStr).getTime()`)로 변환 후 `ScannerFilters`에 `number | undefined`로 전달하는 방식이 `ScannerFiltersProps.lastUpdatedAt?: number` 계약을 유지해 하위 컴포넌트 무변경 가능.

**Implementation hint:** `scanner-client.tsx`에서 `lastUpdatedAt: new Date(data?.lastUpdatedAt ?? '').getTime() || undefined` 변환 인라인. `usePolling` 내부 `setLastUpdatedAt(Date.now())` 라인만 제거, `UsePollingResult` 타입은 유지해도 됨(외부 타입 계약 안정성).

### 6. 거래대금 Formatter 설계

**Finding:** `number.tsx`의 `formatByType`는 `switch(format)` 구조로 `'trade-amount'` 케이스 추가가 자연스럽다. 단, D-16 "단일 진리원" 원칙상 `number.tsx` 내 추가 또는 `lib/format.ts` 순수 util 분리 중 planner 결정이 남아 있다. [VERIFIED: number.tsx 직접 확인]

**엣지케이스 완전 검증 (로컬 실행):**

| 입력 | 기대 출력 | 검증 |
|------|-----------|------|
| `0` | `"-"` | ✓ |
| `null` | `"-"` | ✓ (Number(null) = 0) |
| `1e8` (1억) | `"1억"` | ✓ |
| `1234e8` (1234억) | `"1,234억"` | ✓ |
| `9999e8` (9999억) | `"9,999억"` | ✓ (1조 미만) |
| `1e12` (정확히 1조) | `"1.0조"` | ✓ |
| `2.3e12` (2.3조) | `"2.3조"` | ✓ |
| `1234567890` (12억대) | `"12억"` | ✓ (억 미만 버림) |

**구현 참조:**
```typescript
case 'trade-amount': {
  if (!value || value === 0) return '-';
  const trillion = 1e12;
  const uk = 1e8;
  if (value >= trillion) return `${(value / trillion).toFixed(1)}조`;
  return `${new Intl.NumberFormat('ko-KR').format(Math.floor(value / uk))}억`;
}
```

**권장:** `number.tsx`에 format prop으로 추가 — 기존 `marketCap` 포맷(`조원` 스케일)과 유사 패턴이고 vitest에서 `formatByType` 단위 테스트가 가능하다(`number.tsx` export 추가). 단, vitest 테스트 용이성은 순수 util 분리가 더 명확하므로 planner가 결정.

### 7. 배포 순서 Pitfall

**Finding:** `scanner-table.tsx` + `scanner-card-list.tsx`가 `stock.volume`을 `Number` 포맷으로 렌더한다. `stock.trade_amount`는 기존 응답에 없으므로 webapp이 먼저 배포되면 `undefined`를 포맷해 `"-"` 또는 `"0"` 표시가 된다 — 런타임 크래시는 없지만 UX가 깨진다. [VERIFIED: scanner-table.tsx, scanner-card-list.tsx 직접 확인]

**안전한 배포 순서:**

```
1. supabase db push (migration) — 기존 server/ingestion은 trade_amount 없이도 동작
2. server 재배포 (Cloud Run) — COLS에 trade_amount 추가, CORS 헤더 추가
3. ingestion worker 재배포 (Cloud Run Job) — trade_amount upsert 시작
4. webapp 재배포 (Vercel) — 새 필드 렌더
```

**Pitfall 목록:**

- **P-1: 역방향 배포** — webapp 먼저 배포 시 `trade_amount: undefined` → formatter가 `"-"` 표시(런타임 안전하나 잘못된 정보). stock 타입 변경 후 `trade_amount: number` non-optional이므로 TypeScript 빌드는 통과(Vercel 빌드는 types 기준) → 런타임에서만 `undefined` 가능.
- **P-2: COLS 누락** — `scanner.ts`의 `COLS` 상수에 `trade_amount` 추가를 빠트리면 서버 응답에 필드 없음 → webapp이 `undefined` 받음.
- **P-3: upsert.ts 누락** — `upsert.ts`의 rows 매핑에 `trade_amount: s.tradeAmount` 추가 빠트리면 DB에 DEFAULT 0만 남음.
- **P-4: StockRow 타입 누락** — `server/src/mappers/stock.ts`의 `StockRow` 타입과 `rowToStock` 함수에 `trade_amount` 추가 필요. 현재 `StockRow`는 DB 컬럼 스네이크케이스 타입이므로 `trade_amount: number` 추가 필요.
- **P-5: `apiFetch` → raw fetch 전환 시 타임아웃 처리** — `apiFetch`의 8초 타임아웃 로직을 직접 구현하지 않으면 네트워크 hang 가능. `AbortSignal.timeout(8000)` 또는 수동 `AbortController` 필요.
