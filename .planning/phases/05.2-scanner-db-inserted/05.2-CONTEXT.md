## Phase Boundary

Phase 5.1에서 ingestion이 production으로 돌기 시작한 뒤 Scanner UI를 실사용하며 드러난 2개 이슈를 해결한다.

**포함:**
- Supabase `stocks` 테이블에 `trade_amount` 필드 추가 (거래대금, bigint)
- KIS 등락률 순위 응답 `acml_tr_pbmn` 파싱 → ingestion이 매 cycle upsert
- `@gh-radar/shared` Stock 타입에 `trade_amount` 추가
- `/api/scanner` 응답 헤더 `X-Last-Updated-At: <ISO>` 추가 (body 계약은 그대로)
- Scanner UI: 거래량 컬럼 → 거래대금 컬럼으로 완전 교체 (table + card + skeleton + a11y)
- 거래대금 포맷터 유틸 (1조 미만 "N,NNN억" / 1조 이상 "N.N조")
- 갱신시각 표시: `Date.now()` 대신 API 응답 헤더 값 사용
- REQUIREMENTS.md 신규 SCAN-08 등록 (갱신시각 서버 기준)

**제외:**
- 수동 Job 트리거 엔드포인트 (지연/비용/동시성 trade-off로 deferred — 새로고침은 현행 fetch-only 유지)

<decisions>
## Implementation Decisions

### DB 스키마
- **D-01:** `stocks` 테이블에 `trade_amount bigint NOT NULL DEFAULT 0` 신규 컬럼 추가. `volume`은 **유지**(제거하지 않음) — 향후 분석/대시보드 용도 유연성 확보, Phase 1의 실증 데이터 보존.
- **D-02:** 마이그레이션 파일명: `supabase/migrations/YYYYMMDDhhmmss_add_trade_amount.sql`. 기존 row는 DEFAULT 0으로 채워지고 다음 ingestion cycle에서 자동 갱신됨. 별도 backfill 불필요.
- **D-03:** RLS 정책은 Phase 1 D-17 승계(anon SELECT, service_role 쓰기) — 컬럼 추가는 기존 정책에 자동 포함.

### KIS 매핑 + Ingestion
- **D-04:** `packages/shared/src/kis.ts`의 `KisRankingRow` 타입에 `acml_tr_pbmn: string` 필드 추가. KIS 공식 문서에 `누적 거래대금` 필드로 명시됨.
- **D-05:** `workers/ingestion/src/pipeline/map.ts`의 `mapRankingRow`에서 `trade_amount: Number(row.acml_tr_pbmn)` 매핑 추가. `volume`과 동일하게 `Number(...)` 캐스팅(bigint 범위 초과 없음 — 일일 거래대금 최대치는 안전).
- **D-06:** ingestion 단위 테스트 1개 추가 — KIS fixture에 `acml_tr_pbmn` 포함, map/upsert 경로가 trade_amount를 정확히 DB에 저장하는지 검증.

### 공용 타입
- **D-07:** `packages/shared/src/stock.ts`의 `Stock` 타입에 `trade_amount: number` 필드 추가(non-optional). 이는 프론트/백엔드/워커가 공유하는 계약.
- **D-08:** `stock.volume`은 타입에서 **유지**한다 — 서버가 여전히 응답에 포함하며 미래 확장성 보장. UI는 쓰지 않을 뿐.

### API 응답 (server/src)
- **D-09:** `/api/scanner` body 계약은 **불변** — 여전히 `Stock[]` 루트 배열 반환(Phase 2 D-13 envelope 지양 원칙 유지).
- **D-10:** 갱신시각은 **HTTP 응답 헤더 `X-Last-Updated-At`** 로 반환 — 값은 해당 응답에 포함된 stocks 중 `MAX(updated_at)`의 ISO 8601 문자열(UTC, 예: `2026-04-14T10:01:33.796Z`). Phase 2 D-13 주석 "메타데이터 필요 시 헤더 사용"을 첫 적용.
- **D-11:** 데이터가 0건일 때 헤더는 **생략** (빈 문자열도 세팅 금지) — 클라이언트는 헤더 부재 시 "갱신 시각 없음"으로 표시.
- **D-12:** CORS 응답에서 `X-Last-Updated-At`을 `Access-Control-Expose-Headers`로 명시 — 미하면 브라우저에서 읽을 수 없음.
- **D-13:** 헤더 값 계산은 쿼리 결과 배열에서 JS `Math.max(...updated_at.getTime())` 방식. Supabase SQL 단에서 별도 aggregate 쿼리 추가하지 않음(단순성 우선).

### Scanner UI
- **D-14:** `scanner-table.tsx`/`scanner-card-list.tsx`/`scanner-skeleton.tsx`의 **거래량 컬럼을 거래대금으로 완전 교체**. 헤더 라벨 "거래량" → "거래대금". `stock.volume` 참조 전부 `stock.trade_amount`로.
- **D-15:** 거래대금 포맷 규칙(신규 util `formatTradeAmount(n)`):
  - 0 또는 null: `"-"` (더미 0 노출 방지)
  - < 1조 (10^12 KRW): `"N,NNN억"` (정수 억 + 천단위 콤마, 예: `1,234억`)
  - >= 1조: `"N.N조"` (소수점 1자리, 예: `2.3조`)
  - 단위/스타일 참고: 네이버증권, 키움 HTS 컨벤션
- **D-16:** `<Number value={...} format="trade-amount" />` 신규 format을 `webapp/src/components/ui/number.tsx`에 추가하거나 전용 util로 import — planner 결정. 중요한 건 **단일 진리원(one-place formatter)**.
- **D-17:** 갱신시각 소스 교체 — `webapp/src/hooks/use-polling.ts`의 `setLastUpdatedAt(Date.now())`를 제거하고, `fetchScannerStocks`가 fetch 응답의 `X-Last-Updated-At` 헤더를 반환 값에 포함하도록 시그니처 확장. `usePolling`은 fetcher의 return에서 `lastUpdatedAt`를 꺼내 상태로 세팅.
- **D-18:** 헤더가 없을 때(서버가 빈 결과) UI는 "갱신 시각: -"로 표시, 새로고침 버튼은 항상 클릭 가능(refresh 시 다시 fetch).
- **D-19:** Scanner-filters의 "마지막 갱신" 표시 라벨/포맷은 기존 유지(예: `오전 7:01:33`), 다만 소스만 서버 헤더로 교체. 시각은 브라우저 로컬타임(`toLocaleTimeString('ko-KR')`)으로 변환 표시.

### REQUIREMENTS 등록
- **D-20:** `.planning/REQUIREMENTS.md` Scanner 섹션에 **SCAN-08: 갱신시각 서버 DB 기준 표시** 신규 요구사항 추가 및 Phase 5.2에서 [x]로 마킹.
- **D-21:** SCAN-04("현재가+등락률+거래량") 의미 재해석 — 요구사항 텍스트는 유지하되 CONTEXT에서 "거래량 = 거래대금"으로 해석 기록. REQUIREMENTS 본문에 각주 추가("Phase 05.2에서 거래대금으로 전환").

### 수동 Job 트리거 (scope 제외)
- **D-22:** 사용자 새로고침 시 Cloud Run Job을 실제 재실행하는 방식은 **이 페이즈에서 구현하지 않음**. 이유: Job 실행 지연(~40s), 장 외 시간 KIS API 호출 비용/rate-limit 위험, 동시 클릭 시 queue 지연, 공개 앱 남용 방지 쓰로틀링 등 설계 비용이 큼. 현재 fetch-only 새로고침(DB 재조회)로 충분 — 2번 갱신시각 픽스 완료 시 UX 오해도 자동 해소.

### Claude's Discretion
- 마이그레이션 타임스탬프(초 단위까지)와 파일명 slug(`add_trade_amount` 등)는 planner가 결정.
- 거래대금 포맷터를 `webapp/src/components/ui/number.tsx`에 format prop으로 추가할지, `webapp/src/lib/format.ts` 같은 순수 util로 분리할지 planner 판단. vitest 테스트는 단위 util로 분리되면 더 쉬움.
- server에서 `MAX(updated_at)` 계산 위치(route handler vs service 함수)는 planner 결정.
- CORS `exposedHeaders` 업데이트 지점(app.ts 미들웨어 설정 or 환경변수)은 planner 결정.

</decisions>

<specifics>
## Specific Ideas

- **거래대금 포맷 컨벤션**: 네이버증권, 키움 HTS "억/조" 표기 — 1조 미만은 "N,NNN억"(정수), 1조 이상은 "N.N조"(소수점 1자리). 사용자가 직관적으로 "대량 거래" 판단 가능.
- **"억 단위 미만은 버린다"**: 예 1,234,567,890원 → 12억 (만 단위는 버림). HTS에서도 흔한 방식이지만 planner가 원본 정확도 vs 가독성 trade-off 판단.
- **volume 필드 정책**: 지금은 UI에서 쓰지 않지만 DB에 유지 — 향후 "대량 거래량 종목" 같은 세부 필터나 분석에 필요할 수 있음. 제거 비용보다 유지 비용이 낮음.
- **헤더 vs body 결정 배경**: 사용자가 명시적으로 Phase 2 D-13 envelope 지양 원칙을 존중하고 "메타데이터는 헤더" 패턴을 선택.

</specifics>
