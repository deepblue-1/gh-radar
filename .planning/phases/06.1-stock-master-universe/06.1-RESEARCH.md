### Locked Decisions

#### 데이터베이스 구조 (3-테이블 분리)
- **D1:** `stocks` = 마스터(정적 메타) · `stock_quotes` = 종목별 최신 시세 · `top_movers` = 스캐너 랭킹 스냅샷(시세 컬럼 없음)
- **D2:** 마이그레이션 경로 — 기존 `stocks`를 `top_movers`로 rename → 시세 컬럼은 `stock_quotes`로 분리 이관 → 새 `stocks`를 마스터로 신설
- **D6:** 마스터 메타 범위 — 풍부하게. code, name, market(KOSPI/KOSDAQ), sector/업종, listing_date, 종목구분(보통/우선주/ETF/ETN), 상장폐지 여부. 최종 필드는 researcher가 소스 스펙 확인 후 확정.

#### API 전환
- **D3:** `/api/stocks/search` — 새 `stocks`(마스터) 대상으로 **완전 전환**. `stock_quotes` LEFT JOIN으로 현재가 필드는 유지(시세 부재 시 em-dash)
- **D4:** `/api/stocks/:code` — 마스터 존재 확인 + `stock_quotes` 조회 + **진입 시 항상 KIS `inquirePrice` on-demand fetch 후 `stock_quotes` upsert**. Phase 1 hotfix `3f9691d`의 inquirePrice 경로 재사용
- **D9:** On-demand 정책 — 캐시 stale 판정 없이 상세 진입 시 무조건 1회 호출. 현재 rate limit 10 req/sec 운영치 대비 여유 충분 *(주: 실제 코드 상수는 5 req/sec — Pitfall 참조)*

#### Scanner 연계
- **D5:** Phase 05.1 ingestion은 `stock_quotes` upsert + `top_movers` (code, rank, ranked_at, scan_id) 기록으로 분해. `/api/scanner`는 `top_movers JOIN stock_quotes JOIN stocks`

#### 마스터 동기화
- **D7:** 데이터 소스 — KRX OpenAPI(상장종목 조회) 우선 검토, 필요 시 KIS 종목기본조회 보완. pykrx는 Node 스택 불일치로 후순위. 최종 소스 선정은 researcher
- **D8:** 동기화 방식 — Cloud Run Job + Cloud Scheduler 일 1회 (장 시작 전). Phase 05.1 ingestion 패턴 재사용

#### 범위
- **D10:** 이번 phase에 전부 포함 — 스키마 rename+분리+신설, 초기 마스터 백필, scanner 코드 경로 업데이트, search/detail 엔드포인트 전환, on-demand fetch 배선, Cloud Run Job 배포, Scheduler 등록, E2E 회귀(삼성전자 검색 PASS)

## Summary

Phase 06.1은 3개의 독립 과제가 하나의 페이즈로 묶인 **중간 규모 리팩터링**이다:

1. **DB 스키마 3-테이블 분리** — 현 `stocks`(마스터+시세+랭킹 혼재)를 `stocks`(마스터) + `stock_quotes`(시세) + `top_movers`(랭킹) 로 분리. rename+신설이라 마이그레이션 순서 + Phase 05.1 ingestion 잡 동시 배포가 관건.
2. **마스터 universe 채우기** — KRX OpenAPI `stk_isu_base_info` / `ksq_isu_base_info` 엔드포인트로 KOSPI+KOSDAQ 상장종목 ~2,800건을 일 1회 upsert. **KRX 인증키는 수동 신청 + 서비스별 승인이 필요**해 실배포 전 사용자 개입이 불가피.
3. **Search/Detail API 재배선** — 검색은 마스터 대상, 상세는 마스터 존재 확인 + on-demand `inquirePrice` fetch로 전환. `inquirePrice` 구현이 현재 `workers/ingestion/src/kis/` 에만 있어 **server 측이 직접 부를 수 있도록 공유 위치로 이동**해야 함.

**Primary recommendation:** (a) **KRX OpenAPI를 1차 소스로 채택**하되 KIS 종목기본조회(`search-info`)를 폴백으로 준비. (b) 마이그레이션은 **단일 트랜잭션 SQL 1개** (`ALTER TABLE stocks RENAME TO top_movers` → `CREATE TABLE stock_quotes AS SELECT ... FROM top_movers` → `ALTER TABLE top_movers DROP COLUMN <시세>` → `CREATE TABLE stocks (마스터)`) 로 다운타임 최소화. (c) **마스터 동기화 잡을 Phase 05.1 패턴 미러링**해 새 Cloud Run Job(`gh-radar-master-sync`) + 일 1회 Scheduler(평일 07:30 KST, 장 시작 전) 로 배포. (d) **KIS `inquirePrice` 함수를 `packages/shared` 또는 `server/src/kis/` 로 복제/이동**해 server·workers 양쪽에서 재사용.

### Pattern 1: 3-테이블 분리 + JOIN 뷰

**What:** 역할을 명확히 분리하고 JOIN 으로 조합.

**When to use:** 하나의 테이블이 "정적 메타 + 동적 시세 + 랭킹 스냅샷" 세 역할을 겸하고 있을 때. 본 phase 의 정확한 맥락.

**Example (마이그레이션):**

```sql
-- Source: 본 RESEARCH, Phase 1 migrations 컨벤션 승계
BEGIN;

-- Step 1. 기존 stocks → top_movers 로 rename (랭킹 스냅샷 역할만 남김)
ALTER TABLE stocks RENAME TO top_movers;
ALTER INDEX idx_stocks_change_rate_desc RENAME TO idx_top_movers_change_rate_desc;
ALTER INDEX idx_stocks_market RENAME TO idx_top_movers_market;

-- Step 2. stock_quotes 분리 생성 (현 시세 컬럼을 옮김)
CREATE TABLE stock_quotes (
  code          text PRIMARY KEY,  -- FK 는 Step 4 의 stocks 생성 후 걸기
  price         numeric(20,2) NOT NULL,
  change_amount numeric(20,2) NOT NULL,
  change_rate   numeric(8,4)  NOT NULL,
  volume        bigint NOT NULL DEFAULT 0,
  trade_amount  bigint NOT NULL DEFAULT 0,
  open          numeric(20,2),
  high          numeric(20,2),
  low           numeric(20,2),
  market_cap    bigint,
  upper_limit   numeric(20,2) NOT NULL,
  lower_limit   numeric(20,2) NOT NULL,
  updated_at    timestamptz NOT NULL DEFAULT now()
);
INSERT INTO stock_quotes (code, price, change_amount, change_rate, volume, trade_amount,
                          open, high, low, market_cap, upper_limit, lower_limit, updated_at)
SELECT code, price, change_amount, change_rate, volume, trade_amount,
       open, high, low, market_cap, upper_limit, lower_limit, updated_at
FROM top_movers;

-- Step 3. top_movers 에서 시세 컬럼 제거 + 랭킹 전용 컬럼 추가
ALTER TABLE top_movers
  DROP COLUMN price,
  DROP COLUMN change_amount,
  DROP COLUMN change_rate,
  DROP COLUMN volume,
  DROP COLUMN trade_amount,
  DROP COLUMN open,
  DROP COLUMN high,
  DROP COLUMN low,
  DROP COLUMN market_cap,
  DROP COLUMN upper_limit,
  DROP COLUMN lower_limit;
-- 인덱스/RLS 정리 후 rank/ranked_at/scan_id 컬럼 추가 (아래 Pitfall 참조)

-- Step 4. stocks 를 새 마스터 테이블로 재생성
CREATE TABLE stocks (
  code          text PRIMARY KEY,
  name          text NOT NULL,
  market        text NOT NULL CHECK (market IN ('KOSPI', 'KOSDAQ')),
  sector        text,                -- nullable: ETF/일부 종목 누락 가능
  security_type text NOT NULL,       -- '보통주'/'우선주'/'ETF'/'ETN'/'REIT' 등 — KRX SECUGRP_NM 기반
  listing_date  date,                -- nullable: 과거 종목 정보 누락 가능
  is_delisted   boolean NOT NULL DEFAULT false,
  updated_at    timestamptz NOT NULL DEFAULT now()
);

-- Step 5. 기존 top_movers 의 종목을 새 stocks 로 부트스트랩 (다음 마스터 동기화 이전의 공백 메움)
INSERT INTO stocks (code, name, market, security_type, updated_at)
SELECT code, name, market, '보통주', now()
FROM top_movers
ON CONFLICT (code) DO NOTHING;

-- Step 6. FK + 인덱스 부여
ALTER TABLE stock_quotes
  ADD CONSTRAINT fk_stock_quotes_code FOREIGN KEY (code) REFERENCES stocks(code) ON DELETE CASCADE;
ALTER TABLE top_movers
  ADD COLUMN rank int,
  ADD COLUMN ranked_at timestamptz NOT NULL DEFAULT now(),
  ADD COLUMN scan_id uuid;
-- (Discretion) top_movers 를 "매 cycle 교체" 스냅샷 모델로 할지, scan_id 별 누적으로 할지 planner 결정

-- 인덱스
CREATE INDEX idx_stocks_market ON stocks (market);
CREATE INDEX idx_stocks_updated_at ON stocks (updated_at DESC);
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_stocks_name_trgm ON stocks USING GIN (name gin_trgm_ops);
CREATE INDEX idx_stock_quotes_change_rate_desc ON stock_quotes (change_rate DESC NULLS LAST);

-- RLS 승계 (Phase 1 D-17)
ALTER TABLE stocks        ENABLE ROW LEVEL SECURITY;
ALTER TABLE stock_quotes  ENABLE ROW LEVEL SECURITY;
-- top_movers 는 rename 시 RLS 자동 승계됨(확인 필요)
CREATE POLICY "anon_read_stocks_master" ON stocks FOR SELECT TO anon USING (true);
CREATE POLICY "anon_read_stock_quotes" ON stock_quotes FOR SELECT TO anon USING (true);

-- news_articles.stock_code FK 는 기존에 stocks 를 참조 → 새 stocks 로 자연 승계 (테이블명 유지 효과)

COMMIT;
```

**주의:** `news_articles.stock_code REFERENCES stocks(code) ON DELETE CASCADE` 이 기존 마이그레이션에 있다. `DROP TABLE stocks` 을 하면 CASCADE 가 news_articles FK 를 파괴한다. 본 전략은 **기존 stocks 를 rename** 하는 것이므로 news_articles FK 는 `top_movers` 를 참조하게 되어버린다 → **Step 4 의 새 stocks 생성 후 FK 재부여**가 필요 (현재는 news_articles 가 비어있어 무해하지만 명시적 re-point 필수).

### Pattern 2: On-demand 시세 fetch + upsert (server 측)

**What:** 상세 페이지 요청 시 server 가 직접 KIS `inquirePrice` 호출 → `stock_quotes` upsert → 응답 병합.

**Example:**

```typescript
// Source: server/src/routes/stocks.ts 전환 가이드
// Depends on: server/src/kis/inquirePrice.ts (신규 — ingestion 에서 복제)
stocksRouter.get("/:code", async (req, res, next) => {
  try {
    const code = req.params.code;
    if (!/^[A-Za-z0-9]{1,10}$/.test(code)) throw new ApiError(400, "INVALID_QUERY_PARAM", "code: invalid format");

    const supabase = req.app.locals.supabase as SupabaseClient;

    // 1. 마스터 존재 확인
    const { data: master, error: mErr } = await supabase
      .from("stocks")
      .select("code,name,market,sector,security_type,listing_date,is_delisted")
      .eq("code", code)
      .maybeSingle();
    if (mErr) throw mErr;
    if (!master) throw StockNotFound(code);

    // 2. On-demand inquirePrice (D9 — 캐시 stale 판정 없이 무조건 1회)
    let quote: StockQuote | null = null;
    try {
      const price = await fetchInquirePrice(kisClient, code);
      await supabase.from("stock_quotes").upsert({
        code,
        price: Number(price.stck_prpr),
        change_amount: Number(price.prdy_vrss),
        change_rate: Number(price.prdy_ctrt),
        volume: Number(price.acml_vol),
        trade_amount: Number(price.acml_tr_pbmn),
        open: Number(price.stck_oprc),
        high: Number(price.stck_hgpr),
        low: Number(price.stck_lwpr),
        market_cap: Number(price.stck_avls),
        upper_limit: Number(price.stck_mxpr),
        lower_limit: Number(price.stck_llam),
        updated_at: new Date().toISOString(),
      }, { onConflict: "code" });
      quote = /* 위 값 그대로 */;
    } catch (err) {
      // 3. 폴백: 캐시된 stock_quotes 가 있으면 그걸로 응답
      logger.warn({ code, err }, "inquirePrice failed — falling back to cached stock_quotes");
      const { data: cached } = await supabase
        .from("stock_quotes").select("*").eq("code", code).maybeSingle();
      quote = cached;
    }

    // 4. 응답 병합 — 시세가 완전히 없으면 em-dash 용 null
    res.json(mergeMasterAndQuote(master, quote));
  } catch (e) { next(e); }
});
```

### Pattern 3: 마스터 동기화 Cloud Run Job (Phase 05.1 미러)

**What:** `workers/master-sync/` 을 Phase 05.1 `workers/ingestion/` 과 동일 패턴으로 구성. Scheduler 는 **평일 07:30 KST** 1회 (ingestion 의 `* 9-15 * * 1-5` 보다 앞서서 장 시작 전 upsert 완료).

#   - runtime: gh-radar-ingestion-sa (Supabase + KRX secret 접근)

# Secret 신규: gh-radar-krx-auth-key ← KRX_AUTH_KEY

### Anti-Patterns to Avoid

- **news_articles FK 파괴:** 기존 `stocks` 를 `DROP` 하면 `ON DELETE CASCADE` 가 news_articles 데이터를 삭제한다. rename 경로 선택 + FK 명시적 re-point 필수.
- **마스터와 ingestion 경쟁 upsert:** 둘 다 `stocks` 에 쓰던 기존 모델에서 3-테이블 전환 시 ingestion 이 실수로 `stocks` 를 upsert 하면 마스터 데이터(sector/listing_date/security_type)가 덮어씌워진다. **ingestion 은 `stock_quotes` 와 `top_movers` 에만 쓴다 — `stocks` 쓰기 금지 불변식** 필수.
- **stocks upsert 경로에서 FK 미존재 종목:** KRX 마스터 초기 백필 이전에 ingestion 이 동작하면 `stock_quotes` FK 제약이 터진다. 마이그레이션 시 Step 5 부트스트랩 또는 초기 백필 1회를 ingestion 재개 전에 완료.
- **검색 응답 스키마 부주의 변경:** `/api/stocks/search` 가 현재 `StockWithProximity` 반환. 시세가 없는 종목에 대해 `price=0` 대신 `null` 반환 시 webapp `format.ts` + `mock-api.ts` + 9개 E2E 가 깨진다. **현 Stock 타입 필드는 유지하고 값이 NULL 시 0 또는 `—` 폴백** 하도록 mapper 만 수정(CONTEXT Specifics 와 일치).
- **KRX OpenAPI 로 당일 시세 기대:** KRX 는 "전일 데이터 익영업일 08:00 KST 갱신" — 당일 시세는 불가. 마스터용 메타만 수용 [CITED: openkrx-mcp README, i-whale.com 블로그].
- **AUTH_KEY 공백 미제거:** KRX 인증키에 공백/개행 섞이면 401 — Python 샘플이 `auth_key.strip()` 을 명시한 이유 [CITED: i-whale.com].

## Common Pitfalls

### Pitfall 1: news_articles FK 의 암시적 승계 실패
**What goes wrong:** `ALTER TABLE stocks RENAME TO top_movers` 후 `CREATE TABLE stocks (...)` 로 새 마스터를 만들면 news_articles.stock_code 는 여전히 **top_movers** 를 참조 (rename 은 FK 를 이름이 아닌 OID 로 추적).
**Why:** Postgres FK 는 내부적으로 테이블 OID 를 참조. rename 은 OID 불변.
**How to avoid:** 마이그레이션에 **명시적 FK drop + re-create** 포함. `ALTER TABLE news_articles DROP CONSTRAINT news_articles_stock_code_fkey, ADD CONSTRAINT news_articles_stock_code_fkey FOREIGN KEY (stock_code) REFERENCES stocks(code) ON DELETE CASCADE;` — discussions 도 동일.
**Warning signs:** 마이그레이션 후 `\d news_articles` 출력의 References 가 `top_movers(code)` 로 남음.

### Pitfall 2: KRX 인증키 신청 + 서비스별 승인 딜레이
**What goes wrong:** 배포 직전 개발자가 KRX 회원가입 + AUTH_KEY 발급 + "유가증권/코스닥 종목기본정보" 서비스 별도 승인 신청까지 마쳐야 하는데 **승인에 최대 1일 소요** [CITED: i-whale.com].
**Why:** KRX OpenAPI 정책 — 키 발급과 서비스 사용 신청이 별개의 승인 절차.
**How to avoid:** Phase 시작 시 **Task 0 (사용자 수동 준비)** 를 명시 — 개발자가 (1) `openapi.krx.co.kr` 회원가입, (2) AUTH_KEY 발급, (3) "유가증권 종목기본정보" + "코스닥 종목기본정보" 서비스 사용 신청, (4) 승인 후 키 공유. 이 단계 없이 plan 진행 금지.
**Warning signs:** 첫 호출에서 `401 Unauthorized` — 승인 미완.

### Pitfall 3: ingestion · server · master-sync 의 KIS 앱키 공유로 인한 rate limit 충돌
**What goes wrong:** 동일 KIS 앱키로 ingestion(매분 폴링) + server(상세 페이지 on-demand) 가 동시에 호출 → 5 req/sec 버킷을 공유하지 않으면 EGW00201 발생.
**Why:** KIS rate limit 은 **앱키 단위** 전역. 현재 rateLimiter 는 프로세스 로컬 토큰버킷이라 process 별로 별도 버킷.
**How to avoid (단기):** server 측은 on-demand 빈도가 낮으므로 별도 rateLimiter 인스턴스로 운영. 단 장중 ingestion(평균 24s/cycle, idle 36s) + server 상세 페이지 동시 폭주 시 위험. **모니터링 + 필요 시 Redis 기반 shared token bucket** 은 deferred.
**How to avoid (장기, deferred):** Upstash Redis 에 공유 토큰 버킷 — 본 phase 스코프 아님.
**Warning signs:** 상세 페이지 로딩 중 `EGW00201` 로그 급증.

### Pitfall 4: 마스터에 없는 종목 코드로 상세 페이지 진입
**What goes wrong:** 사용자가 URL 에 임의 코드 입력(예: `/stocks/999999`) — 마스터에 없으니 404 반환이 맞지만, on-demand `inquirePrice` 를 무조건 호출하는 로직이 순서를 잘못 잡으면 존재하지 않는 종목에 KIS 호출이 낭비되고 에러로그만 쌓인다.
**Why:** D4 "진입 시 항상 inquirePrice 호출" 의 해석 오류.
**How to avoid:** **순서 불변식** — 마스터 존재 확인 먼저 → 없으면 404 즉시 → 있으면 inquirePrice → upsert → 응답.

### Pitfall 5: trigram 검색 성능 저하 — 짧은 쿼리
**What goes wrong:** `ilike '%삼%'` 같은 1~2자 쿼리는 trigram 인덱스가 효과 없다(trigram 은 3글자 단위). 결과적으로 sequential scan.
**Why:** pg_trgm 은 3-gram 기반 — 2자 이하 검색어에 최적화 안 됨.
**How to avoid:** webapp 측 `useDebouncedSearch` 가 이미 `q.length >= 2` 가드 중이지만, **서버에서도 2자 미만은 빈 배열 반환** (기존 `sanitizeSearchTerm` 로직 확장). 또는 한국어 빈도 고려 시 1~2자도 허용하되 `limit 20` + `order by name` 으로 커버.
**Warning signs:** `EXPLAIN ANALYZE` 에서 짧은 쿼리의 plan 이 Seq Scan.

### Pitfall 6: KIS `inquirePrice` 모듈 server 로 복제 시 순환 의존성
**What goes wrong:** `workers/ingestion/src/kis/` 의 `rateLimiter.ts` 는 모듈 스코프 전역 상태(`let tokens = 5`)를 가짐. 이걸 `server/src/kis/` 로 단순 복사하면 **서버 프로세스 내 독립 버킷** 이 생겨 ingestion 과 별개로 동작(Pitfall 3 의 원인).
**Why:** 토큰 버킷이 프로세스 로컬.
**How to avoid:** Phase 06.1 스코프에서는 **의도적으로 별도 버킷** (server 는 on-demand 빈도 낮음). Pitfall 3 관리로 대응. 공유 모듈로 승격하되 각 프로세스에서 독립 인스턴스화 — `packages/shared/src/kis/` 가 아닌 **각 워크스페이스에 복사 허용**.

// Source: openkrx-mcp client.py 포팅 + i-whale.com 샘플 + CLAUDE.md Technology Stack
// Note: AUTH_KEY 는 HTTP header (공식) — query param 방식은 일부 라이브러리 관용
import axios from "axios";

const krxClient = axios.create({
  baseURL: "https://data-dbg.krx.co.kr/svc/apis",
  headers: {
    AUTH_KEY: process.env.KRX_AUTH_KEY!.trim(),  // ⚠ .strip() 필수
    "Content-Type": "application/json",
    Accept: "application/json",
  },
  timeout: 30_000,
});

export async function fetchMasterFromKrx(basDd: string): Promise<KrxBaseInfoRow[]> {
  const [kospi, kosdaq] = await Promise.all([
    krxClient.get("/sto/stk_isu_base_info", { params: { basDd } }),
    krxClient.get("/sto/ksq_isu_base_info", { params: { basDd } }),
  ]);
  return [
    ...(kospi.data.OutBlock_1 ?? []).map((r: any) => ({ ...r, market: "KOSPI" as const })),
    ...(kosdaq.data.OutBlock_1 ?? []).map((r: any) => ({ ...r, market: "KOSDAQ" as const })),

### KRX 응답 → Stock master row 매핑 (필드 추정 — LOW confidence)

```typescript
// Source: 공공데이터포털 KRX상장종목정보 문서 + 일반 KRX 필드 컨벤션 [CITED + ASSUMED]
// ⚠ 실제 필드명은 승인된 인증키로 1회 호출해 JSON 샘플 캡처 후 확정 필요
type KrxBaseInfoRow = {
  ISU_CD?: string;         // ISIN 코드
  ISU_SRT_CD?: string;     // 단축코드(종목코드) ← 핵심
  ISU_NM?: string;         // 종목명(풀)
  ISU_ABBRV?: string;      // 종목약명 ← 표시용 우선
  MKT_TP_NM?: string;      // "KOSPI"/"KOSDAQ"
  SECUGRP_NM?: string;     // 증권구분 (보통주/우선주/ETF/ETN 등)
  KIND_STKCERT_TP_NM?: string; // 주식종류
  LIST_DD?: string;        // 상장일 YYYYMMDD
  SECT_TP_NM?: string;     // 업종(섹터)
  market: "KOSPI" | "KOSDAQ";  // 호출 엔드포인트로 결정 (응답에 없을 수도)
};

export function krxToMasterRow(r: KrxBaseInfoRow) {
  return {
    code: r.ISU_SRT_CD!,
    name: r.ISU_ABBRV ?? r.ISU_NM!,
    market: r.market,
    sector: r.SECT_TP_NM || null,
    security_type: r.SECUGRP_NM ?? "보통주",
    listing_date: r.LIST_DD ? `${r.LIST_DD.slice(0,4)}-${r.LIST_DD.slice(4,6)}-${r.LIST_DD.slice(6,8)}` : null,
    is_delisted: false,
    updated_at: new Date().toISOString(),
  };
}
```
