# Phase 06.1: stock-master-universe - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

전 종목(KRX 상장 ~2,800종목) 마스터 테이블을 확보하여 검색 universe를 완전 종목으로 확장하고, 기존 `stocks` 테이블에 섞여 있던 마스터/시세/랭킹 역할을 3테이블 구조로 분리한다. 검색 API를 마스터 기준으로 전환하고, 상세 페이지는 진입 시 on-demand 시세 fetch를 수행한다. 관심종목(watchlist) 기능은 본 phase의 스키마로 수용 가능한 상태까지만 준비.

**긴급 삽입 사유 (2026-04-15):** 실사용 중 "삼성전자" 검색 불가 발견 → SRCH-01 미충족. 현재 `stocks` 테이블이 scanner로 ingested된 ~58종목만 담고 있어 발생. (STATE.md:97)

</domain>

<decisions>
## Implementation Decisions

### 데이터베이스 구조 (3-테이블 분리)
- **D1:** `stocks` = 마스터(정적 메타) · `stock_quotes` = 종목별 최신 시세 · `top_movers` = 스캐너 랭킹 스냅샷(시세 컬럼 없음)
- **D2:** 마이그레이션 경로 — 기존 `stocks`를 `top_movers`로 rename → 시세 컬럼은 `stock_quotes`로 분리 이관 → 새 `stocks`를 마스터로 신설
- **D6:** 마스터 메타 범위 — 풍부하게. code, name, market(KOSPI/KOSDAQ), sector/업종, listing_date, 종목구분(보통/우선주/ETF/ETN), 상장폐지 여부. 최종 필드는 researcher가 소스 스펙 확인 후 확정.

### API 전환
- **D3:** `/api/stocks/search` — 새 `stocks`(마스터) 대상으로 **완전 전환**. `stock_quotes` LEFT JOIN으로 현재가 필드는 유지(시세 부재 시 em-dash)
- **D4:** `/api/stocks/:code` — 마스터 존재 확인 + `stock_quotes` 조회 + **진입 시 항상 KIS `inquirePrice` on-demand fetch 후 `stock_quotes` upsert**. Phase 1 hotfix `3f9691d`의 inquirePrice 경로 재사용
- **D9:** On-demand 정책 — 캐시 stale 판정 없이 상세 진입 시 무조건 1회 호출. 현재 rate limit 10 req/sec 운영치 대비 여유 충분

### Scanner 연계
- **D5:** Phase 05.1 ingestion은 `stock_quotes` upsert + `top_movers` (code, rank, ranked_at, scan_id) 기록으로 분해. `/api/scanner`는 top_movers JOIN stock_quotes JOIN stocks

### 마스터 동기화
- **D7:** 데이터 소스 — KRX OpenAPI(상장종목 조회) 우선 검토, 필요 시 KIS 종목기본조회 보완. pykrx는 Node 스택 불일치로 후순위. 최종 소스 선정은 researcher
- **D8:** 동기화 방식 — Cloud Run Job + Cloud Scheduler 일 1회 (장 시작 전). Phase 05.1 ingestion 패턴 재사용

### 범위
- **D10:** 이번 phase에 전부 포함 — 스키마 rename+분리+신설, 초기 마스터 백필, scanner 코드 경로 업데이트, search/detail 엔드포인트 전환, on-demand fetch 배선, Cloud Run Job 배포, Scheduler 등록, E2E 회귀(삼성전자 검색 PASS)

### Claude's Discretion
- `stocks` 마스터 스키마 상세 필드 NOT NULL/NULLABLE 구분
- 인덱스 전략 (name trigram / code btree / market / updated_at)
- 마스터 초기 백필 방식 — npm 일회성 스크립트 vs Cloud Run Job 수동 1회 실행
- scanner ingestion의 `top_movers` + `stock_quotes` 이중 쓰기 트랜잭션 경계
- RLS 정책 — 기존 stocks 정책을 3테이블로 승계하는 구체 구현
- inquirePrice 실패 시 상세 응답 폴백 (기존 em-dash 정책 재사용)
- 검색 API 응답 스키마 — 시세 필드 optional 표현 방식

</decisions>
