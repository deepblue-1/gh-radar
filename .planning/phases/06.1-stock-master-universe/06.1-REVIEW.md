# Phase 06.1 Code Review

## CRITICAL

### CRIT-1: ingestion `index.ts` — `fetchAllRanking` 이중 호출 (KIS API 4회 낭비 + 데이터 일관성)

**파일:** `workers/ingestion/src/index.ts:21`, `workers/ingestion/src/pipeline/run.ts:20`

`main()`에서 `fetchAllRanking(client)` 결과를 `isHoliday()` 체크에만 사용하고 버림. 이후 `runPipeline(client, supabase)` 내부에서 다시 호출 → 매 cycle KIS API 4회. 두 호출 사이 시세 갱신 시 holiday 판단 불일치 가능.

**Fix:** `rankings`를 `runPipeline` 파라미터로 전달.

### CRIT-2: `GET /api/scanner` — 상장폐지 종목이 스캐너 응답에 노출

**파일:** `server/src/routes/scanner.ts:58-67`

`stocks` 마스터 조회에 `is_delisted` 필터 없음. 폐지 종목이 top_movers에 잔존 시 병합 결과에 포함.

**Fix:** `.eq("is_delisted", false)` 추가 + 병합 시 master 없는 종목 필터.

## HIGH

### HIGH-1: `KisRankingRow` 타입 — `stck_shrn_iscd`, `acml_hgpr_date` 미선언

**파일:** `packages/shared/src/kis.ts:3`

실제 사용되는 `stck_shrn_iscd`, `acml_hgpr_date`, `stck_oprc` 등이 타입에 없음. 인덱스 시그니처로 런타임은 통과하나 타입 안전성 결여.

**Fix:** 실제 사용 필드 명시적 선언.

### HIGH-2: `map.ts` — open 필드에 고가(stck_hgpr) 오기입

**파일:** `workers/ingestion/src/pipeline/map.ts:22`

`inquirePrice` 실패 폴백이 `stck_hgpr`(고가)를 open에 할당. `stck_oprc`(시가)를 사용해야 함.

**Fix:** `Number(row.stck_hgpr)` → `Number(row.stck_oprc)`

### HIGH-3: delist-sweep 조회가 PostgREST 기본 limit 1000에 걸림

**파일:** `workers/master-sync/src/index.ts:68-71`

`.limit()` 없음. ~2,771종목 중 1,771개 누락 → 상장폐지 마킹 부분 실패.

**Fix:** `.limit(10000)` 추가.

### HIGH-4: master-sync Job이 ingestion-sa 사용 (최소권한 위반)

**파일:** `scripts/deploy-master-sync.sh:84`

master-sync에 불필요한 KIS 시크릿 접근 권한. 전용 SA 분리 필요.

**Fix:** `gh-radar-master-sync-sa` 전용 SA 생성.

## MEDIUM

### MED-1: sanitizeSearchTerm — 싱글쿼트 미제거

**파일:** `server/src/schemas/search.ts:12-14`

PostgREST or-expr 파싱 오류 가능. SQL injection 위험은 낮으나 400/500 응답 유발 가능.

**Fix:** `replace(/[,()%']/g, "")` 로 확장.
