# Phase 09.2: 종목 상세페이지 상단 일봉차트 — Research

**Researched:** 2026-05-15
**Domain:** React 19 / Next.js 15 App Router 환경에서의 일봉 캔들스틱 + Volume bar 차트 라이브러리 비교 (recharts raw / recharts shadcn wrapper / lightweight-charts)
**Confidence:** HIGH (3종 mockup 모두 실제 webapp 환경에서 빌드 + typecheck PASS, 번들 크기 실측 완료)

## Summary

본 RESEARCH 는 Phase 09.2 의 가장 큰 미결 사항인 **차트 라이브러리 선정**을 위해 사용자 명시 후보 3종 (recharts raw / recharts + shadcn / lightweight-charts) 을 동일 fixture (005930 1M) + 동일 디자인 토큰 + 동일 데이터 fetch 경로로 Next.js 15 App Router 라우트로 작성, 실제 production build 까지 통과시켜 측정값을 비교했다.

핵심 발견: **lightweight-charts 5.2.0 이 번들 영향(+4 kB), 캔들+Volume 합성 비용(네이티브 series), 모바일 터치(crosshair 자동) 3개 축에서 모두 우위**. 단, 명령형 API 라 useEffect cleanup 패턴 이해 + CSS 변수 자동 매핑 불가능(다크모드 토글 시 chart.applyOptions 재호출 책임) 이라는 통합 비용이 있다. recharts 계열은 캔들 합성을 Bar `shape` prop 으로 직접 그려야 하며 (recharts 3.x 부터 `Customized` 컴포넌트 불필요, Bar `shape` 만으로 임의 SVG 렌더 가능 [VERIFIED: build PASS + recharts docs]), shadcn wrapper 는 **툴팁/색상 토큰 매핑은 자동화하지만 캔들 합성 비용 자체는 줄여주지 않는다**.

**Primary recommendation:** **lightweight-charts 5.2.0** 채택. 사유 = 번들 영향 최소(+4 kB vs recharts +60 kB), 캔들+Volume 네이티브, 트레이더 친숙도(TradingView 동일 라이브러리). 다만 사용자가 mockup 3종을 직접 비교 후 최종 선정하는 흐름(D-07/08)을 유지한다.

## 권장안

**lightweight-charts 5.2.0** 을 default 로 권장한다. 사유:

1. **번들 효율 압도적** — recharts 대비 1/15 의 번들 증가량 (+4 kB vs +60 kB). webapp 전체 baseline 102 kB shared 에 미미한 영향.
2. **캔들+Volume 네이티브** — Bar shape 으로 SVG 합성하는 recharts 패턴은 OHLC 픽셀 좌표 직접 계산 + doji 가드 등 detail 가 많아 maintainability 가 낮음. lightweight-charts 는 `addSeries(CandlestickSeries, {upColor, downColor, wickUpColor, ...})` + `setData()` 만으로 완성.
3. **트레이더 친숙도** — TradingView 동일 엔진. 3Y/줌/MA 추가 시 라이브러리 한계가 아닌 사용자 만족도 측면에서 자연 확장.
4. **모바일 터치 우수** — Canvas crosshair 가 핀치 줌·드래그 팬·long-press 모두 네이티브 처리. Phase 09.2 v1 은 hover 만 사용하나 향후 phase 에서 자연 확장.

**통합 비용 (planner 가 고지해야 하는 1회성 작업):**
- 다크모드 토글 자동화 = `next-themes` 의 `useTheme()` 훅으로 chart.applyOptions() 재호출 effect 추가 (≈10 라인). globals.css `.dark` 의 oklch 값을 그대로 주입. mockup 3 의 `detectDark()` + 별도 useEffect 로 확장 가능.
- `'use client'` 경계 명시 + `useEffect` cleanup (`chart.remove()`) — mockup 3 에 패턴 코드 존재.

**사용자 선정 트리거:**
1. 사용자가 `./dev.sh` 로 webapp 실행 → 로그인 → `/mockups` 접근 → 3개 mockup 직접 비교
2. 사용자가 1개 선정 → planner 가 `/gsd-plan-phase 09.2` 에서 PLAN 의 D-Lib 결정으로 lock-in
3. 권장안과 다른 선택 시: 본 §"권장안" 의 사유와 사용자 선택 사유를 PLAN CONTEXT 에 명시

**대안 선정 시 참고:**
- **shadcn wrapper 채택 시:** 토큰 일관성이 결정적인 가치라면 합리적 선택. 단 캔들 합성 비용은 raw 와 동일하다는 점 인지 필요. PLAN 에서 wrapper 의 `<ChartStyle>` CSS 변수 인젝트 패턴 기반 재사용성을 강조해야 정당화 가능.
- **recharts raw 채택 시:** 번들/통합 비용 모두 wrapper 보다 약간 우위. 단 `ChartTooltipContent` 같은 고급 툴팁 일관성을 잃는다. shadcn 추가의 4 kB 차이를 감수할 가치가 있는지 사용자 판단.

### Pattern 1: webapp → Supabase 직접 fetch (D-12 mirror watchlist-api.ts)

**What:** RLS 가 anon SELECT 허용된 테이블은 webapp 의 `createBrowserClient()` 로 직접 PostgREST 호출. 신규 server 엔드포인트 추가 없음.

**When to use:** read-only + RLS 자동 필터 가능 + AbortController 로 충분한 캐시 정책일 때 (Phase 9 stock_daily_ohlcv 가 정확히 해당).

**Example (mockup 의 `shared/fetch-daily.ts` 발췌):**

```typescript
// Source: webapp/src/app/mockups/shared/fetch-daily.ts (RESEARCH 산출)
'use client';
import { createClient } from '@/lib/supabase/client';

export async function fetchDailyOhlcv(code: string, range: RangeKey = '1M') {
  const supabase = createClient();
  const from = rangeToFromDate(range); // 1M=60d, 3M=120d, 6M=220d, 1Y=400d 마진
  const { data, error } = await supabase
    .from('stock_daily_ohlcv')
    .select('date, open, high, low, close, volume, change_amount, change_rate')
    .eq('code', code)
    .gte('date', from)
    .order('date', { ascending: true });
  if (error) throw error;
  return (data ?? []).map(toRow);
}
```

PLAN 단계에서 추가할 항목 (D-13/D-14):
- `signal?: AbortSignal` 파라미터 + 이전 요청 취소
- 에러 envelope 표준화 (Phase 6 `ApiClientError` 참고)
- packages/shared 로 타입 이전 (D-15)

### Pattern 2: lightweight-charts 명령형 useEffect 패턴 (권장안)

**What:** chart 인스턴스를 React virtual DOM 외부에서 관리. 3개 useEffect 분리: (1) fetch, (2) chart create/destroy (mount/unmount 1회), (3) data 갱신 (`setData()`).

**When to use:** Canvas 기반 차트 라이브러리 (lightweight-charts, chart.js) 통합 시.

**Example (mockup 3 발췌):**

```typescript
// Source: webapp/src/app/mockups/chart-lightweight/page.tsx (RESEARCH 산출)
'use client';
import { CandlestickSeries, HistogramSeries, createChart } from 'lightweight-charts';

const chartRef = useRef<IChartApi | null>(null);
const candleSeriesRef = useRef<ISeriesApi<'Candlestick'> | null>(null);

useEffect(() => {
  if (!containerRef.current) return;
  const chart = createChart(containerRef.current, {
    autoSize: true,
    layout: { background: { color: palette.bg }, textColor: palette.text },
    grid: { vertLines: { color: palette.grid }, horzLines: { color: palette.grid } },
  });
  // v5 breaking change: addSeries(SeriesDefinition, options) 통합 패턴
  const candleSeries = chart.addSeries(CandlestickSeries, {
    upColor: palette.up, downColor: palette.down,
    borderUpColor: palette.up, borderDownColor: palette.down,
    wickUpColor: palette.up, wickDownColor: palette.down,
  });
  const volumeSeries = chart.addSeries(HistogramSeries, {
    priceFormat: { type: 'volume' },
    priceScaleId: 'volume',
  });
  chart.priceScale('volume').applyOptions({ scaleMargins: { top: 0.75, bottom: 0 } });
  chart.priceScale('right').applyOptions({ scaleMargins: { top: 0.05, bottom: 0.3 } });
  chartRef.current = chart;
  candleSeriesRef.current = candleSeries;
  volumeSeriesRef.current = volumeSeries;
  return () => { chart.remove(); chartRef.current = null; };
}, []);  // mount/unmount 1회

useEffect(() => {
  if (!rows || !candleSeriesRef.current) return;
  candleSeriesRef.current.setData(rows.map(r => ({
    time: r.date, open: r.open, high: r.high, low: r.low, close: r.close,
  })));
  // ...
}, [rows]);
```

### Pattern 3: recharts Bar `shape` prop 으로 캔들 합성 (대안 시)

**What:** recharts 3.x 부터 `<Customized>` 없이도 Bar 의 `shape` prop 에 함수형 컴포넌트를 주입하면 임의 SVG 렌더 가능. Bar `dataKey="hl"` 가 [low, high] tuple → recharts 가 wick 영역의 y/height 픽셀을 계산 → 함수가 받은 props 의 x/y/width/height 와 payload(OHLC) 를 결합해 body rect + wick line 직접 그림.

**When to use:** lightweight-charts 가 부적합한 케이스 (예: SSR 친화 SVG 가 필수, virtual DOM 통합이 결정적).

**Example (mockup 1 발췌):** 위 §Code Examples 의 CandleShape 함수 참조.

[CITED: https://recharts.github.io/en-US/guide/customize/ — "Customized component used to be necessary in 2.x... starting from 3.x all charts are able to render arbitrary elements" + WebSearch 검증]

### Anti-Patterns to Avoid

- **chart 인스턴스를 useState 로 관리:** lightweight-charts 의 chart 인스턴스는 React state 외부 자원. `useRef` 로 보관하고 `useEffect` cleanup 에서 `.remove()`. useState 사용 시 매 render 마다 chart 재생성 우려.
- **server component 에서 chart 컴포넌트 렌더:** 3종 모두 `'use client'` 필수. lightweight-charts 는 `window`/`document` 직접 접근. recharts 는 SVG 라 SSR 자체는 가능하나 ResponsiveContainer 가 클라이언트 sizing 필요.
- **데이터 변경마다 chart 재생성:** lightweight-charts 는 `setData()` 만 호출. chart create 는 mount 1회.
- **Bar shape 함수 안에서 yAxis scale 직접 사용 시도:** recharts 는 외부에서 scale 함수 노출 안 함. mockup 1/2 처럼 `(high - value) / (high - low) * height` 비율 환산.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| 캔들스틱 차트 합성 | SVG 직접 작성 (path/rect/line 수동 계산) | lightweight-charts `addSeries(CandlestickSeries, ...)` 또는 recharts Bar `shape` prop | 가독성 픽셀 계산, doji 처리, wick 클리핑, 시간축 alignment 등 edge case 다수 |
| Volume bar 합성 | SVG bar 수동 | lightweight-charts `addSeries(HistogramSeries, ...)` 또는 recharts `<Bar dataKey="volume">` + `<Cell>` | priceFormat, scale 분리, 캔들 색상 동기화 등 표준 기능 |
| 다크모드 토큰 분기 (recharts) | 컴포넌트 내부 if 분기 | shadcn `<ChartStyle>` (Mockup 2) 또는 globals.css `.dark` 셀렉터 (Mockup 1) | CSS 변수가 자동 cascade, JS 분기보다 안전 + 빠름 |
| 다크모드 토큰 분기 (lightweight-charts) | 컴포넌트 내부 if 분기 | `next-themes` `useTheme()` 훅 + `chart.applyOptions()` 재호출 (PLAN 단계 추가 책임) | 명령형 lib 의 표준 패턴 — globals.css 의 `.dark` 변수 raw 값(oklch) 직접 주입 |
| Tooltip 본체 | 수제 popover | recharts `<Tooltip content={...}>` 또는 shadcn `<ChartTooltipContent>` 또는 lightweight-charts `subscribeCrosshairMove` + 자체 DOM | 위치 계산, 화면 경계 충돌, accessibility 모두 라이브러리가 처리 |
| OHLC 시간축 정렬 | Date string 수동 비교 + ASC sort | Supabase `.order('date', { ascending: true })` (mockup `fetch-daily.ts` 패턴) | DB 인덱스 활용, 클라이언트 정렬 회피 |

**Key insight:** 이 phase 의 Don't Hand-Roll 는 사실 단일 메시지 — **캔들 합성과 다크모드 토큰 주입을 라이브러리가 이미 해결한 문제로 다루어라**. 라이브러리 선정의 본질은 "어느 라이브러리의 합성 비용이 가장 낮은가" 다.

## Common Pitfalls

### Pitfall 1: recharts Bar `shape` 함수에서 OHLC 좌표 픽셀 계산 오류
**What goes wrong:** body rect 의 top/bottom 을 yAxis scale 함수 없이 추정 → 차트가 그려지긴 하지만 캔들 실루엣이 미세하게 어긋남.
**Why it happens:** Bar 가 받는 y/height 는 [low, high] 범위만 반영. open/close 는 별도로 비율 환산해야 함.
**How to avoid:** mockup 1/2 의 CandleShape 패턴 — `openY = y + ((high - open) / (high - low)) * height`. doji (high == low) 가드 필수.
**Warning signs:** 모든 캔들의 body 가 wick 끝에 붙어 있거나, 길이가 0인 doji 가 사라짐.

### Pitfall 2: recharts isAnimationActive 미설정 시 깜빡임
**What goes wrong:** range 토글 시 매번 fade-in 애니메이션이 실행되어 트레이더의 시선 추적을 방해.
**Why it happens:** recharts 기본 `isAnimationActive=true`.
**How to avoid:** 모든 Bar/Line 에 `isAnimationActive={false}` 명시. mockup 1/2 적용됨.
**Warning signs:** 1M ↔ 3M 토글 시 캔들이 한꺼번에 팝업.

### Pitfall 3: shadcn ChartConfig 의 `theme` 키 vs `color` 키 혼동
**What goes wrong:** `color: 'var(--up)'` 만 지정 → 다크모드에서도 light up 색상이 그대로 사용됨.
**Why it happens:** shadcn ChartConfig 는 `color: string` (단일) 또는 `theme: { light, dark }` (분기) 둘 중 하나만 받는 union 타입. CSS 변수 자체를 color 에 넣으면 wrapper 의 ChartStyle 인젝트 메커니즘 (light/dark selector 분기) 이 작동 안 함.
**How to avoid:** mockup 2 처럼 `theme: { light: 'oklch(0.66 0.20 22)', dark: 'oklch(0.72 0.19 22)' }` 직접 raw 값 분기. var() 안 쓰고 globals.css 의 oklch 값 그대로 인용.
**Warning signs:** 다크모드 토글 시 캔들 색상이 안 바뀜.

### Pitfall 4: lightweight-charts v5 breaking change — `addCandlestickSeries()` 미사용
**What goes wrong:** v4 docs 보고 `chart.addCandlestickSeries(...)` 호출 → `TypeError: chart.addCandlestickSeries is not a function`.
**Why it happens:** v5 부터 `chart.addSeries(SeriesDefinition, options)` 통합 패턴. CandlestickSeries / HistogramSeries / LineSeries 모두 별도 import 필요.
**How to avoid:** mockup 3 의 import 패턴 — `import { CandlestickSeries, HistogramSeries, createChart } from 'lightweight-charts'` + `chart.addSeries(CandlestickSeries, opts)`.
**Warning signs:** Runtime TypeError. typecheck 가 잡지만 v4 API 글을 먼저 본 경우 헷갈림.

[VERIFIED: lightweight-charts GitHub Issue #1797 + 공식 release notes — v5.0.0 breaking changes 섹션]

### Pitfall 5: lightweight-charts container display:none 시 chart 크기 0
**What goes wrong:** mockup 3 처럼 rows null 일 때 container 를 `display: none` 으로 숨기면 `autoSize` 가 0×0 으로 인식 → rows 도착 후에도 빈 차트.
**Why it happens:** Canvas 기반 차트는 mount 시점의 부모 element 크기를 측정. display:none 은 `getBoundingClientRect()` 결과 0.
**How to avoid:** **container 는 항상 visible 유지** + skeleton/empty 를 위에 오버레이 또는 별도 영역으로 배치. mockup 3 의 현재 패턴 (display:none 토글) 은 PLAN 에서 정정 필요. 권장: container 항상 mount + rows 없으면 chart 인스턴스만 생성 안 함 (별도 useEffect 가드).
**Warning signs:** 첫 로드는 정상, range 토글 후 빈 화면.

### Pitfall 6: lightweight-charts 다크모드 자동 반영 안 됨
**What goes wrong:** ThemeToggle 누르면 webapp 전체는 다크 → 차트만 라이트 색상 유지.
**Why it happens:** chart options 의 layout.background/textColor 가 mount 시점에 한 번 주입된 정적 값. CSS 변수와 무관.
**How to avoid:** PLAN 에서 `useTheme()` (next-themes) 구독 + theme 변경 시 `chart.applyOptions({ layout: { background: ..., textColor: ... }, grid: ... })` + 모든 series 의 `applyOptions({ upColor, downColor, ... })` 재호출. mockup 3 은 RESEARCH 한정으로 detectDark() 단발 호출만 — production 에선 effect 로 승격.
**Warning signs:** 다크 토글 후 차트 색상 미반영.

### Pitfall 7: Supabase `.order('date', ascending: true)` 누락 시 캔들 시간축 역순
**What goes wrong:** PostgREST 기본 정렬은 보장되지 않음 → 캔들이 시간 역순으로 그려져 "급등→급락" 흐름이 거꾸로 보임.
**Why it happens:** stock_daily_ohlcv 의 `(date DESC)` 인덱스 (Phase 9 D-03) 가 우선 적용될 수 있음.
**How to avoid:** mockup `shared/fetch-daily.ts` 처럼 `.order('date', { ascending: true })` 명시. 차트 라이브러리는 시간 ASC 가정.
**Warning signs:** 1M 토글 시 가장 최신 데이터가 좌측에 표시.

### Pitfall 8: `'use client'` 누락 시 hydration mismatch
**What goes wrong:** 차트 컴포넌트가 server component 로 평가되어 useState/useEffect 사용 불가 또는 lightweight-charts 의 window 접근으로 build 실패.
**Why it happens:** Next.js 15 App Router 의 server-by-default. CONTEXT D-12 + Phase 6 D-? 정책상 차트 컴포넌트는 client.
**How to avoid:** 차트 컴포넌트 + 데이터 fetch 모듈 모두 `'use client'` 명시. mockup 3종 모두 적용. `StockDetailClient` 가 이미 client 라 자연 호환.
**Warning signs:** Next build 에서 `useEffect is not defined in server` 또는 `window is not defined`.
