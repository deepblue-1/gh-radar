**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Google 로그인 기반 인증 레이어를 도입하고, 로그인 사용자별 관심종목(Watchlist) CRUD 기능을 제공한다. **Phase 06.2 이후 gh-radar 앱 전체가 로그인 필수로 전환**된다 — Scanner/Search/Detail 포함 모든 기능 경로는 `auth.uid()` 가 존재해야 접근 가능하며, 공개 경로는 `/`, `/login`, `/auth/*` 세 가지만 유지한다.

**Out of scope (이번 Phase 제외):**
- 이메일/비밀번호 로그인 (AUTH-01) — Google OAuth 전용 노선. 필요 시 별도 phase.
- Kakao 소셜 로그인 (AUTH-02의 Kakao) — Supabase 커스텀 OIDC 도입 부담. 별도 phase.
- 관심종목 드래그 리오더 UI — `position` 컬럼만 확보, UI는 후속.
- 복수 폴더/에이리어스 (다중 watchlists) — 단일 리스트.

**긴급 삽입 사유 (2026-04-16):** Phase 7 뉴스 배치가 "사용자별 관심종목" 타겟팅을 전제로 해야 트레이더 유즈케이스 완성. AUTH-01/02, PERS-01 v2→v1 승격 (STATE.md:97, PROJECT.md:56).

## Implementation Decisions

### Auth 아키텍처 — weekly-wine-bot admin 패턴 차용
- **D-01:** `@supabase/ssr` + `@supabase/supabase-js` webapp에 신규 설치. Express server는 변경 없음 (기존 service_role 경로 그대로).
- **D-02:** Supabase 클라이언트 3-파일 구조:
  - `webapp/src/lib/supabase/client.ts` — `createBrowserClient` (클라이언트 컴포넌트/훅)
  - `webapp/src/lib/supabase/server.ts` — `createServerClient` + `next/headers cookies()` (Server Component/Route Handler)
  - `webapp/src/lib/supabase/middleware.ts` — `createServerClient` + request/response 쿠키 (middleware 전용)
- **D-03:** 루트 `webapp/middleware.ts` 파일에 `updateSession()` 호출. weekly-wine-bot의 `proxy.ts` 네이밍은 비표준 — Next.js 공식 `middleware.ts` 사용. `config.matcher`로 정적 자산/이미지 제외.
- **D-04:** `webapp/src/app/auth/callback/route.ts` — `exchangeCodeForSession(code)` 후 `?next` 파라미터 따라 리다이렉트. weekly-wine-bot 패턴 재사용하되 user_roles 화이트리스트 체크는 **제거** (일반 사용자 대상).
- **D-05:** `AuthProvider` context (`webapp/src/lib/auth-context.tsx`) — `getSession()` + `onAuthStateChange` 구독 + bfcache `pageshow`/`visibilitychange` 재검증. 2초 타임아웃 안전장치. weekly-wine-bot 패턴 그대로 차용 (user_roles 조회 제거).
- **D-06:** Watchlist CRUD는 **webapp에서 Supabase 직접** (`createBrowserClient`로 `from('watchlists')` CRUD). RLS가 `auth.uid() = user_id`로 본인 row만 허용. Express 라우트 신규 없음.

### 인증 프로바이더 — Google ONLY
- **D-07:** `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: '${origin}/auth/callback', queryParams: { prompt: 'select_account' } } })`. Supabase Dashboard에서 Google provider 활성화 + Google Cloud Console OAuth 2.0 클라이언트 발급 필요.
- **D-08:** 이메일/비밀번호 로그인 UI·Supabase `confirm email` 설정 **도입하지 않음**. signUp/signInWithPassword 경로 미구현.
- **D-09:** Kakao 로그인 **이번 phase 범위 외**. REQUIREMENTS.md의 AUTH-02는 "Google 완료, Kakao 별도 phase"로 업데이트 필요.

### 라우트 가드 — 기본 차단 + 공개 whitelist
- **D-10:** middleware가 **모든 경로 기본 차단**. 공개 whitelist: `/`, `/login`, `/auth/*`. 그 외 미인증 요청은 `/login?next=<원래경로>` 로 302 리다이렉트.
- **D-11:** 로그인 성공 후 리다이렉트 기본값 — `?next=` 존재 시 그대로 따름, 없으면 `/scanner` (핵심 가치 경로).
- **D-12:** 로그인 상태에서 `/login` 진입 시 → `/scanner` 리다이렉트 (루프 방지, weekly-wine-bot 동일 패턴).
- **D-13:** `/api/*` (Express) 경로는 webapp middleware 영향 외 — 기존 CORS/공개 API 동작 유지. 단 webapp 화면에서 호출하는 것이므로 UI 레벨에서 이미 로그인 벽 뒤.

### 로그인 UI
- **D-14:** `/login` 페이지 (모달/시트 아님). weekly-wine-bot `admin/app/login/page.tsx` 레이아웃 참조 — Card + 중앙 정렬 + Google 버튼 + `Suspense`로 `useSearchParams` 래핑 + `?error=` 파라미터 메시지.
- **D-15:** 에러 파라미터: `?error=auth_failed` (exchangeCodeForSession 실패), `?error=oauth_denied` (사용자 취소 등). 한글 메시지.

### 헤더/사이드바 유저 섹션
- **D-16:** AppShell의 사이드바 활성화 — 현재 `hideSidebar=true` 기본값을 Phase 06.2 전역 레이아웃 기준으로 재조정. 사이드바 하단에 "유저 섹션" (아바타/이메일/로그아웃).
- **D-17:** `/design` 카탈로그 페이지는 기존대로 `hideSidebar`를 유지해야 회귀 없음 (planner가 prop 유지 확인).

### Watchlist 스키마
- **D-18:** `public.watchlists(user_id uuid, stock_code text, added_at timestamptz default now(), position int)` — PK `(user_id, stock_code)` 중복 방지.
- **D-19:** FK: `user_id` → `auth.users(id)` ON DELETE CASCADE, `stock_code` → `stocks(code)` ON DELETE CASCADE (Phase 06.1 3-테이블 마스터 기준).
- **D-20:** 사용자당 최대 50개 제한 — RLS INSERT 정책에 `(SELECT count(*) FROM watchlists WHERE user_id = auth.uid()) < 50` 체크 혹은 DB trigger.
- **D-21:** `position` 컬럼 확보 — v1 UI는 `added_at DESC` 고정 정렬. 드래그 리오더 UI는 후속 phase.
- **D-22:** RLS 정책:
  - `SELECT`: `auth.uid() = user_id`
  - `INSERT`: `auth.uid() = user_id` + count limit
  - `UPDATE`: `auth.uid() = user_id`
  - `DELETE`: `auth.uid() = user_id`

### Watchlist UI
- **D-23:** `/watchlist` 페이지 — Scanner와 **동일한 Table+Card 듀얼 레이아웃**. 기존 Scanner 컴포넌트(useScannerQuery 패턴 · Table/Card) 로직 재사용.
- **D-24:** 표시 컬럼: 종목명, 현재가, 등락률, 거래대금, 갱신시각 — Scanner와 동일 (Phase 05.2 `X-Last-Updated-At` 패턴 계승).
- **D-25:** 데이터 fetch — `watchlists` JOIN `stocks` JOIN `stock_quotes` (Phase 06.1 3-테이블). 폴링 1분 (Scanner 동일).
- **D-26:** ⭐ 토글 배치 — `/stocks/[code]` 상세 페이지 StockHero 내 + `/scanner` 각 행. 추가/해제 모두 동일 토글. Optimistic update + 실패 시 롤백.
- **D-27:** 빈 watchlist UX — "아직 관심종목이 없습니다. Scanner에서 ⭐를 눌러 추가해보세요" + Scanner 링크 CTA.

### Claude's Discretion
- 사이드바 유저 섹션 구체 디자인 (아바타 이니셜 vs Google 프로필 이미지, 팝오버 vs 항상 표시)
- 50개 제한 초과 시 에러 메시지 문구 (UI)
- `?error=` 파라미터의 전체 에러 케이스 매핑
- AuthProvider 타임아웃/재검증 값의 tuning
- 기존 Scanner E2E(Playwright 9종) 로그인 fixture 전환 방식
- Supabase Dashboard 설정 자동화 범위 (수동 vs seed/migration)
- ⭐ 토글 버튼의 구체 shadcn 컴포넌트 선택 (Button variant vs Toggle)

### Reusable Assets
- **weekly-wine-bot admin 인증 구조** — Supabase 클라이언트 3파일, middleware, callback route, AuthProvider: 직접 복사 후 whitelist(user_roles) 로직 제거하면 90% 재사용.

- Env: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` 이미 Vercel에 설정되어 있는지 확인 필요 (Phase 4 환경변수).
