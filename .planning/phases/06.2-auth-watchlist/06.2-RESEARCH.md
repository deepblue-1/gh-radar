### Locked Decisions

**Auth 아키텍처 — weekly-wine-bot admin 패턴 차용**
- **D-01:** `@supabase/ssr` + `@supabase/supabase-js` webapp에 신규 설치. Express server는 변경 없음 (기존 service_role 경로 그대로).
- **D-02:** Supabase 클라이언트 3-파일 구조:
  - `webapp/src/lib/supabase/client.ts` — `createBrowserClient`
  - `webapp/src/lib/supabase/server.ts` — `createServerClient` + `next/headers cookies()`
  - `webapp/src/lib/supabase/middleware.ts` — `createServerClient` + request/response 쿠키
- **D-03:** 루트 `webapp/middleware.ts` 파일에 `updateSession()` 호출. weekly-wine-bot의 `proxy.ts` 네이밍은 비표준 — Next.js 공식 `middleware.ts` 사용. `config.matcher`로 정적 자산/이미지 제외.
- **D-04:** `webapp/src/app/auth/callback/route.ts` — `exchangeCodeForSession(code)` 후 `?next` 파라미터 따라 리다이렉트. weekly-wine-bot 패턴 재사용하되 user_roles 화이트리스트 체크는 **제거**.
- **D-05:** `AuthProvider` context (`webapp/src/lib/auth-context.tsx`) — `getSession()` + `onAuthStateChange` + bfcache `pageshow`/`visibilitychange` 재검증. 2초 타임아웃. weekly-wine-bot 패턴 그대로 (user_roles 조회 제거).
- **D-06:** Watchlist CRUD는 **webapp에서 Supabase 직접** (`createBrowserClient`로 `from('watchlists')` CRUD). RLS가 본인 row만 허용. Express 라우트 신규 없음.

**인증 프로바이더 — Google ONLY**
- **D-07:** `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: '${origin}/auth/callback', queryParams: { prompt: 'select_account' } } })`.
- **D-08:** 이메일/비밀번호 로그인 UI·`confirm email` 설정 **도입하지 않음**.
- **D-09:** Kakao 로그인 **이번 phase 범위 외**. REQUIREMENTS.md 상태 갱신 필요.

**라우트 가드 — 기본 차단 + 공개 whitelist**

- **D-11:** 로그인 성공 후 리다이렉트 — `?next=` 존재 시 그대로, 없으면 `/scanner`.

## Summary

Phase 06.2 는 **Google OAuth 기반 인증 레이어 도입** + **앱 전체 로그인 필수 전환** + **사용자별 watchlist CRUD** + **Scanner/Watchlist 반응형 듀얼(`InfoStockCard`) + ⭐ 토글(Ghost variant)** 을 구현한다. Auth 인프라는 `@supabase/ssr 0.10.2` 기반으로 weekly-wine-bot admin 패턴을 차용하되 user_roles 화이트리스트를 제거하는 얕은 수정만 필요하다. Watchlist 는 Supabase RLS(authenticated role)로 **기존 `anon_read_*` 정책 패턴에서 처음으로 이탈**하며, 50개 제한은 RLS INSERT 정책의 subquery 대신 **`BEFORE INSERT` trigger**로 구현하는 것이 에러 코드 명확성과 성능 양면에서 유리하다 `[VERIFIED: Supabase RLS 공식 토론]`. Scanner/Watchlist 듀얼은 기존 `scanner-table.tsx`/`scanner-card-list.tsx` 의 prop 인터페이스를 유지하면서 `InfoStockCard` 로 내부만 교체하는 shallow refactor 로 Phase 05 회귀 없이 달성 가능하다.

**Primary recommendation:** weekly-wine-bot 의 4-파일 Auth 스택(`client.ts` + `server.ts` + `middleware.ts` + `auth-context.tsx`)을 **직접 복사 → user_roles/permissions 호출 제거 → matcher 정책 재조정**하고, watchlist 50-limit 은 RLS + `BEFORE INSERT` trigger 조합으로 구현하되 **클라이언트에서는 에러 코드(`P0001` trigger-raise vs `42501` RLS-violation) 대신 메시지 prefix 매칭**으로 분기한다. Scanner 카드 재디자인은 `scanner-card-list.tsx` 내부 JSX 를 `InfoStockCard` 렌더로 교체만 하여 prop 호환 유지한다.

**파일 3 — `webapp/src/lib/supabase/middleware.ts` (middleware 전용)** `[VERIFIED: weekly-wine-bot/admin/lib/supabase/middleware.ts]`

```typescript
import { createServerClient } from "@supabase/ssr";
import { NextResponse, type NextRequest } from "next/server";

const PUBLIC_PREFIXES = ["/login", "/auth"];
const PUBLIC_EXACT = ["/"];

export async function updateSession(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request });

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll();
        },
        setAll(cookiesToSet) {
          // CRITICAL: request.cookies.set + response 재생성 + response.cookies.set 3단 구조
          cookiesToSet.forEach(({ name, value }) =>
            request.cookies.set(name, value)
          );
          supabaseResponse = NextResponse.next({ request });
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          );
        },
      },
    }
  );

  // Always use getUser() / getClaims() in middleware (Supabase 공식)
  const {
    data: { user },
  } = await supabase.auth.getUser();

  const pathname = request.nextUrl.pathname;
  const isPublic =
    PUBLIC_EXACT.includes(pathname) ||
    PUBLIC_PREFIXES.some((p) => pathname === p || pathname.startsWith(`${p}/`));

  // D-10: 기본 차단 + 공개 whitelist
  if (!user && !isPublic) {
    const url = request.nextUrl.clone();
    url.pathname = "/login";
    url.searchParams.set("next", pathname + request.nextUrl.search);
    return NextResponse.redirect(url);
  }

  // D-12: 로그인 상태에서 /login → /scanner
  if (user && pathname === "/login") {
    const url = request.nextUrl.clone();
    url.pathname = "/scanner";
    url.search = "";
    return NextResponse.redirect(url);
  }

  return supabaseResponse;
}
```

**중요 주의점:**
1. **request.cookies.set + response 재생성 + response.cookies.set 3단 구조** 는 `@supabase/ssr` 가 요구하는 캐논. request 만 set 하면 다음 Server Component 호출이 stale cookie 를 읽고, response 만 set 하면 동일 request 내 후속 읽기가 실패 `[CITED: supabase.com/docs/guides/auth/server-side/nextjs]`.
2. **`getUser()` 권장 (공식 문서 워닝):** `getSession()` 은 middleware 에서 revalidation 보장 없음. 최신 docs (2025/2026) 는 `getClaims()` 를 명시적으로 권장 (JWT 서명 검증) `[CITED: supabase.com/docs/guides/auth/server-side/nextjs — "Always use supabase.auth.getClaims()"]`. 0.10.x 에서 `getClaims()` 사용 가능 — 단 weekly-wine-bot 참조 구현은 `getUser()` 사용이므로, **실행자 재량: `getUser()` 로 패턴 일치 유지 또는 `getClaims()` 로 업그레이드**. 안전한 선택은 `getUser()` (AuthProvider 와 data shape 일치).
3. `supabase.auth.getUser()` 는 Auth 서버 RT 토큰 재발급까지 동반 — middleware 는 모든 요청에서 호출되므로 `matcher` 로 정적 자산 반드시 제외 (§Pattern 2).

### Pattern 3: Watchlist 50-limit — RLS + BEFORE INSERT Trigger

**문제 정의:** D-20 은 "RLS INSERT 정책에 count 체크 **혹은** DB trigger" 두 옵션을 열어둠. 본 research 는 trigger 방식이 더 우수함을 권고.

**`supabase/migrations/20260416120000_watchlists.sql`:**

```sql
BEGIN;

-- D-18: 스키마
CREATE TABLE watchlists (
  user_id    uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  stock_code text NOT NULL REFERENCES stocks(code)   ON DELETE CASCADE,
  added_at   timestamptz NOT NULL DEFAULT now(),
  position   int,
  PRIMARY KEY (user_id, stock_code)
);

-- 리스트 조회 (added_at DESC) 및 JOIN 시 활용
CREATE INDEX idx_watchlists_user_added_at
  ON watchlists (user_id, added_at DESC);

-- D-22: RLS 활성화 + 4종 정책
ALTER TABLE watchlists ENABLE ROW LEVEL SECURITY;

CREATE POLICY "auth_select_own_watchlists"
  ON watchlists FOR SELECT
  TO authenticated
  USING (auth.uid() = user_id);

CREATE POLICY "auth_insert_own_watchlists"
  ON watchlists FOR INSERT
  TO authenticated
  WITH CHECK (auth.uid() = user_id);  -- count 체크는 trigger 에서

CREATE POLICY "auth_update_own_watchlists"
  ON watchlists FOR UPDATE
  TO authenticated
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "auth_delete_own_watchlists"
  ON watchlists FOR DELETE
  TO authenticated
  USING (auth.uid() = user_id);

-- D-20: 50개 제한을 BEFORE INSERT trigger 로
-- RLS WITH CHECK 에 count 서브쿼리를 넣는 대신 trigger 를 쓰는 이유:
--   1. RLS 위반은 에러 코드 42501 (permission_denied) 만 반환 — "limit 도달" 과
--      "타인 row 조작" 이 구분되지 않아 UX 메시지 분기 불가.
--   2. RLS count 서브쿼리는 infinite-recursion 위험 (discussion #14576/18715).
--   3. Trigger 는 RAISE EXCEPTION 'msg' USING ERRCODE='P0001' 로 명확한 커스텀 에러.
CREATE OR REPLACE FUNCTION enforce_watchlist_limit()
RETURNS TRIGGER AS $$
DECLARE
  cnt int;
BEGIN
  SELECT count(*) INTO cnt FROM watchlists WHERE user_id = NEW.user_id;
  IF cnt >= 50 THEN
    RAISE EXCEPTION 'watchlist_limit_exceeded'
      USING HINT = '관심종목은 최대 50개까지 저장할 수 있습니다.',
            ERRCODE = 'P0001';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_enforce_watchlist_limit
  BEFORE INSERT ON watchlists
  FOR EACH ROW
  EXECUTE FUNCTION enforce_watchlist_limit();

COMMIT;
```

**클라이언트 에러 분기 패턴 (`watchlist-toggle.tsx`):**

```typescript
const { error } = await supabase.from("watchlists").insert({
  user_id: user.id,
  stock_code: stockCode,
});

if (error) {
  // trigger-raise → P0001 + message === 'watchlist_limit_exceeded'
  // RLS violation → 42501 + message prefix 'new row ... violates row-level security'
  // PK 중복(이미 추가됨) → 23505 (unique_violation) — UX 상 set 상태로 간주
  if (error.code === "P0001" || error.message === "watchlist_limit_exceeded") {
    setError("관심종목은 최대 50개까지 저장할 수 있습니다.");
  } else if (error.code === "23505") {
    // 이미 추가되어 있음 — optimistic 과 정합성 맞춤, 에러 메시지 없음
  } else {
    setError("관심종목 변경에 실패했습니다.");
    rollback();  // optimistic rollback
  }
}
```

**UI-SPEC 에 기재된 `23514`(CHECK constraint) 코드는 본 구현에서 발생하지 않음** — UI-SPEC §5 Optimistic update 규칙 본문에 "`23514` check constraint or 커스텀 에러"로 되어있으나, RLS WITH CHECK 위반은 **42501** 을 반환 `[VERIFIED: supabase.com/docs/guides/troubleshooting/database-api-42501-errors + PostgreSQL SQLSTATE 참조]`. trigger 기반으로 구현하면 **P0001** 이 반환되므로 UI-SPEC 은 documentation 상 조정 필요 (planner 가 sync).

import { NextResponse } from "next/server";
import { createClient } from "@/lib/supabase/server";

export async function GET(request: Request) {
  const { searchParams, origin } = new URL(request.url);
  const code = searchParams.get("code");
  const next = searchParams.get("next") ?? "/scanner";  // D-11

  // Google 취소: Google → Supabase 리다이렉트 시 error=access_denied 전달됨
  const oauthError = searchParams.get("error");

  if (code) {

    const { error } = await supabase.auth.exchangeCodeForSession(code);

      return NextResponse.redirect(`${origin}${next}`);
    }
  }

  return NextResponse.redirect(`${origin}/login?error=auth_failed`);
}

### Pattern 9: Watchlist JOIN Query (D-25)

**PostgREST embedded resource 문법** `[CITED: supabase.com/docs/guides/database/joins-and-nesting]`:

```typescript
// webapp/src/lib/watchlist-api.ts
export async function fetchWatchlist(supabase: SupabaseClient) {
  return supabase
    .from('watchlists')
    .select(`
      stock_code,
      added_at,
      position,
      stock:stocks!inner (
        code,
        name,
        market,
        kosdaq_segment
      ),
      quote:stock_quotes (
        price,
        change_amount,
        change_rate,
        volume,
        trade_amount,
        updated_at
      )
    `)
    .order('added_at', { ascending: false });
}
```

- `stock:stocks!inner(...)` — inner join, stocks 마스터 없으면 row 제외 (stocks CASCADE delete 로 이미 일관성 보장).
- `quote:stock_quotes(...)` — left join, 시세 미수신 종목도 표시 (StockHero em-dash 정책 계승 — Accumulated Context Phase 06).
- **RLS:** `watchlists` 는 authenticated 역할 기반, `stocks`/`stock_quotes` 는 `anon` SELECT 기반 — authenticated 역할은 anon 정책 계승 (PostgreSQL 역할 상속). 또는 `authenticated` 용 정책 별도 추가.

**⚠ 확인 필요:** `stocks`/`stock_quotes` 의 기존 RLS 정책 (`anon_read_stocks_master`, `anon_read_stock_quotes`) 은 `TO anon` 지정 — authenticated 사용자는 anon 역할이 아니라 **별도 정책 필요** 또는 `TO anon, authenticated` 로 확장 필요 `[CITED: supabase.com/docs/guides/database/postgres/row-level-security]`.

**마이그레이션 수정 필요 (planner 가 포함):**
```sql
-- stocks 와 stock_quotes 의 기존 anon 정책을 authenticated 에도 확장
DROP POLICY "anon_read_stocks_master" ON stocks;
CREATE POLICY "read_stocks_master" ON stocks FOR SELECT TO anon, authenticated USING (true);

DROP POLICY "anon_read_stock_quotes" ON stock_quotes;
CREATE POLICY "read_stock_quotes" ON stock_quotes FOR SELECT TO anon, authenticated USING (true);
```

### Pattern 10: 페이지 헤더 갱신시각 — 클라이언트 MAX 계산 (D-24)

Scanner 는 Express 가 `X-Last-Updated-At` 헤더를 반환 `[VERIFIED: scanner-client.tsx:48]`. Watchlist 는 Supabase 직접 호출이므로 헤더 없음 → **클라이언트에서 `MAX(quote.updated_at)` 계산**.

```typescript
// use-watchlist-query.ts
const stocks = data ?? [];
const lastUpdatedAtMs = stocks.reduce((max, row) => {
  const t = row.quote?.updated_at ? new Date(row.quote.updated_at).getTime() : 0;
  return t > max ? t : max;
}, 0);
const lastUpdatedAt = lastUpdatedAtMs > 0 ? lastUpdatedAtMs : undefined;
```

- stocks 가 없거나 모두 quote 없음이면 `undefined` (페이지 헤더에서 "—" 렌더)
- stale-but-visible 정책 scanner-client 와 동일하게 유지

### Pattern 11: Playwright Auth Fixture — storageState + REST Seeding (Claude's Discretion)

**목표:** 기존 Phase 06 E2E 4 spec (search/stock-detail/smoke/a11y) 이 비로그인 상태를 전제 → 로그인 벽 뒤로 이동한 후에도 동일 테스트가 통과해야 함.

**권고 패턴** `[VERIFIED: mokkapps.de 블로그 + playwright.dev/docs/auth]`:

1. **테스트 전용 테스트 유저 (dev Supabase 프로젝트에 seed):**

   - 또는 Supabase **admin API** (`service_role` key) 로 CI 시작 시 1회 프로비저닝

2. **`webapp/e2e/auth.setup.ts` (Playwright setup project):**

3. **`playwright.config.ts` 확장:**

**권고:** 로컬 Supabase (`supabase start`) + 테스트 유저 고정 방식이 가장 신뢰성 높음. CI 에서는 GitHub Secrets 의 dev 프로젝트 credentials 사용.

### Anti-Patterns to Avoid

- **❌ RLS `WITH CHECK` 에 count 서브쿼리:** 에러 코드 구분 불가 (42501 하나로 "타인 row" 와 "50 초과" 가 합쳐짐). infinite recursion 위험. → **Trigger 사용** (Pattern 3).
- **❌ `getSession()` 을 middleware/Server Component 에서 사용:** 쿠키 값 그대로 반환하고 revalidation 안 함 → 만료 토큰으로 API 호출 실패. → **`getUser()` 또는 `getClaims()`** `[CITED: supabase.com/docs]`.
- **❌ middleware `matcher` 에서 `_next` 미제외:** Next.js 정적 자산마다 Supabase Auth API 호출 → 성능/비용 폭증.
- **❌ `<Link>` 전체 감싸기 + 내부 `<WatchlistToggle>`:** 클릭 경합, 키보드 Tab 이슈. → **Link 셀 단위 축소**.
- **❌ watchlist `SELECT` RLS 정책 누락:** authenticated 역할은 anon `USING (true)` 자동 상속 안 됨 → 본인 row 도 안 보임. → **명시적 `CREATE POLICY TO authenticated`** (Pattern 3).
- **❌ `auth.users` FK 에 다른 컬럼 쓰기:** Supabase 내부 스키마이므로 변경 금지. `user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE` 만 사용.
- **❌ Google OAuth Console redirect URI 를 앱 URL 로 설정:** Google Cloud Console 에서는 **Supabase 프로젝트 URL** (`https://<ref>.supabase.co/auth/v1/callback`) 을 등록해야 함. 앱 URL (`/auth/callback`) 은 Supabase Dashboard 의 "Site URL" / "Redirect URLs" 섹션에 등록 `[CITED: supabase.com/docs/guides/auth/social-login/auth-google]`.

## Common Pitfalls

### Pitfall 1: Supabase Cookie Sync 3단 오류 (@supabase/ssr middleware)

**What goes wrong:** middleware 에서 `setAll` 구현이 `request.cookies.set` 만 하거나 `response.cookies.set` 만 하면, 다음 Server Component 호출에서 stale 세션으로 `getUser()` 실패 → 무한 리다이렉트 루프 또는 `/login` 오진.
**Why it happens:** `createServerClient` 는 request 쿠키로 Auth 상태를 읽고, response 쿠키로 브라우저에 다음 요청의 세션을 전달. 두 path 가 서로 다른 cookie store 를 가짐.
**How to avoid:** Pattern 1 의 3단 구조 **정확히** 복사. `supabaseResponse = NextResponse.next({ request })` 재생성이 중간에 반드시 있어야 다음 Server Component 가 새 request 쿠키를 읽음 `[CITED: supabase.com/docs/guides/auth/server-side/nextjs]`.
**Warning signs:** 로그인 직후 `/scanner` 진입 → 1~2초 후 `/login` 으로 튕김. Vercel logs 에서 `302 → /login` 반복.

### Pitfall 2: Google OAuth Redirect URI 혼동

**What goes wrong:** Google Cloud Console 에 앱 URL (`https://gh-radar-webapp.vercel.app/auth/callback`) 을 직접 등록하면 OAuth 플로우가 "redirect_uri_mismatch" 로 실패.
**Why it happens:** Supabase 가 Google → Supabase (`<ref>.supabase.co/auth/v1/callback`) 로 토큰을 받은 후 **자체 리다이렉트 로직**으로 앱의 `/auth/callback?code=...` 에 302. Google 관점에서는 Supabase URL 만 보임.
**How to avoid:**
- **Google Cloud Console Authorized redirect URIs:** `https://<supabase-ref>.supabase.co/auth/v1/callback` (이것만)
- **Supabase Dashboard > Authentication > URL Configuration:**
  - Site URL: `https://gh-radar-webapp.vercel.app`
  - Redirect URLs: `https://gh-radar-webapp.vercel.app/auth/callback`, `http://localhost:3000/auth/callback`
- **앱 코드:** `redirectTo: '${window.location.origin}/auth/callback'` (D-07)
**Warning signs:** 브라우저 탭에서 Google 로그인 직후 "Error 400: redirect_uri_mismatch" 표시 (Google 화면).

### Pitfall 3: `authenticated` 역할이 `anon` 정책 자동 상속 안 함

**What goes wrong:** watchlist JOIN 쿼리 (`stocks`/`stock_quotes`) 에서 로그인 사용자가 row 를 못 봄. JSON 응답에 `stock: null`.
**Why it happens:** RLS 정책이 `TO anon USING (true)` 로만 지정 → authenticated 역할은 별도 정책 없으면 암묵 차단.
**How to avoid:** 정책을 `TO anon, authenticated` 로 수정 (Pattern 9 마이그레이션 조각).
**Warning signs:** watchlist 페이지에서 종목명이 비어있고 stock_quotes null. 로컬 Supabase 에서 테스트 유저로 select 시 0 row.

### Pitfall 4: `@supabase/ssr` 쿠키 청크 이슈 (JWT > 4KB)

**What goes wrong:** 로그인 후 쿠키가 브라우저 4KB 한계 초과로 일부 청크만 저장 → `getUser()` 가 broken JSON 만남.
**Why it happens:** Google OAuth JWT 에 프로필 이미지 URL 등 포함 시 2-4KB. `@supabase/ssr` 은 자동 chunk 분할 (`sb-<ref>-auth-token.0`, `.1`, ...) 지원 — 단 middleware 의 `getAll`/`setAll` 이 모든 chunk 쿠키 전달 안 하면 결합 실패.
**How to avoid:** `getAll()` 은 `request.cookies.getAll()` 전체 반환, `setAll(cookiesToSet)` 은 배열 전체 순회 — 위 Pattern 1 이미 준수. 하드코딩으로 특정 쿠키 이름만 다루지 말 것.
**Warning signs:** Safari/Firefox 에서만 로그아웃됨 (쿠키 저장 정책 차이). Chrome 정상.

### Pitfall 5: Next.js 15 `cookies()` async 전환

**What goes wrong:** `server.ts` 에서 `const cookieStore = cookies()` (await 없음) 쓰면 TypeScript 는 통과하지만 런타임에서 `cookieStore.getAll is not a function`.
**Why it happens:** Next.js 15 에서 `cookies()` 가 `Promise<ReadonlyRequestCookies>` 반환으로 변경.
**How to avoid:** Pattern 1 의 `const cookieStore = await cookies()` 엄수. weekly-wine-bot 참조 코드는 올바름.
**Warning signs:** 빌드는 통과, 런타임 시 Server Component 에서 error page.

### Pitfall 6: middleware 가 Express `/api/*` 에 영향 주지 않는다는 오해

**What goes wrong:** planner 가 "Express API 도 미인증 시 401 반환해야 하나?" 고민.
**Why it happens:** webapp middleware 는 **webapp 도메인 (vercel.app) 의 요청만** 가로채므로 Express Cloud Run 도메인 요청은 당연히 영향 없음. 단 webapp 의 UI 레벨에서 이미 로그인 벽 뒤라서 사용자가 `/api/*` 를 직접 호출할 경로가 없음.
**How to avoid:** D-13 그대로 — Express 쪽 auth 는 별도 phase. `/api/*` 는 공개 API 성격 유지. 단 webapp UI 에서 호출할 때 로그인 사용자만 접근 가능하므로 실질 보안 문제 없음.
**Warning signs:** "Express 도 JWT 검증해야 하는지" 질문이 나옴 → 본 phase 범위 아님을 명확히.

### Pitfall 7: Scanner Table `md:block` vs `lg:block` 의 브레이크포인트 실수

**What goes wrong:** D-23/D-23.1 확정은 `lg+ Table, <lg 카드`. 하지만 기존 코드는 `md:block`/`md:hidden` 사용 `[VERIFIED: scanner-table.tsx:25, scanner-card-list.tsx:21]`. Phase 06.2 에서 `InfoStockCard` 로만 교체하고 브레이크포인트 그대로 두면 Tablet 사이즈(md..lg)에서 Table 표시됨 — 사용자 의도와 불일치.
**How to avoid:** Scanner + Watchlist 양쪽 모두 `md:` → `lg:` prefix 로 브레이크포인트 통일. Tailwind 4.x default `lg = 1024px`.
**Warning signs:** iPad 해상도에서 Scanner 가 Table 로 나오고 Watchlist 는 카드로 나옴 (불일치).

### Pitfall 8: Vercel Preview Deploy URL 과 Supabase Redirect URL 불일치

**What goes wrong:** PR 별로 생성되는 Vercel preview URL (`gh-radar-webapp-<hash>.vercel.app`) 이 Supabase Redirect URLs 에 등록되지 않아 OAuth 실패.
**How to avoid:** Supabase Dashboard > Authentication > URL Configuration 에서 wildcard 지원:
  - `https://*-vercel.app/auth/callback` 형태는 지원 안 됨 — **개별 등록** 필요
  - 대안: preview 배포는 로그인 테스트 skip, 또는 `vercel.json` 에서 preview 에 고정 별칭 alias 설정
**Warning signs:** Preview 배포에서만 로그인 실패, Production 정상.

### 예시 2: Watchlist Toggle with Optimistic Update

```tsx
// webapp/src/components/watchlist/watchlist-toggle.tsx
"use client";

import { Star } from "lucide-react";
import { useOptimistic, useTransition, useState } from "react";
import { Toggle } from "@/components/ui/toggle";
import { createClient } from "@/lib/supabase/client";
import { useAuth } from "@/lib/auth-context";
import { cn } from "@/lib/utils";

interface Props {
  stockCode: string;
  stockName: string;
  initialWatched: boolean;
  isAtLimit?: boolean;  // 페이지 상위에서 count=50 판단하여 전달
}

export function WatchlistToggle({ stockCode, stockName, initialWatched, isAtLimit }: Props) {
  const { user } = useAuth();
  const [isPending, startTransition] = useTransition();
  const [error, setError] = useState<string | null>(null);
  const [watched, setWatched] = useOptimistic(initialWatched);

  if (!user) return null;  // 미로그인 시 렌더 안 함 (middleware 가 벽이지만 방어)

  const handleToggle = async (pressed: boolean) => {
    startTransition(async () => {
      setWatched(pressed);  // optimistic
      setError(null);
      const supabase = createClient();

      if (pressed) {
        const { error: err } = await supabase
          .from("watchlists")
          .insert({ user_id: user.id, stock_code: stockCode });
        if (err) {
          setWatched(false);  // rollback
          if (err.code === "P0001" || err.message?.includes("watchlist_limit_exceeded")) {
            setError("관심종목은 최대 50개까지 저장할 수 있습니다.");
          } else if (err.code === "23505") {
            setWatched(true);  // 이미 있음 — 실상 set 상태
          } else {
            setError("관심종목 변경에 실패했습니다.");
          }
        }
      } else {
        const { error: err } = await supabase
          .from("watchlists")
          .delete()
          .eq("user_id", user.id)
          .eq("stock_code", stockCode);
        if (err) {
          setWatched(true);  // rollback
          setError("관심종목 변경에 실패했습니다.");
        }
      }
    });

    // 2초 후 에러 자동 소거
    if (error) setTimeout(() => setError(null), 2000);
  };

  return (
    <>
      <Toggle
        pressed={watched}
        onPressedChange={handleToggle}
        disabled={isPending || (!watched && isAtLimit)}
        aria-label={watched ? `${stockName} 관심종목 해제` : `${stockName} 관심종목 추가`}
        aria-busy={isPending || undefined}
        className={cn(
          "size-9 hover:bg-transparent data-[state=on]:bg-transparent",  // Ghost variant
          watched
            ? "text-[var(--primary)]"
            : "text-[var(--muted-fg)]/70 hover:text-[var(--primary)]",
          isPending && "animate-pulse",
          !watched && isAtLimit && "opacity-40"
        )}
        title={!watched && isAtLimit ? "관심종목은 최대 50개까지 저장할 수 있습니다" : undefined}
      >
        <Star
          className={cn("size-4", watched && "fill-[var(--primary)]")}
          strokeWidth={1.5}
          aria-hidden="true"
        />
      </Toggle>
      {error && (
        <p role="alert" className="mt-1 text-xs text-[var(--destructive)]">
          {error}
        </p>
      )}
    </>
  );
}
```

## Open Questions

1. **Vercel 환경변수 존재 여부 (A1)**
   - What we know: Phase 4 DEPLOY-LOG 에 언급. CLAUDE.md "기존 credentials 재요청 금지" 지침.
   - What's unclear: 현재 Vercel Production/Preview 에 `NEXT_PUBLIC_SUPABASE_*` 변수가 실제 설정되어 있는지.

   - What we know: Phase 1~6 은 Google OAuth 를 쓰지 않음.

   - What we know: Phase 1 에서 remote Supabase 만 사용 (`supabase/config.toml` 은 있으나 로컬 CLI 사용 이력 없음).
   - What's unclear: E2E 테스트 격리를 위한 로컬 Supabase 기동을 이번 phase 에서 활성화할지, 아니면 dev 프로젝트 직접 찌르는 경로 유지할지.

4. **Scanner 브레이크포인트 변경 (A8)**
   - What we know: UI-SPEC §4.3 "`<lg` 인포그래픽 카드" 명문.

   - What we know: Discretion 항목.
   - Recommendation: planner 가 실행자에게 맡김. Phase 06.2 Plan 에서 명시.

6. **Supabase Dashboard 설정을 seed migration 으로 자동화할지**
   - What we know: Google Provider 활성화, Site URL, Redirect URLs 는 Dashboard 수동 설정이 **기본 경로**. Supabase CLI `supabase/config.toml` 로 일부 자동화 가능하나 remote 적용은 per-project manual.
   - Recommendation: 이번 phase 는 **수동 + SETUP.md 문서화**. 자동화는 별도 phase.

## Security Domain

> Required — `security_enforcement` default enabled (config 에 명시 false 없음).

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | **yes** | Supabase Auth + Google OAuth 2.0 (PKCE 자동). JWT RS256 asymmetric key. |
| V3 Session Management | **yes** | @supabase/ssr httpOnly 쿠키 (chunk 분할 자동), `SameSite=Lax`, refresh token 자동 rotation. |
| V4 Access Control | **yes** | PostgreSQL RLS — `auth.uid() = user_id` 정책. authenticated role. |
| V5 Input Validation | **yes** | zod (선택) + DB CHECK constraint (stock_code 6자리 패턴은 `stocks` FK 로 자동 보장). |
| V6 Cryptography | **no** | Supabase 가 모든 암호화 관리 — 직접 구현 금지. |
| V7 Error Handling | **yes** | `role="alert"` + 한글 메시지 + `?error=` 4종 매핑 (UI-SPEC §Copywriting) |
| V8 Data Protection | **yes** | anon key 는 공개 가능 (RLS 가 본체 방어). service_role key 는 webapp 에 절대 노출 금지. |
| V9 Communications | **yes** | Vercel HTTPS only (자동). Supabase 연결도 HTTPS 강제. |

### Known Threat Patterns for Next.js 15 + Supabase Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| CSRF on OAuth callback | Spoofing | Supabase PKCE state 쿠키 자동 검증 (`@supabase/ssr` 기본) |
| JWT replay / session fixation | Spoofing | httpOnly cookie + `SameSite=Lax` + refresh rotation |
| Horizontal privilege escalation (타인 watchlist 읽기) | Elevation | RLS `auth.uid() = user_id` 정책 4종 (SELECT/INSERT/UPDATE/DELETE) |
| Open redirect via `?next=` | Tampering | **⚠ 검증 필요** — `next` 파라미터를 그대로 `NextResponse.redirect` 에 넘기면 외부 URL open redirect 가능. **planner: `?next=` 값이 `/` 로 시작하는지 검증 (relative path only)** |
| SQL injection through stock_code | Tampering | Supabase client 가 parameterized query 강제 (PostgREST 레벨 방어) |
| XSS through user_metadata display | Tampering | React 자동 escape, `dangerouslySetInnerHTML` 금지 |
| Clickjacking on `/login` | Tampering | `X-Frame-Options: DENY` 또는 `frame-ancestors 'none'` — Vercel 기본 CSP 검토 |
| OAuth scope over-request | Information Disclosure | Google Provider scope 기본값 (`openid profile email`) 만 사용 — 추가 scope 불필요 |
| Service role key 유출 | Information Disclosure | webapp 은 anon key 만 사용. service role 은 Cloud Run Secret Manager (기존 Phase 1/05.1 패턴) |
| 50 limit bypass via concurrent insert | Tampering | BEFORE INSERT trigger 가 DB 레벨 원자성 보장 — race window 없음 |

**✅ 추가 지침:**
- **Open redirect 방어 (Pattern 4 보완):**
  ```typescript
  const next = searchParams.get("next");
  const safeNext = next && next.startsWith("/") && !next.startsWith("//") ? next : "/scanner";
  return NextResponse.redirect(`${origin}${safeNext}`);
  ```
  `//example.com` 같은 protocol-relative URL 방어 필수.
- **CSP 검토:** Vercel 기본 CSP 는 `frame-ancestors` 미설정. `/login` 에 `X-Frame-Options: DENY` 헤더 추가 (next.config.ts 에서).
- **환경변수 노출 검토:** `NEXT_PUBLIC_*` 는 클라이언트 번들에 포함되므로 **anon key 만** 저장. service_role key 는 절대 NEXT_PUBLIC 접두사 사용 금지.
