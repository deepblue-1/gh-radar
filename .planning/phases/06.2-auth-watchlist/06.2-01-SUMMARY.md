## Open Redirect 가드 구현 인용

`webapp/src/app/auth/callback/route.ts` (T-06.2-02 mitigation):

```typescript
// Open redirect 방어 (T-06.2-02, ASVS V4.1.1, V5.1.5)
// - `/` 로 시작하지만 `//` 로 시작하는 protocol-relative URL 은 차단
const rawNext = searchParams.get("next");
const safeNext =
  rawNext && rawNext.startsWith("/") && !rawNext.startsWith("//")
    ? rawNext
    : "/scanner";
```

- 공격자가 `?next=//attacker.com/phish` 를 주입해도 `!rawNext.startsWith("//")` 에서 차단 → fallback `/scanner`
- ASVS V4.1.1 (server-side URL validation) + V5.1.5 (redirect allow-list) 충족

## Decisions Made

- **리다이렉트 가드 지연 활성화:** `updateSession` 내부의 비인증 차단 블록은 주석 TODO 로 남겨 Plan 06.2-03 에서 활성화. 이번 plan 은 쿠키 동기화만 수행하여 기존 페이지(`/scanner`, `/stocks/:code`, `/mockup/*`) 에 회귀가 없음. Plan 03 에서 `/login` 페이지 + 모든 진입 경로 정비가 완료된 시점에 가드를 켜는 것이 안전.
- **`useAuth()` Provider 바깥 호출 시 throw:** weekly-wine-bot 은 default context 반환하지만, gh-radar 는 AuthProvider 래핑 누락을 개발 시점에 즉시 포착하기 위해 throw 선택 (`useAuth must be used within <AuthProvider>`).
- **Context value 를 `AuthState | null` 로 둠:** `createContext<AuthState | null>(null)` — null 이면 Provider 밖이라는 신호. weekly-wine-bot 의 `EMPTY` 기본값 대비 실수 방지 장치.
