## Decisions Made

- **에러 키 fallback → `unknown`:** 4종 외 임의 값은 `unknown` 메시지 단일로 처리하여 내부 상태(중단 원인/스택) 유출 차단 (T-06.2-15 accept 라도 내부 힌트 주지 않는 방향으로 강화).
- **공개 whitelist 정교 매칭:** Plan 01 의 `PUBLIC_PREFIXES.some(p => pathname.startsWith(p))` 는 `/loginxyz` 같은 prefix 오매칭이 가능 → `pathname === prefix || pathname.startsWith(`${prefix}/`)` 로 교체. 보안상 해가 없어도 설계 의도 명확화.
- **보안 헤더는 next.config.ts:** middleware 에서 response 조작하는 방안 대비 선언적이고 Vercel Edge/CDN 캐싱과 무충돌. middleware 는 리다이렉트/쿠키 동기화에만 집중.
- **`/login` 페이지에도 safeNext 가드:** `/auth/callback` 에만 있으면 충분할 수도 있으나 `redirectTo` URL 조립 시 사용자 통제 가능한 입력(`?next=`) 이 들어가므로 이중 방어 유지 — ASVS V5.1.5 redirect allow-list 사상.

### Strict spec-compliance nudges

**2. [Plan spec literal] ERROR_MESSAGES 객체 키를 문자열 리터럴로 표기**

- **Found during:** Task 1 acceptance_criteria grep 검증
- **Issue:** plan 의 acceptance_criteria 에서 `grep -q '"auth_failed"'` 등 따옴표 포함 문자열을 요구. 초기 구현은 JS identifier 표기(`auth_failed: ...`) — 컴파일/기능 동등하지만 grep literal 검증을 통과하지 못함.
- **Fix:** 객체 키를 `"auth_failed": ...` 로 명시적 문자열 표기로 변경. 동작은 완전 동일.
- **Rationale:** acceptance_criteria 는 스펙의 표기 약속 — 기능 동등이라도 스펙 문자 그대로 반영.
