### Task 1 — watchlist-api.ts

**3 exports + WatchlistRow 인터페이스:**

```typescript
export interface WatchlistRow {
  stockCode: string;
  addedAt: string;
  position: number | null;
  stock: {
    code: string;
    name: string;
    market: "KOSPI" | "KOSDAQ";
    kosdaqSegment: string | null;
  };
  quote: {
    price: number;
    changeAmount: number;
    changeRate: number;
    tradeAmount: number;
    updatedAt: string;
  } | null;
}

export async function fetchWatchlist(supabase): Promise<{ data, error }>;
export async function addWatchlistItem(supabase, userId, stockCode);
export async function removeWatchlistItem(supabase, userId, stockCode);
```

**JOIN 쿼리 (RESEARCH §Pattern 9 준수):**
```
watchlists
  stock:stocks!inner (code, name, market, kosdaq_segment)
  quote:stock_quotes (price, change_amount, change_rate, trade_amount, updated_at)
.order('added_at', { ascending: false })
```

**단위테스트 6 + skipIf 1:**

6. removeWatchlistItem: delete().eq(user_id).eq(stock_code) 체인

### Task 2 — useWatchlistQuery

**반환:** `UseWatchlistQueryResult = { data, isLoading, isRefreshing, error, lastUpdatedAt, refresh }`

**핵심 패턴:**
- `POLL_INTERVAL_MS = 60_000` — Scanner 동일

**단위테스트 6/6:**
1. mount 즉시 fetchWatchlist 1회 호출

4. lastUpdatedAt = MAX(quote.updatedAt) epoch ms
5. 에러 발생 시 이전 data 유지, error state 세팅

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Correctness] `isLoading` 플래그 관리 — `load(initial: boolean)` 시그니처 → `isInitialRef` 로 교체**
- **Found during:** Task 2 구현 중, 플랜 action 예시 `load(initial)` 를 그대로 두면 `refresh: () => load(false)` 가 최초 렌더 직후 호출되는 경우 state 충돌 여지 발견.
- **Fix:** `load()` 시그니처는 파라미터 없음으로 단순화하고, `isInitialRef = useRef(true)` 로 "최초 1회 여부" 를 ref 내부에서 관리. `refresh = load` 단일 함수 노출.
- **영향:** 테스트는 동일하게 통과. 외부 API 는 플랜과 동일 (`refresh(): Promise<void>`). 내부 구현 단순화만.
- **Commit:** 6697a03

### Plan 본문과의 사소한 차이

**Plan 본문 `rowToStock` 의 `as any` cast → 구체 return type `StockWithProximity` 로 강화**
- 플랜 action 은 "실행자 재량: 공통 StockCardView 타입 분리 or cast" 라고 명시. 별도 view 타입 분리는 Plan 05 에 영향 없는 선에서 최소 impact 원칙에 따라 회피.
- 대신 Stock 타입의 필요 필드를 모두 채운 리터럴 객체로 빌드하고 return type 만 `StockWithProximity` 로 선언 — any 사용 회피.

**그 외 deviation 없음.** Plan 본문 action 예시를 그대로 따름 (grid-cols, 카피, 파일 경로 포함).
