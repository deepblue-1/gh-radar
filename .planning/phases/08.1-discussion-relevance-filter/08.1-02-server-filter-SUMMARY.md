# Phase 08.1 Plan 02: Server Filter Summary

Express 서버의 `GET /api/stocks/:code/discussions` 가 `?filter=meaningful` 쿼리 파라미터를 지원하도록 schema + route + mapper 3파일을 확장하고, supertest 통합 테스트 4건으로 회귀 방지. 응답에 `relevance` / `classifiedAt` 필드를 camelCase 로 노출하여 DISC-01.1 의 "서버 필터링" 계층을 성립.

## One-liner

Zod enum + Supabase `.or()` 단일 체이닝으로 meaningful 필터(relevance IS NULL OR relevance != 'noise')를 구현하고, toDiscussion 이 relevance/classifiedAt 2 필드를 camelCase 로 노출 — 총 33 production LoC + 192 test LoC, 103/103 PASS.

## What Was Built

### T-01: `DiscussionListQuery.filter` enum 추가

- `server/src/schemas/discussions.ts` — `.object({...})` 에 `filter: z.enum(['all', 'meaningful']).optional()` 추가.
- `.transform((q) => { ... })` 반환 객체에 `filter: q.filter ?? 'all'` — 미지정 시 `'all'` 로 정규화.
- 잘못된 값 (`?filter=foo`) 은 Zod enum 이 거부 → server error handler 가 400 `INVALID_QUERY_PARAM`.
- Phase 주석을 `Phase 08 + 08.1` 로 업데이트, filter 계약 문서화.

### T-02: GET 핸들러 relevance select + filter 체이닝 + mapper 확장

**`server/src/routes/discussions.ts`:**

- `DISCUSSION_SELECT` 상수: `"...,scraped_at" → "...,scraped_at,relevance,classified_at"`.
- `queryParsed.data` 분해에 `filter` 추가.
- Supabase query 체인을 4단계로 재구성:

  ```ts
  let q = supabase.from("discussions").select(DISCUSSION_SELECT).eq("stock_code", code).gte("posted_at", since);
  if (filter === 'meaningful') q = q.or('relevance.is.null,relevance.neq.noise');
  q = q.order("posted_at", { ascending: false }).limit(limit);
  if (before) q = q.lt("posted_at", before);
  ```

- `POST /refresh` 는 filter 무시(approved plan). 단 캐시 로드 select 는 동일 상수 공유 — refresh 후 응답에도 relevance/classifiedAt 포함 (side-benefit, webapp 에서 즉시 활용 가능).

**`server/src/mappers/discussions.ts`:**

- `DiscussionRow` 에 `relevance: string | null` + `classified_at: string | null` 2 필드.
- `toDiscussion` 반환 객체에 camelCase 매핑 추가:

  ```ts
  relevance: (row.relevance as Discussion['relevance']) ?? null,
  classifiedAt: row.classified_at ?? null,
  ```

- `as Discussion['relevance']` cast 는 DB CHECK 제약(4 라벨 | NULL) 불변성으로 안전.
