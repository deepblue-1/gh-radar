# Phase 08.1 Plan 05: Backfill Script Summary

`workers/discussion-sync/src/backfill.ts` 일회성 backfill 스크립트 구현 — 기존 15,463+ `classified_at IS NULL` 행을 chunk(100) 단위로 `classifyBatch` + `persistRelevance` 재사용 호출. DI-friendly `runBackfill(opts)` export 로 unit test 3건 PASS. production 실제 실행은 비용(~$23) + 시간(~50분) 으로 사용자 승인 하 manual follow-up 으로 deferred.

## One-liner

`runBackfill({ supabase, log, maxRows, chunkSize, selectPage, classify, persist, shouldStop? })` — page(10000 rows) × chunk(100 rows) 이중 루프로 `classified_at IS NULL` 행을 08.1-03 모듈 재사용해 분류, SIGINT graceful shutdown 지원, idempotent 재실행 안전 — DISC-01.1 의 "기존 15k 행 의미성 분류" 엔진 완성.

export interface BackfillOptions {
  supabase: SupabaseClient;
  log: pino.Logger;

  persist: typeof persistRelevance;    // DI — 테스트에서 fake 주입
  shouldStop?: () => boolean;          // SIGINT/SIGTERM 대체
}

export async function runBackfill(opts): Promise<BackfillResult> {
  while (!shouldStop() && processed < maxRows) {

  }
  return { processed, classified, failed, elapsedMs };
}

async function main() {
  // loadConfig → createLogger → createSupabaseClient
  // env: MAX_BACKFILL_ROWS(20000) / BACKFILL_CHUNK_SIZE(100) / BACKFILL_SELECT_PAGE(10000)
  // SIGINT/SIGTERM → shuttingDown=true
  await runBackfill({ ..., classify: classifyBatch, persist: persistRelevance,
                      shouldStop: () => shuttingDown });
  process.exit(0);
}

if (process.argv[1]?.endsWith("backfill.js") || endsWith("backfill.ts")) {
  main().catch(err => { console.error("[backfill] fatal", err); process.exit(1); });

핵심:

- **로직 재사용**: 08.1-03 의 `classifyBatch` / `persistRelevance` 그대로 호출 — 분류/저장 로직 중복 금지.
- **이중 루프**: 외부 `while` = SELECT page 단위(10000 rows), 내부 `for` = chunk 단위(100 rows). chunk 경계에서만 shouldStop + maxRows 체크.
- **Idempotent**: `classified_at IS NULL` 만 pickup → 재실행 시 이미 분류된 행 skip. Ctrl-C 후 재실행도 안전.
- **PROGRESS_EVERY=500**: 누적 processed/classified/failed + ratePerMin 로그 (ETA 추정 용이).
